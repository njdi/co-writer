Software After Software
26K
  1. Everything is changing. 1.1. We are in the midst of a transformation so large that nearly everything we assumed to be true about how software and business work must be rethought. 1.2. It will take years to play out. 1.3. The end game is not yet apparent. 1.4. It is better to try to keep up, instead of sitting it out.
  2. Still underestimated: how fast these models will get better and how good they will be. 2.1. Even if you think you're factoring in exponential improvements, you're likely still underestimating. 2.2. Models don't have to become perfect to break the old economics of software. They just need to be better than average. 2.3. Many of the things we currently do to help the models won't be needed anymore. 2.3.1. Why put training wheels on someone who never wobbles? 2.3.2. Why give a perfect typist a spellchecker? 2.4. Quality of output will solely depend on input, not capability of the model. 2.4.1. The need to review code written by agents will fade. 2.4.2. Errors of expression will be gone. 2.5. Every eight weeks you should give more leash to these models. Otherwise you'll be stuck on a low point of the curve.
  3. In the future, most code will be written by agents. 3.1. The assistant sidebar is dead. Agents will no longer be mere assistants. 3.2. Agents will run regardless of whether someone is sitting in front of the computer or not. 3.3. They will run for longer and with less supervision than most people think. 3.4. An agent forced to work like a human is a wasted agent. 3.5. The unit of work becomes the delegated task, not the code to be written.
  4. The bottleneck in software development has moved. 4.1. Writing valid code is trivial now. 4.2. Errors of engineering (priorities, sequencing, and tradeoffs) remain. 4.3. Review will shift from code to decisions.
  5. Software development as we've known it for the last 40 years is dead. 5.1. Software, as a profession, is built on the assumption that writing code is hard and error-prone. 5.2. Software, as an industry, is built on the assumption that code is scarce. 5.3. Both assumptions no longer hold.
  6. The old processes and practices of working with software are no longer apt. 6.1. Existing rituals and processes (planning, prioritization, handoffs, reviews) were made in a world in which implementation was slow, costly, error-prone, and human-bound. 6.2. In a world of abundant code, these rituals and processes become drag. 6.2.1. Why spend an hour prioritizing what can be done in thirty minutes?6.2.2. Why consider building a prototype when you can talk one into existence? 6.2.3. Why wait for a review when you can spin up five agents to review in parallel? 6.2.4. Why pick a single idea to try when you can try them all in parallel and dismiss four out of five without hurting a single ego? 6.5. AI is not only an accelerator for X. It changes whether X should exist at all.
  7. Software changes form. 7.1. Software, so far, was built for human use. In the future, software will be built mainly for agent use. 7.2. More software will be built just-in-time than ahead of time. Less pre-built software will be required. 7.3. The line between "using software" and "building software" will blur. It might disappear.
  8. Value moves away from code. 8.1 Software to do X has no value when an agent can do X. 8.2. The value of software as code-someone-else-wrote-for you declines. 8.3. Software that encodes workflows becomes less valuable. 8.4. Data, permissions, distribution, trust, compliance, regulatory position, and physical assets become more valuable. 8.5. Squeezed will be vendors whose moat was "customers cannot justify building this themselves". 8.6. The last ones to admit all of this will be software companies. Their business models were built on the old scarcity.
  9. Winners will reorganize around models. 9.1. It will not be enough to fit models into existing systems, org charts, and processes. 9.1.1. Agents are wasted when made to work like people. 9.2. The winners of this transformation are those who organized around the models. 9.2.1. A small team with strong judgment and many agents will outrun a large team trying to fit AI into processes from before the transformation. 9.3. Ability to keep up matters more than headcount.
  10. Every serious instituion needs a camp at the frontier. 10.1. Large corporations cannot sit this transformation out. 10.2. They cannot learn the future through committee. 10.3. They need a small autonomous team at the frontier, built around the models. 10.3.1. Its purpose is not to add AI to the old way of working. 10.3.2. Its purpose is to discover the new way of working and pull the corporation toward it. 10.3.3. It must be close enough to touch real systems, real data, real constraints, and real consequences. 10.3.4. Its output is not only software, but also people and practices.
  11. The engineer changes. 11.1. The software engineer does not disappear, but changes. 11.2. Software engineering shifts away from code and toward product thinking, shaping systems, judging tradeoffs, and business outcomes. 12.3. The most valuable engineers will be system thinkers and operators that can increase business value. That is and will remain unchanged.
  12. Therefore. 12.1. We build for a world in which intelligence is abundant, continuous, and cheap. 12.2. We optimize for learning what that world looks like. 12.3. We do not preserve old software processes out of habit. 12.4. We do not wait for the end state to become obvious. The best move is to play the game. 12.5. We go to the frontier first, because that is where the future is learned.
That's what we believe in at Amp and why we created
