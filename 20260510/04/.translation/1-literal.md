# 文字即思想，思想是神圣的

**作者：** ᴅᴀɴɪᴇʟ ᴍɪᴇssʟᴇʀ ([@DanielMiessler](https://x.com/DanielMiessler))  
**日期：** 2026年5月10日  
**来源：** [Text is Thought, and Thought is Holy](https://x.com/Zephyr_hg/status/2053157960717836318)

Anthropic 的 Thariq Shihipar 最近发表了一篇发人深省的文章，他在文中主张，对于大多数 AI 用例，我们应该从 Markdown 转向 HTML。

我理解他的观点，也在一定程度上表示赞同，但我不同意他提出的解决方案。

以下是我对他论点的理解：

- Markdown 对于长文档来说，阅读和分享都很困难
- 纯文本在表达想法方面不如带有格式、图片和视觉层次的内容
- 因此我们应该将主要规范格式从 Markdown 迁移到 HTML

他关于 Markdown 在分享和传递想法方面存在局限的论点很有道理，因为格式和视觉效果确实可以让内容极易理解。

我对这种方案的问题在于，用可编辑性换取可读性，会让我们人类与创作过程进一步分离。

我重视 Markdown，因为我重视文字。我重视文字，因为我认为它离思想只有一步之遥。

我认为思考是我们唯一需要小心不要外包出去的事情，而我担心这个想法夹带进来的是一个重大步骤，将让我们的创作对人类变得不透明。不只是 AI 的创作，也包括我们自己的创作。

我如此重视 Paul Graham 的原因，在于他在写出极简洁散文时所做的观点压缩工作。清晰地写作是困难的，因为它要求清晰地思考。

文字让你的想法裸露，我喜欢这样。

- 问题究竟是什么？
- 我们应该怎么解决？
- 为什么我们的解决方案比其他方案更好？

我喜欢在任何技术介入之前，用纯文字将这类关键内容结晶化的挑战。

如果我们不自己写那段文字，然后再去编辑它，这就开始感觉很像把一个强壮的机器人带进健身房。

我担心，如果我们用模糊的想法喂给 AI，让它吐出精美的 HTML，我们就立刻与那个想法断开了联系。就像，那个想法去哪了？它开始时只是一些模糊的感觉，然后被放进碎木机，变成了别人的 HTML。

我能在 4 个简洁的要点里看到它吗？我能盯着它看吗？我能与它搏斗吗？我能调整它吗？这是一个想法。我需要与它接触。

当然，我们可以让 AI 把那份出色的 HTML 文档总结成四个要点，但在压缩和展开的过程中，我们会损失一定比例的原内容。

也许我在这里太情绪化了。

我只是觉得，如果你没有把艰辛的思考和写作工作投入到原始想法中，然后以一种人类易于阅读和编辑的格式维护它，那么你就以某种方式把某种神圣的东西交给了机器。我这样说，而我是一个彻头彻尾的 AI 最大化主义者。

但我理解他要表达的观点，我认为它非常有道理。

用一个巨大的文本文件来解释或说服别人是困难的。格式有巨大的帮助。图片有巨大的帮助。甚至是一个界面、视频或其他什么东西。

所以我们在这一点上是同步的。我只是认为，也许可以用不同的方式达到同样的结果。

- **MARKDOWN：** 人类容易写，人类难以阅读。
- **HTML：** 人类难以写，人类容易阅读。

也许解决方案不是将第一步移至 HTML（这样对代理和人类都变得更不透明，加上 Thariq 谈到的版本控制问题）。也许解决方案是类似文档配对这样疯狂的东西：就像你有一个思想文件和一个（或多个）展示文件。

（顺便说一下，他有一个包含 HTML 示例的代码库，非常漂亮。）

提议是让 AI 直接写 HTML，对吧？那么为什么不只是为此单独建一个链接的文件呢？一个用于人类与 AI 之间清晰的创作和同步。简洁、清晰、精确，且人类可编辑。

我们不必二选一。我们可以（我个人也会）保留我们的 .md 或 .mdx 文件作为权威来源，然后简单地添加一个 post-commit 钩子，在每次更改后自动构建一个出色的展示优化的 HTML 文件或其他什么。

我讨厌多个文件的想法，但我认为这远比失去你用文字获得的那种透明的、可编辑的与想法的联系要好得多。

基本上，我认为让 AI 额外生成我们干净的、对代理友好的 Markdown 文件的一个丰富且可分享的版本，比让人类在想法被混入不透明的 HTML 时保持与想法的联系容易得多。

我认为人类思想到文字的连接是我们需要保留的最高优先位。

然而，我还在继续思考这个问题，非常感谢 Thariq 推动我们所有人在这方面进化。
