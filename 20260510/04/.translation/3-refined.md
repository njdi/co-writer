# 文字即思想，思想是神圣的

**作者：** ᴅᴀɴɪᴇʟ ᴍɪᴇssʟᴇʀ ([@DanielMiessler](https://x.com/DanielMiessler))  
**日期：** 2026年5月10日  
**来源：** [Text is Thought, and Thought is Holy](https://x.com/Zephyr_hg/status/2053157960717836318)

Anthropic 的 Thariq Shihipar 最近写了一篇很有意思的文章，主张在大多数 AI 场景里，我们应该放弃 Markdown，改用 HTML。

我理解他的出发点，也有一定认同，但不赞成他给出的解决方案。

他的核心论点大致是这样：

- 长文档用 Markdown 写，读起来和分享起来都很痛苦
- 纯文本传递想法的效率，远不如有格式、有图、有视觉结构的内容
- 所以，应该把默认的写作格式从 Markdown 换成 HTML

他说 Markdown 在分享和表达上有局限，这一点说得很对——格式和视觉确实能让内容一目了然。

但我的问题是：用可编辑性换可读性，是在让我们人类离创作这件事越来越远。

我重视 Markdown，因为我重视文字。我重视文字，因为文字离思想只有一步之遥。

我觉得，思考这件事是我们最不该外包的。而这个提议暗地里推着我们走向一个危险的方向：让我们的创作对人类越来越不透明。不只是 AI 的创作，我们自己的也是。

我特别欣赏 Paul Graham，原因之一就是他写出那种极简洁文章背后的思维压缩功夫。写清楚是很难的，因为它要求你想清楚。

文字让你的想法赤裸裸地摆在那里，我喜欢这种感觉。

- 问题究竟是什么？
- 怎么解决？
- 为什么这个方案比其他的好？

在任何技术介入之前，先用纯文字把这些关键问题说清楚——这种挑战，我很享受。

如果我们不自己动手写这段文字，也不自己去打磨它，就像是带着一个举重机器人去健身房——你去健身，但力气是机器人出的。

我担心的是：如果我们把模糊的想法塞给 AI，让它吐出漂亮的 HTML，我们就立刻和那个想法脱节了。那个想法去哪儿了？它开始时还只是些感觉，然后被丢进碎木机，出来就成了 AI 生成的 HTML。

我还能在四个简洁的要点里看到它吗？能盯着它想吗？能跟它较劲吗？能调整它吗？这是一个想法，我需要能触碰到它。

当然，我们可以再让 AI 把那份精美的 HTML 文档压缩成四条要点——但在一来一去的压缩和展开里，原来的东西已经流失了一部分。

也许我真的太感性了。

但我就是觉得：如果没有亲自下功夫去思考和写作，也没有用人类易读、易改的格式把想法留住，就是把某种神圣的东西拱手交给了机器。说这话的我，可是个死忠的 AI 派。

不过我也完全理解他的意思，他说的确实有道理。

靠一大坨纯文本去解释一件事、说服一个人，真的很难。格式帮大忙，图片帮大忙，甚至一个界面、一段视频，都能让理解容易很多。

这一点我们是一致的。我只是觉得，也许可以用另一种方式达到同样的效果。

- **MARKDOWN：** 人写起来容易，读起来费劲。
- **HTML：** 人读起来轻松，写起来麻烦。

解决方案也许不是把第一步就切换到 HTML——那样对 AI agent 和人来说都更不透明，还有 Thariq 提到的版本控制问题。也许答案是一个听起来有点疯的方案：文档配对，一个思想文件，一个（或多个）展示文件。

（顺便说一句，他有个放了 HTML 示例的代码库，做得很漂亮。）

他的提案是让 AI 直接生成 HTML，对吗？那为什么不单独建一个关联文件来做这件事？一个文件负责人与 AI 之间清晰、可编辑的创作和同步，追求简洁、精准；另一个文件负责展示。

我们不必二选一。我们可以（我个人就会这么做）继续用 .md 或 .mdx 文件作为内容的权威来源，然后加一个 post-commit 钩子，每次修改后自动生成一份展示优化的 HTML。

多个文件这个想法我不太喜欢，但我觉得它远比失去"用文字直接触碰想法"的那种透明感要好。

说到底：让 AI 在我们干净的、对 agent 友好的 Markdown 文件基础上额外生成一个丰富的、可分享的版本，比让我们人类在想法已经被融进不透明 HTML 之后还想保持与想法的联系，要容易得多。

人类从思想到文字的这条线，才是我们最不能断的那根。

当然，我还在想这件事。也非常感谢 Thariq 的文章——推着我们所有人在这件事上更进一步。
