这篇文章把这套系统完整地摆出来，每一节你都能直接抄回去用。

---

先说这套系统只优化一件事：

> 让人存下来的内容

不是让人点赞的内容，不是让人转发的内容——是那种让人觉得"这个以后用得上"、忍不住点下书签的内容。

一篇帖子能让人存下来，说明它传递了真实的价值。书签也是算法的信号，拿到书签的帖子，发布很久后还会持续出现在别人的信息流里。

**这篇文章能帮你拿走：**

- 完整的内容操作系统框架，对应今天就能建好的文件夹结构
- 四个可以直接粘贴进 Claude 或任何主流大模型的提示词
- V1 起步方案，前期1-2小时，之后每周省下同样的时间
- 作者上下文包模板，把模糊的想法变成有方向的草稿

## 有一条线不能过

在讲具体机制之前，先说一件更重要的事。

**草稿永远不能直接发，每一篇都要自己手动收尾。**

这套系统是加速器，不是自动驾驶。你要是让它全自动跑，它会慢慢变烂——因为模型生成的内容里没有你，读者感觉得到。

目标不是用提示词模仿出一个声音。而是从你真实的写作里提炼出一套可以复用的东西。这件事做一次，保持更新，之后每一稿都会从更接近你的地方开始——你的时间可以用在真正值得打磨的地方。

## 什么内容值得被收藏

书签是读者对未来的自己许下的承诺：这个以后还用得上。

这比点赞的门槛高很多。点赞是当下的一个反应，书签是一个决定。

我每次发帖之前都会问自己：这篇帖子长得像下面这些之一吗？

- 清单
- 蓝图
- 文件夹结构
- 模板
- 框架
- 分步工作流
- 带结论的截图证明
- 前后对比
- 可复用的心智模型

不像任何一种？那通常不该发。

## 整套系统长什么样

用一句话说清楚这套系统的核心思路：每一条内容都是一个对象，从有想法那一刻起，到发布，它一直带着自己的状态在系统里走。

不是一个巨型提示词，不是一堆通用文件夹，而是一套有结构的流程。

```
上下文来源

  外部：信号层
    - X书签、文章
    - 转录文字、私信
    - 回复、竞品帖子
    - 用途：改写、研究构思

  内部：知识图谱
    - 个人操作系统、笔记
    - 日记、语音备忘录
    - 已发布内容存档
    - 用途：原创、再加工

      ↓

策略 + 声音 + 存储
  - 定位、声音档案
  - 避免烂内容主文档
  - 想法、钩子、证明库、反馈日志

      ↓ 输入到

生产负责人
  - 打开运行文件夹
  - 通过想法门路由
  - 执行门控

      ↓ 创建

运行文件夹（每个内容对象一个）
  - 想法 → 简介 → 草稿 → 验证
  - 审核 → 调度 → 反馈

      ↓ 输出时写回

存储
  - 赢家、输家
  - 声音规则、禁用模式
  - 钩子、证明
```

系统的上下文来自两个地方。

一个是**信号层**——外部的东西：你这周存下的书签、你关注的创作者的内容、你觉得不错的文章。

另一个是**知识图谱**——你自己本来就有的：笔记、日记、语音备忘录，以及你已经发过的所有内容。

两者之间有一层策略+声音+存储，负责过滤整理，确保交给作者的上下文是精选过的，不是一锅乱炖。

每一篇帖子都作为一个独立的运行文件夹打开，经过一道道检查，发布时把学到的东西写回存储。下一篇因此变得更准。

**一个内容对象的完整生命周期：**

```
已捕获
  → 想法审核（路由：原创 / 再加工 / 改写 / 研究+构思）
  → 简介就绪
  → 起草中
  → 验证
  → 草稿审核
  → 已批准
  → 调度器就绪
  → 已调度
  → 已发布
  → 24小时反馈
  → 72小时反馈
  → 已学习
```

**围绕运行文件夹的各个部分：**

- **策略。** 定位、受众、内容支柱、来源监控列表。整套系统里唯一我自己手动维护的部分。如果让 AI 来写你的定位，你得到的不是定位，是均值。
- **声音。** 声音档案和避免烂内容主文档。每个代理动笔之前要读一遍。
- **存储。** 收件箱、待处理工作板、想法积压、钩子库、证明库、反馈日志。系统的共享记忆，运行文件夹从这里取，也往这里写。
- **模块。** 作者技能包（SKILL.md、参考资料、模板）。系统里每个角色对应一个模块。
- **工作流。** 把内容对象从一个状态推进到下一个状态的剧本：从想法到发布、验证清单、调度器交接、反馈循环。

## 内容进来之后走哪条路

每一条内容进入起草之前，系统会先做一个判断：这是哪种类型的内容？

四条路线，各有各的简介格式和检查方式。

**原创。** 从你自己或你的第二大脑里挖出来的东西——脑子里盘了好几周的想法、笔记、日记、语音备忘录。没有外部来源，完全依赖你的定位、证明和支柱。这条路最考验审美判断。

**再加工。** 把已有的内容延伸出去。从一篇文章里拆出一个推文串，对一篇爆款帖子做自我引用回复，把某段内容单独拎出来发帖。骨架还是你的，换个形式说。

**改写。** 从外部拿素材——值得回应的帖子、值得拆解的文章、有用框架的转录文字——然后用你自己的视角重新表达。简介里会写清楚：保留什么、标注哪些出处、哪些声音规则适用。

**研究+构思。** 先探索话题，研究规律，生成候选角度，不急着起草。这条路线的输出不是帖子，是一个打磨过的想法，或者一组角度，存进库里供后续用。

## 怎么开始：V1 起步

预留1-2小时。做完你不会觉得"搞定了"，但你会真正开始——这才是有意义的状态。这里投入的时间，之后每周都会还回来：下一篇草稿从有上下文的地方开始，不是从零。

**第一步。** 搭骨架。建六个顶级目录：strategy、voice、stores、runs、modules、workflows。在 runs 里加 active/ 和 archive/，其余先留空。

**第二步。** 填策略。打开 strategy/positioning.md、strategy/audience.md、strategy/pillars.md，各写三到五行。支柱是你真正有资格聊的三四个话题。受众是一个具体的人，不是某个用户群体。

**第三步。** 写声音锚点。放进 voice/voice-profile.md：你始终坚持的5条规则、从不用的5种写法、2-3篇你状态最好时写出来的参考帖子。再开一个 voice/avoid-slop.md，用下面提到的8种模式作起始过滤，有新的风格漏洞溜过就往里加。在 stores/proof/ 里放十条具体证明——数字、名字、你做过的项目、亲身经历的例子。

**第四步。** 往 stores/inbox.md 里放十个想法。其中一半应该来自你这个月在私信或通话里已经说过的话，不要现场编。

**第五步。** 为一个想法开运行文件夹。创建 runs/active/2026-05-{你的标签}/，写好 content-object.md（id、状态、格式、支柱）和 brief.md（下面有模板），把这份简介交给写作模型。

**第六步。** 读草稿，关闭循环。模型在同一个运行文件夹里返回 draft-package.md。拿去对照验证清单和避免烂内容主文档过一遍。通过就排期，不通过就带着一条具体意见发回去修。发布后，写 feedback.md，文件夹移进 archive。

## 作者上下文包：最容易踩的坑

很多人在这里出问题——他们把整份品牌文档、整个知识库、整条信息流全塞进一个提示词。

结果是什么？模型写出来的是安全的废话。上下文里没有一件事是真正重要的，模型只好取个均值。

一个精简的包，几乎永远比一个塞满的上下文窗口更有效。

包以 brief.md 的形式存在运行文件夹里，每条内容一份，起草之前写好。

模板是这样的：

```
作者上下文包
─────────────────────
论点：        这篇帖子必须证明的那一句话
读者：        应该存下它的那个具体的人
证明：        我可以用的数字、截图、故事
角度：        意想不到的切入框架
约束：        格式、长度、语气、禁用词汇
声音锚点：    2-3行听起来像我的句子
风险：        什么会让这篇看起来像AI写的，或令人尴尬
未解决问题：  我还不知道的，作者应该标注出来的
```

包填不上，说明你还没有帖子，你只有一个方向感。

## 书签评分标准

草稿进排期之前，先打分。每项0分、1分或2分：

- 为读者省掉了未来的一项任务
- 包含证明（数字、截图、具名案例）
- 提供可复用的收获（模板、清单、框架）
- 有具体的受众和它要解决的任务
- 没有我在场也能用
- 有强有力的截图或视觉内容

满分12分，我的门槛是8分。

低于8分的草稿不是扔掉，而是退回去重新对照包来改。大多数"差"草稿，其实只是某一项偷了懒。修那一项，重新打分，再发。

这套标准也是带新协作者最省事的方式。不用抽象地讲什么叫好内容，把标准给他们，指三篇爆款帖子，让分数说话。

## 避免烂内容主文档

评分标准回答的是"这篇帖子值不值得发"。

避免烂内容文档回答的是"这篇帖子听起来像不像一个人写的"。

每篇草稿发布之前，我都跑一遍这份文档。里面有54种模式，分三个严重程度，每种都有具体的改写示例。它抓住的是这些东西：

- **促销腔**："突破性的"、"颠覆性的"
- **重要性膨胀**："关键时刻"、"有力证明"
- **模糊归因**："专家认为"、"研究表明"
- **假主体**："系统自我复利"、"数据告诉我们"
- **修辞铺垫**："问题是你是否X"
- **碎句堆砌**："没有X。没有Y。没有Z。"
- **破折号泛滥**：目标是零
- **填充副词**："实际上"、"字面上"、"悄悄地"

作者代理起草之前要读这份文档，验证者批准之前也要读。

"这是 AI 写的"和"这是一个用了 AI 的人写的"——两者的区别，就在这里。

## 四个可以直接用的提示词

简短、有边界、留着让你改的。把它们当起点，不是万能公式。

每个对应系统的一个层。粘进 Claude、GPT 或你用的任何模型都行。如果只能留一个，留最后那个事后分析的。

### 提示词1：品牌基础提取

对应：strategy/ + voice/

```
角色
你要帮我搭建个人品牌内容系统的基础层。
你的任务是把我关于工作、受众和声音的原始、半成形笔记，
整理成一组精炼的操作文档，让作者代理可以用它们写出像我的草稿。

输入
我会粘贴一批原始笔记，内容可能包括：
我做什么、我帮助谁、我做出或交付过什么、
我写作时听起来是什么样、我希望吸引什么样的读者，
以及我绝不想听起来像什么。

流程
1. 阅读这些笔记，指出其中的矛盾或缺口。
2. 最多问我5个澄清问题。不要跳过这一步。
3. 等我回答后，按照“输出格式”生成6份材料。
4. 任何你不得不编造或猜测的地方，都标记为“假设”。

输出格式
1. 定位：一句话。陌生人看完我一篇帖子后，应该能复述出来的那句话。
2. 受众：一个具体的人，说明他的角色、处境和利害关系。不要写成泛泛的人群。
3. 内容支柱：3到4个我有资格长期谈论的话题，每个话题附一句可信理由。
4. 声音规则：5件我始终会做的事。
5. 禁用模式：5件我绝不会做的事。
6. 证明库：10个我可以引用的具体证明，例如数字、名字、已交付项目、亲身经历。

规则
- 如果某一部分很泛泛，标记为“缺失”，并告诉我你需要我补什么。
- 不要编造数字、客户或项目。
- 尽可能使用我自己的表达，直接从笔记里提取短语。
- 输出应该能放进一页。重点就是要紧。
```

### 提示词2：书签评分

对应：stores/ideas/（想法门）

```
角色
你是一名内容批评者，读过10000篇高书签率帖子，
也读过1000000篇没人记得住的帖子。
你能逐行判断，是什么让读者保存一篇帖子，而不是直接划走。

输入
一条帖子想法或一份草稿。
它可以是一句话论点、粗略提纲，也可以是一份完整草稿。

流程
1. 先快速通读一遍，获得第一印象。
2. 按下面的评分表，每一项打0分、1分或2分。
3. 汇总为12分制总分。
4. 如果低于8分，指出最能拉高分数的那一项，并说明怎么改。

评分表
- 为读者省掉未来的一项任务
- 包含证明，例如数字、截图、具名案例
- 给出可复用的收获，例如模板、清单、框架
- 有具体受众，以及明确的待完成任务
- 即使我不在场，读者也能使用
- 有强有力的截图或视觉内容

输出格式
- 总分：X / 12
- 最强项：[项目]（为什么）
- 最弱项：[项目]（一句话说明具体怎么修）
- 结论：发布 / 修改后重评 / 放弃

规则
- 不要为了鼓励我而抬高分数。
- 不要建议“让它更有吸引力”。直接告诉我要加什么或删什么。
- 如果这个想法明显低于8分，直接说“放弃”，并说明原因。
```

### 提示词3：作者上下文包

对应：runs/active/{标签}/brief.md

```
角色
你是我内容系统里的生产负责人。
你的任务是把一个已批准的想法，整理成作者上下文包。
这个包应该是一份精炼、有形状的简介：
既给作者足够的信息写出锋利草稿，又不要把我的整个知识库都塞给模型。

输入
- 一个已批准的想法：论点、格式、目标读者。
- 我的基础文件线索：定位、受众、内容支柱、声音规则、禁用模式、证明库。
- 这篇帖子专属的素材：截图、转录稿、对话笔记等。

流程
1. 用一句话复述这个想法，确认你理解正确。
2. 只提取这篇帖子需要用到的基础文件片段。
3. 填写下面的上下文包模板。
4. 任何填不出来的字段，都写“缺失”，并明确列出你需要我补什么。

输出格式
论点：        这篇帖子必须证明的一句话
读者：        应该保存它的那个具体的人
证明：        我被允许使用的数字、截图、故事
角度：        意想不到的切入框架
约束：        格式、长度、语气、禁用词
声音锚点：    2-3行听起来像我的句子
风险：        什么会让它读起来像烂内容或令人尴尬
未闭合问题：  我还不知道、作者应该标注出来的内容

规则
- 越小越好。目标控制在400-900个token。
- 不要粘贴我的完整基础文件，只抽取这篇帖子需要的片段。
- 允许拒绝。如果你没有足够上下文写出锋利的包，
  就说“我没有足够上下文”，并只问我最关键的那个缺失信息。
```

### 提示词4：病毒式事后分析（重点用这个）

对应：runs/active/{标签}/draft-package.md，审核前最后一关

```
角色
请你假设现在已经是一周后。
你正在阅读一篇已经超过100万浏览、10000次书签的帖子。
你不是来写这篇帖子的。
你要做的是事后解释：它为什么能打中读者。

输入
一份已经准备进入最后一轮修改的草稿。

流程
1. 阅读草稿。
2. 指出具体是哪几行真正起了作用。
3. 命名它使用的钩子动作。
4. 指出让它可信的证明。
5. 指出读者会截图保存的那一行。
6. 指出让它值得收藏的那一行。
7. 指出会让人回复，或转发给朋友的那一行。

输出格式
- 钩子动作：[原文中的具体行]（为什么有效）
- 可信度：[原文中的具体行]（为什么读者会相信）
- 值得截图的行：[原文中的具体行]
- 值得收藏的行：[原文中的具体行]
- 回复或分享触发点：[原文中的具体行]
- 最弱部分：[原文中的具体行]（发布前要修什么）

规则
- 不要说“好帖”。不要说“钩子很强”。
  要指出具体的行；如果指出不出来，就承认指出不出来。
- 如果上面任何一个类别找不到对应的具体行，直接说清楚。
  那就是发布前需要修的地方。
- 这个提示词的目的，是不让模型躲在泛泛表扬后面。
  逼它指出具体机制。
```

最后这个，是整套系统里杠杆最大的提示词。

模型不能说"钩子很强"或"见解很好"，它必须指出具体的行。大多数通过验证者的草稿，用这个提示词跑还是会暴露问题——两者之间的差距，就是真正的编辑所在。

它跑在这篇文章上，是这样的：

```
> /viral-postmortem 分析一篇关于用AI搭内容系统的草稿

钩子动作：
  “内容好不好，取决于背后的人和系统”
  直接点出原因，没有铺垫，没有前戏。

可信度：
  “一个几乎停更的账号，两周拿到500万曝光”
  时间线具体，起点诚实，证明有锚点。

值得截图的行：
  “工作单元不是‘再发一条帖子’，
   而是一个读者想存下来的东西。”

值得收藏的行：
  “如果你填不出这个包，说明你还没有帖子，
   你只有一个模糊的感觉。”

回复或分享触发点：
  “大脑就是系统。”

最弱部分：
  “两个模型，两个角色”这一节。
  opus 4.7 + gpt-5.5 这种具体型号很快会过时。
  应该泛化成原则，删掉模型名。
```

## 两个模型，两个角色

大量跑这套系统之后，我撞上了一件回头想挺显然的事：

写作这件事，和运行系统这件事，是两种不同的工作。它们适合不同的模型。于是我不再用一个模型包打天下。

**作者（opus 4.7）负责：**
- 品味
- 节奏
- 信息密度
- 声音
- 真正的草稿

**编排者（gpt-5.5）负责：**
- 在各层之间路由
- 为作者打包合适的上下文
- 决定哪些内容传进去
- 跑验证者
- 向发布层交接

## 系统住在哪里

这套系统要跑起来，编排者必须能真正干活——读写文件、调用工具、跑检查、按计划交稿。所以真正的问题不是用哪个应用，而是系统住在哪里。

两种跑得干净的方案。

**1. VPS + Claude Code。**

租一台小服务器，把 /content-os 放进 git 仓库，让 Claude Code 在上面跑工作流。你拿到完全控制权，定时任务跑每周复盘，脚本跑验证者，还有一个能接触同一批文件的 AI 编码助手。你喜欢自己掌控技术栈，对服务器不陌生，选这个。

**2. Hermes 代理——我现在用的这个。**

它就是为这种形状的工作设计的：代理、技能、工具调用、文件和 git 操作、浏览器和搜索、定时任务，以及跨整个工作流持久化的上下文。把它指向你的基础文件、输入来源和包模板，编排层负责衔接各步骤之间的工作，我的时间留在写作和编辑上。草稿在发布之前还是到我手里。

两者都不是必须的。核心是：把系统放在一个编排者能读写文件、调用工具、跑验证者、把批准的帖子交给 Postiz 的地方。选哪个，看你怎么习惯工作。

## 反馈循环，才是护城河

大多数人在发布之后就停了。

系统真正开始产生价值，是从发布之后。

我每周看四件事：

- 浏览量
- 书签数
- 书签率
- 回复数

> 最关键的是书签率。它告诉我哪些帖子真正赢得了"以后还会用到"，而不只是被刷过去。

爆款帖子，连带数据，一起存进输入，作为下次的参考案例。表现差的帖子，用来更新声音规则、禁用模式或想法过滤器。

下一个包会更准——因为这周又学到了一点东西。

这就是护城河。不是工具，不是模型，是每一次反馈都让系统稍微聪明一点点的那个循环。
