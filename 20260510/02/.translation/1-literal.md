## 不可妥协的原则

在任何机制之前，整个系统运行所依赖的规则：

**永远不要发布未经编辑的内容，每个草稿都要手动完成。** 系统是加速器，不是自动驾驶仪。作为自动化使用，它会衰减。

目标不是用提示词伪造一个声音。而是从你真实的写作中构建一个可重用的运营资产。一次完成这项工作，保持其最新状态，模型产生的每个草稿都将从更接近你的地方开始，这样你的时间就可以用于打磨想法。

## "值得书签"实际上意味着什么

书签是读者对未来自己的一个小承诺。它说"我以后还会需要这个。"这比点赞的门槛高得多，在算法中的表现也不同。

我不会夸大排名数学，但作为每周运行这个的营销人员，这个模式是一致的。书签是对未来实用性的投票，获得书签的帖子在发布日期很久之后仍然会出现在信息流中。

在我发布任何内容之前，我会问这个帖子是否看起来像这些之一：

- 清单
- 蓝图
- 文件夹结构
- 模板
- 框架
- 分步工作流程
- 带有结论的截图证明
- 前后对比
- 可重用的心智模型

如果草稿不像其中任何一个，它通常不应该被发布。

上下文存在于两个地方。

**信号层**是你引入的所有外部内容：你本周保存的书签，你监控列表上创作者的内容，你喜欢的一篇文章。

**知识图谱**是你已经拥有的所有内部内容：你的个人操作系统、笔记、日记、语音备忘录，以及你已经发布的内容档案。

路由决定哪个来源为简介提供信息，策略+声音+存储位于两者和作者之间，以便对上下文进行策划。

每篇帖子、文章、线程或活动都作为新的运行文件夹打开。该文件夹是内容对象。它从系统的共享部分提取，通过门控的生命周期移动，并在发布时将学到的内容写回。

**一个内容对象的生命周期：**

```
已捕获
  → 想法审核（路由：原创/再加工/改写/研究+构思）
  → 简介就绪
  → 起草中
  → 验证
  → shann草稿审核
  → 已批准
  → 调度器就绪
  → 已调度
  → 已发布
  → 24小时反馈
  → 72小时反馈
  → 已学习
```

**围绕运行文件夹的内容：**

- **策略。** 定位、受众、支柱、来源监控列表。我唯一手动编辑的层。如果你让AI写你的定位，你没有定位。你有的是平均值。
- **声音。** 声音档案和避免烂内容主文档。每个代理在起草任何一行之前都要读。
- **存储。** 原始输入的收件箱、需要关注的工作板、想法积压、钩子库、证明库、反馈日志。运行文件夹读取和写回的共享记忆。
- **模块。** 作者技能（SKILL.md、参考资料、模板）。生产代码。系统给予的每个角色一个模块。
- **工作流程。** 将内容对象通过其状态移动的剧本：从想法到发布、验证者清单、调度器交接、反馈循环。

## 四条路线

在内容对象进入起草之前，想法门做出一个决定：这是什么类型的内容？

四条路线，每条都有自己的简介、自己的参考资料和自己的门控：

**原创。** 直接从你或你的第二大脑创作（个人操作系统、笔记、日记、语音备忘录、你已经考虑了好几周的想法）。简介依赖于你的基础：定位、证明库、支柱。没有外部来源。高品味投入。

**再加工。** 将已拥有的内容进行延伸。内容系列衍生品、从你的一篇文章中提炼出的线程、对一篇成功帖子的自我引用回复，或者从你已经发布的内容中提取单行文字的推文。骨架是你的。格式改变。

**改写。** 从信号层获取外部来源材料（值得回应的推文、值得拆解的文章、带有有用框架的转录文字），并通过你的观点和声音进行转化。简介明确说明保留什么、归功于什么，以及适用哪些声音规则（根据避免烂内容主文档）。

**研究+构思。** 探索一个话题，研究模式，在任何起草开始之前生成候选角度。输出不是帖子。它是一个被打磨的想法或一组角度，反馈到存储/想法/中以供以后生产。

![](images/img-07.jpg)

每条路线仍然产生一个运行文件夹，路线在content-object.md中声明：

```
runs/active/2026-05-bookmark-flywheel/
  content-object.md   # 路线、当前状态、下一步操作
  idea.md             # 想法门决定
  brief.md            # 作者交接（用于原创、再加工、改写）
  draft-package.md    # 渲染草稿、验证者输出、审核笔记
  feedback.md         # 24小时/72小时学习
```

## 这之前是什么

在当前内容操作系统之前，我构建了一个四代理系统，运行了同一循环的一个版本：研究者、想法创造者、作者和一个在它们之间路由的编排者。每个代理都有自己的记忆，跨会话持久化。

我在claude code中构建了整个东西，使用markdown提示词、数据库和CLI工具。

![](images/img-08.jpg)

它有效。但它构建过度了。四个代理用于结构上只是三个生产步骤和一个反馈循环的东西。运行它让我学到了大多数关于代理群的博客文章遗漏的东西：

> 代理数量不是杠杆。给作者提供信息的知识层才是。

当前设置是更精简的版本。更少的代理，更多的工作流程，相同的循环，更清晰的输出。

## 你的V1设置

计划1-2小时的前期工作。你不会完成。你将开始，这是唯一值得处于的状态。你在这里花费的时间会以每周节省数小时的形式得到回报，因为下一个草稿从上下文开始，而不是从零开始。

**步骤1。** 搭建结构。创建六个顶级目录：strategy、voice、stores、runs、modules、workflows。在runs中，添加active/和archive/。暂时让其余部分大致保持空白。

**步骤2。** 填写策略。打开strategy/positioning.md、strategy/audience.md和strategy/pillars.md。每个三到五行。支柱是你已经赢得了谈论权利的三到四个话题。受众是一个具体的人，不是一个细分市场。

**步骤3。** 写声音锚点。放入voice/voice-profile.md（你始终遵循的5条规则，你从不使用的5种模式，2-3篇在你最犀利的那天听起来像你的参考帖子），并开始你自己的voice/avoid-slop.md（使用下面章节中的八种模式作为起始过滤器，每次草稿让一个明显特征溜过时就添加）。在stores/proof/中放入十个具体证明，比如数字、名字、你已经发布的项目、亲身经历的例子。

**步骤4。** 在stores/inbox.md中放入十个想法。其中一半应该来自你这个月在私信或通话中已经说过的事情，而不是当场编造的。

**步骤5。** 为一个想法打开一个运行文件夹。创建runs/active/2026-05-{你的标签}/。在里面，写content-object.md（id、状态、格式、支柱）和brief.md（作者上下文包模板，见下文）。将简介交给你的写作模型。

**步骤6。** 读取草稿并关闭循环。模型在同一运行文件夹中返回draft-package.md。对照验证者和避免烂内容主文档进行检查。批准并排队，或者带着一个具体的注释发送回去。发布后，写feedback.md并存档文件夹。

## 作者上下文包

这是大多数人出错的部分。他们将整个品牌文档、整个知识库和整个信息流都放入一个提示词中。

模型写出安全的废话，因为上下文中没有任何东西是承重的。一个紧凑的包几乎总是比一个巨大的上下文窗口更好。

包作为brief.md存在于运行文件夹中。每个内容对象一个包，在起草开始前写好。

复制这个模板：

```
作者上下文包
─────────────────────
论点：        帖子必须证明的一句话
读者：        应该保存它的具体人物
证明：        我被允许使用的数字、截图、故事
角度：        意想不到的框架
约束：        格式、长度、语气、禁用短语
声音锚点：    2-3行听起来像我的句子
风险：        什么会让这看起来像烂内容或令人尴尬
未解决问题：  我还不知道的，作者应该标记的
```

## 书签评分标准

在草稿进入排期之前，对其评分。每项零分、一分或两分：

- 为读者节省了未来的一项任务
- 包含证明（数字、截图、具名示例）
- 提供可重用的收获（模板、清单、框架）
- 有具体的受众和需要完成的工作
- 没有我在场也可以应用
- 有强有力的截图或视觉内容

满分12分。我个人的门槛是8分。低于8分的内容回到包中，而不是丢弃。大多数"差"的草稿是跳过了标准中某一行的好草稿。修复那一行，重新评分，发布。

这个标准也是培训新合作者最经济的方式。你不必抽象地教授品味。你把标准给他们，指出三篇获奖帖子，让分数说话。

## 避免烂内容主文档

标准告诉你一篇帖子是否值得保存。避免烂内容文档告诉你一篇帖子听起来是否像一个人写的。

我在每篇草稿发布前都会对照一个文档运行。它有54种模式，分为三个严重程度层级，每种都有具体的改写。它能捕捉到以下内容：

- 促销性语言（"突破性的"、"改变游戏规则的"）
- 重要性膨胀（"关键时刻"、"证明"）
- 模糊归因（"专家认为"、"研究表明"）
- 虚假能动性（"系统复合"、"数据告诉我们"）
- 修辞性设置（"问题是你是否X"）
- 断奏碎片化（"没有X。没有Y。没有Z。"）
- 破折号过度使用（目标是零）
- 填充副词（"实际上"、"字面上"、"悄悄地"）

这是我的作者代理在起草之前加载的文档，也是我的验证者在批准之前加载的文档。

这就是"AI写了这个"和"一个碰巧使用AI的人写了这个"之间的区别。

## 你可以复制的四个提示词

简短、有范围、旨在被编辑。把它们当作起始形状，而不是魔法咒语。

每个都映射到系统的一个层。将它们放入claude、gpt或你使用的任何前沿模型中。如果你只保留一个，保留最后的事后分析那个。

### 提示词1：品牌基础提取

映射到：strategy/ + voice/

```
角色
你正在帮助我构建个人品牌内容系统的基础层。你将把关于我的
工作、受众和声音的粗糙、半成形的笔记转化为一套紧凑的操作
文档，我的作者代理可以用这些文档以我的声音起草。

输入
我将粘贴涵盖以下内容的原始笔记：我做什么，我帮助谁，我已
经发布或构建了什么，我写作时的声音，我想要的读者类型，以
及任何我拒绝发出的声音。

流程
1. 阅读笔记。注意任何矛盾或空白。
2. 向我最多问5个澄清问题。不要跳过这一步。
3. 一旦我回答，按输出格式产生六个产物。
4. 标记任何你必须发明或猜测的内容。标记为"假设"。

1. 定位。一句话。陌生人在看了我的一篇帖子之后应该能够复述的话。

4. 声音规则。我始终做的5件事。
5. 禁用模式。我从不做的5件事。

- 如果某个部分是通用的，标记为"缺失"并告诉我你需要什么。
- 不要发明数字、客户或项目。

```

### 提示词2：书签评分

映射到：stores/ideas/（想法门）

```
角色
你是一个读过10,000篇高书签帖子和1,000,000篇可被遗忘帖子的
批评者。你能逐行判断什么让读者保存一篇帖子，什么让他们滚
动跳过。

输入
一个帖子想法或草稿。可能是一行论点、粗略草图或完整草稿。

流程
1. 通读一遍看表面阅读。
2. 对下面标准的每一行评0、1或2分。
3. 总分满分12分。
4. 如果低于8分，说出哪一行最能提高分数，以及如何提高。

标准
- 为读者节省了未来的任务
- 包含证明（数字、截图、具名示例）
- 提供可重用的收获（模板、清单、框架）
- 有具体的受众和需要完成的工作
- 没有我在场也可以应用
- 有强有力的截图或视觉内容

输出格式

- 最强项：[行] （原因）
- 最弱项：[行] （具体修复，一行内）
```

### 提示词4：病毒式事后分析（用这个）

映射到：runs/active/{标签}/draft-package.md，审核前的最终检查

```
角色
你正在阅读一篇一周后已经获得100万次浏览和1万次书签的帖子。
你不是在写它。你是在事后解释为什么它会成功。

输入
一个准备进行最终检查的草稿。

流程
1. 阅读草稿。
2. 指出做了工作的具体行。
3. 命名钩子动作。
4. 命名使其可信的证明。
5. 命名读者会截图的那行。
6. 命名使其值得保存的那行。
7. 命名让某人回复或发送给朋友的那行。

输出格式
- 钩子动作：[精确行]（为什么有效）
- 可信度：[精确行]（为什么读者相信）
- 可截图的行：[精确行]
- 值得保存的行：[精确行]
- 回复或分享触发器：[精确行]
- 最弱的部分：[精确行]（发布前需要修复什么）

规则
- 不要说"好帖子"。不要说"强钩子"。指出具体的行，或承认你做不到。
- 如果你无法为上述任何类别指出一行，直说。那就是我需要在发布前修复的行。
- 整个重点是模型不能躲在通用赞美后面。强迫它指出机制。
```

这最后一个是系统中杠杆最大的提示词。模型不能说"强钩子"或"好见解。"它必须指出具体的行。大多数通过验证者的草稿仍然无法通过这个提示词，两者之间的差距是真正的编辑所在。

这是它在这篇文章上运行的样子：

```
> /viral-postmortem on draft about content system with AI

钩子动作：
  "内容的质量只取决于背后的人和系统"
  直接命名原因，没有铺垫，没有前言

可信度：
  "两周内500万次曝光，从一个几乎沉寂的账号开始"
  具体时间线，诚实的起始状态，固定了证明

可截图的行：
  "工作单元不是'又一篇帖子'。它是读者想要保留的东西。"

值得保存的行：
  "如果你无法填写包，你还没有帖子。你只有一种感觉。"

回复或分享触发器：
  "大脑就是系统。"

最弱的部分：
  "两个模型，两个角色"章节
  opus 4.7 + gpt-5.5的具体性会很快过时
  泛化原则，去掉模型名称
```

## 两个模型，两个角色

一旦我开始大量运行系统，我遇到了回顾起来很明显的事情：

写作的工作和运行系统的工作是不同的工作，它们奖励不同的模型。所以我停止了为所有事情使用一个模型。

目前我的分工看起来像这样：

**作者（opus 4.7）处理：**
- 品味
- 节奏
- 压缩
- 声音
- 实际草稿

**编排者（gpt-5.5）处理：**
- 在层之间路由
- 为作者打包正确的上下文
- 决定传入什么
- 运行验证者
- 向发布层的交接

## 在哪里运行系统

像这样的系统只有在编排者能够完成工作时才有效。它需要读写文件、调用工具、运行检查，并按计划将已批准的草稿交给发布层。所以真正的问题不是你使用哪个应用程序，而是系统在哪里运行。

我见过两种干净运行的设置。

**1. 带有claude code的VPS。**

你租一个小型服务器，将你的/content-os文件夹放在git仓库中，让claude code在其上运行工作流程。你获得完全控制权，一台你拥有的机器，用于每周审查的定时任务，用于验证者的脚本，以及一个能接触你同样文件的AI编码器。如果你喜欢拥有堆栈且对服务器感到舒适，这是你的路线。

**2. hermes代理，这是我运行的。**

它是为这种形状的工作构建的：代理、技能、工具访问、文件和git操作、浏览器和搜索、定时任务，以及跨整个工作流程持久化的上下文。将它指向你的基础、你的输入和你的包模板，编排层在步骤之间承担负担，这样我就可以把时间花在写作和编辑上。草稿在任何东西发布之前仍然来到我这里。

两者都不是必需的。原则才是重要的：在你的编排者可以读写你的文件、调用它需要的工具、运行验证者并将已批准的帖子交给postiz的地方托管系统。选择最适合你工作方式的那个。
