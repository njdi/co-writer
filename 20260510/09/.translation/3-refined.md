# 为什么我们应该停止替AI智能体设计脚手架

**作者：** Kangwook Lee ([@Kangwook_Lee](https://x.com/Kangwook_Lee))  
**日期：** 2026年5月9日  
**来源：** [Why We Should Stop Designing Harnesses for AI Agents](https://x.com/Zephyr_hg/status/2052925157606568217)

*（如果你不熟悉"马与挽具的类比"，建议先读我最近的那篇文章）*

这篇文章，我想说清楚一件事：为什么我们都应该停止替AI智能体手工设计运行框架（harness）。

**简单说：** AI智能体最好的框架，可能已经不是人类替它设计的那个，而是它自己搭出来的那个。如果我们还在替它设计，我们会成为整件事的瓶颈。

## 框架划定了一条门槛线

先提出一种看问题的新角度——关于框架和模型之间的关系。

现在大多数人的思路是：**模型固定，换框架**。

我想换一个角度：**框架固定，让模型能力去变化**。

这样一来，对于同一个任务，框架会在模型能力上划出一条门槛线。

模型比门槛聪明，就能用这套框架解决问题。低于门槛，就不行。

我们把这条门槛线记为 θ(h)，它由框架 h 决定。

## 三段位框架

这个角度在比较两套框架时特别有用。

一套是好框架（h_good），一套是差框架（h_bad）。

好框架让任务更容易上手，门槛线低；差框架要求模型更聪明才能完成同样的任务，门槛线高。

这样自然就出现了三个段位。

![](images/img-02.jpg)

**第一段位。** 模型能力太弱，不管用哪套框架都搞不定。

**第二段位。** 模型能力中等，能解决问题——但前提是用好框架。

**第三段位。** 模型足够强，不管用什么框架都能搞定。

注意：框架好不好，只在中间段位才真正有差别。模型不弱但也不算强的时候，好框架才能发挥作用。

我们把这个叫做"三段位框架"。

## 人类造的框架 vs AI造的框架

回到马与挽具的类比。

马和AI之间有一个关键区别。

马没法给自己造挽具。AI可以。

现在考虑两种框架：

- **h_human：** 人类造的框架。
- **h_ai：** AI造的框架。

过去这几年，情况是这样的：

```
h_good = h_human
h_bad  = h_ai
```

人类造的框架比AI造的好。

所以在三段位框架里，中间段位的意思是："AI模型只有在人类造的框架下才能跑起来。"

那时候，我们精心手工搭建的智能体循环、工具调用方式、记忆结构、规划脚手架，效果确实比AI自己鼓捣出来的更好。搭一个好智能体，意思就是替它搭一套好框架。

我自己在那段时间花了大量精力在这件事上。设计了当时最先进的聊天机器人框架（ACL'23）、基于视觉语言模型的图像聚类智能体（ICLR'24）、PUBG的游戏伴侣智能体Ally，还有代码智能体Terminus-KIRA。

挺好玩的。有那么一段时间。

## 但前提变了

AI越来越会写代码，越来越会调试，越来越快地在系统层面迭代。而且迟早，它在搭框架这件事上会比我们做得更好。

局面就翻过来了：

```
h_good = h_ai
h_bad  = h_human
```

这意味着什么？套用三段位框架，中间段位现在的含义变成："AI模型只有在AI造的框架下才能跑起来。"

也就是说，如果我们还在用"人类造的框架"，处于中间段位的模型就发挥不出它真正的潜力。

我们最近有一项工作，Meta Harness（由@yoonholeee主导），已经验证了这一点：在中等规模的模型上，AI在迭代优化框架这件事上比人类做得更好。

为了看清全局，加一条时间轴。假设AI的代码能力随时间单调增长，你会得到下面这张二维相图——我们正处在一次重要相变的临界点上。

![](images/img-03.jpg)

"黄色区域"的出现，也为"被埋没的天才"假说提供了另一种解释：如果我们一直用人工设计的框架，底层模型的真实潜力就始终没机会浮出水面。

## 结论

模型极其强大时，框架好不好几乎不重要。随便一个循环加几个工具，就能搞定大部分事情。

模型太弱时，框架好不好也不重要。没有框架能救它。

问题在于中间那一段。模型还不错，有能力，但没有合适的支撑就跑不起来。这时候，框架的质量是决定性的。

而在不久之前，这个好框架只能靠人来造。

现在不一样了。要把中间段位的模型用到极致，我们得退一步，不再主导框架这件事。

让聪明的马自己造挽具吧。
