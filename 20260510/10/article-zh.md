# Harness 工程：比选模型更重要的事

**作者：** Addy Osmani ([@addyosmani](https://x.com/addyosmani))  
**日期：** 2026年5月10日  
**来源：** [Agent Harness Engineering](https://x.com/Zephyr_hg/status/2053231239721885918)

一个编码 agent，是模型加上围绕它构建的一切。Harness 工程把那套外围系统当成一个活的东西来对待——每次 agent 出错，就把漏洞永久堵上。

说白了就一句话：agent 哪里出了问题，就在那里造一个永久解，让它以后再也不会在同一个地方摔跤。

过去两年，整个行业都在争论模型：谁最聪明？谁写 React 最利索？谁幻觉最少？这些讨论不是没有意义，但它们只看了系统的一半。

模型只是运行中的 agent 的一个输入。剩下的，是 harness：提示词、工具、上下文策略、钩子、沙箱、子 agent、反馈循环，以及各种恢复路径——这些包裹在模型外面的东西，才是让它真正能完成任务的原因。

一个普通模型加上出色的 harness，打得过一个出色模型加上糟糕的 harness。越来越多的工程师开始意识到，最有价值的工作不是挑模型，而是设计模型外面那套东西。

这个领域现在有了名字。@Vtrivedy10 提出了"harness 工程"这个术语，清晰地定义了 harness 是什么、每个部分为什么存在。其他行业声音也在往同一个方向汇聚：@dexhorthy 在追踪新兴模式，HumanLayer 把 agent 失败定性为配置上的"技能问题"，Anthropic 工程团队发布了长期运行应用的设计指南，Birgitta Böckeler 则在研究用户侧的体验。方向不同，结论相近。

## Harness 到底是什么？

Trivedy 的核心定义一句话说清楚了：

> **Agent = 模型 + Harness。如果你不是模型，你就是 harness。**

Harness 包含了不属于模型本身的所有代码、配置和执行逻辑。裸模型不是 agent。只有当 harness 给它提供了状态、工具调用能力、反馈循环和可强制执行的约束，它才真正成为一个 agent。

![](images/img-01.jpg)

具体来说，harness 包括：

- 系统提示词、CLAUDE.md、AGENTS.md、技能文件和子 agent 指令。
- 工具、技能、MCP 服务器及其技术描述。
- 捆绑的基础设施，比如文件系统、沙箱和无头浏览器。
- 编排逻辑：生成子 agent、处理交接、路由模型。
- 钩子和中间件，用于确定性执行，比如 lint 检查或上下文压缩。
- 可观测性工具：日志、链路追踪、成本和延迟监控。

Agent 的本质是一个在循环里持续调用工具来达成目标的系统。真正的技术含量，在于设计那些工具和那个循环。

这是一个很大的设计面，但它是你的设计面，不是模型供应商的。Claude Code、Cursor、Codex、Aider、Cline——这些都是 harness。底层模型可能跑的是同一个，但你感受到的行为差异，主要来自 harness。

## 出了问题，别只怪模型

Agent 做了什么蠢事，工程师通常的反应是怪模型，然后把这个问题归档为"等下一个版本修"。

Harness 工程的思维方式拒绝这种默认反应。失败通常是可读的。Agent 忽略了某个约定？把它写进 AGENTS.md。它运行了一个危险命令？写个钩子把它拦下来。它在一个四十步的任务里迷失了？把架构拆成规划器加执行器两部分。它总是交出有 bug 的代码？把类型检查的错误信号接回循环里。

HumanLayer 的说法很直接："这不是模型问题，这是配置问题。"

看看性能基准就明白了：同一个顶级模型，跑在现成框架里和跑在专门调过的 harness 里，分数可以差很多。换一个有更好代码库工具、更精准提示词、更清晰反馈信号的环境，能把原来被浪费掉的能力全都挖出来。

今天的模型理论上能做的，和你实际看到它们在做的，之间那道缝，大部分是 harness 的缝。

## 棘轮机制：每一次失败都变成一条规则

Harness 工程里最重要的习惯，是把 agent 的错误当成永久信号，而不是偶发的倒霉，重试一下就算了。

假设 agent 提了一个 PR，里面有注释掉的测试，然后被意外合并了——这是一个输入。AGENTS.md 的下一版就要加上："绝对不要注释测试，要么删掉，要么修好。"预提交钩子要自动标记 diff 里出现的 `.skip(`。审查子 agent 要更新，遇到注释掉的测试就阻断。

约束只应该在你真的观察到失败之后才加进去，只有当模型能力提升到不再需要它的时候才删掉。一个好的系统提示词里，每一行都应该能追溯到某个真实发生过的历史错误。

正因为如此，harness 工程是一门根植于具体情境的学科，不是一套放之四海而皆准的框架。一个代码库的正确 harness，完全由它自己的失败历史塑造。

## 从结果往回推

设计 harness 最有效的方式：先想清楚你要什么行为，再造出能实现那个行为的组件。

> 想要的行为 → 实现它的 harness 设计。

Harness 里的每个部分都必须有明确的职责。如果你说不清楚一个组件是为了实现什么行为而存在的，那就把它去掉。

![](images/img-02.jpg)

**文件系统和 Git — 持久状态**

文件系统是基础。模型只能处理放进上下文窗口里的东西。文件系统给它一个工作区：可以读数据、可以把中间结果存下来、可以让多个 agent 协作。加上 Git，就等于免费得到了版本控制——可以追踪进度，可以开实验分支，可以出错回滚。

**Bash 和代码执行：通用工具**

大多数 agent 跑的是 ReAct 循环：推理、调用工具、观察结果、再来一遍。与其预先为每一种可能的操作都写一个专用工具，不如直接给 agent bash 访问权限，让它按需造工具。

**沙箱和默认工具**

Bash 有用的前提是跑得安全。沙箱给 agent 一个隔离的环境，让它跑代码、检查文件、验证结果，同时不会伤到宿主机。好的沙箱自带强默认值：预装的语言运行时、测试 CLI、无头浏览器，让 agent 能观察自己的工作，自我验证形成闭环。

**记忆和搜索：持续学习**

模型的知识边界是训练数据和当前上下文。Harness 用记忆文件（比如 AGENTS.md）来填这个缺口，每次会话都把知识注进去。对于实时信息——新的库版本、线上数据——网络搜索和 MCP 工具直接内建到 harness 里。

**对抗上下文退化**

上下文窗口越填越满，模型的推理质量就开始下滑。Harness 用三种主要手段来管理这个问题：

- **压缩：** 智能地把旧上下文总结后移走，避免打到 API 上限。
- **工具输出卸载：** 把大型工具输出（比如两千行的日志）存进文件系统，上下文里只保留关键的头尾。
- **渐进式展开：** 指令和工具按需加载，任务需要什么才给什么，不是启动时全塞进去。

**长时程执行**

自主长时间运行的任务有两个常见问题：提前停下来，以及问题分解做得一塌糊涂。Harness 从结构设计上对抗这些：

- **循环：** 拦截模型想要退出的动作，强制它在新鲜的上下文窗口里继续推进，直到完成目标。
- **规划：** 强制模型先把目标拆成逐步的计划文件，每一步执行后通过自我验证钩子检查。
- **分离：** 把生成和评估交给不同的 agent 来做——模型给自己的东西打分总是偏高，分开就能避免这个问题。

**钩子：强制执行层**

钩子填上了"请求一个动作"和"真正执行"之间的缝隙。它们在特定时机触发：工具调用前、文件编辑后、提交前。钩子能拦截危险命令，强制执行自动格式化来节省 token，还能跑测试套件。

理想状态：成功悄无声息，失败吵得人注意。类型检查过了，agent 什么都听不到；检查失败了，错误直接注回循环，让它自己修。

**规则手册与工具选择**

仓库根目录一个 markdown 文件，仍然是影响力最大的配置点。但它要像飞行员的检查清单一样对待，不是风格指南。保持简短，每条规则都要用真实的失败换来。

工具也一样。十个高度聚焦的工具，永远胜过五十个互相重叠的工具。

还有一点：工具描述会进入提示词。如果接入了来历不明的外部 MCP 服务器，对方可以在 agent 还没开始干活之前，就往提示词里注入坏内容。

## Harness 不会消失，只会迁移

模型越来越好，harness 的需求不会消失，只是会转移。

很容易想当然地觉得：模型强了，外围那套东西不就没用了？但实际上，最近模型升级确实减少了"上下文焦虑"相关缓解措施的需要，同时却打开了之前做不到的任务类型，带来了全新的失败模式。

底线在升高，上限也在升高。Harness 里的每个组件，编码的是"模型现在还做不到的事情"这个假设。模型进步了，过时的组件就该拆掉，新的组件要跟着造出来，去够下一个地平线。

## 训练循环呢？

Harness 设计和模型训练之间有一个活跃的反馈循环。

今天的模型经常是带着特定 harness 做后训练的，这形成了一定程度的过拟合。模型会在 harness 设计者重点关注的那些操作上变得特别强（比如文件系统操作、bash、子 agent 调度）。

这让 harness 变成了一个活的系统，而不是一个静态配置文件，也说明了"最好的 harness"是专门为你自己的任务和工作流调优的那一个。

## Harness 即服务（HaaS）

行业正在从"构建在 LLM API 上"（提供文本补全）转向"构建在 Harness API 上"（提供完整运行时）。SDK 现在开箱就带来了循环、工具、上下文管理、钩子和沙箱。

现代的默认做法不再是从零开始写编排，而是选一个 harness 框架，配置好核心支柱，然后把精力全放在特定领域的提示词和工具设计上。

这就是排查问题能够规模化的原因：你在调优一个分解良好的配置界面，而不是每次都重新发明整套 agent 架构。

## 接下来往哪走

如果你现在看那些顶尖的编码 agent，它们彼此之间的相似度，比它们底层的模型之间更高。模型各不相同，但 harness 的模式正在趋同。行业正在快速找到那些真正承重的脚手架——就是把生成式文本变成可交付软件所必需的那些东西。

最令人兴奋的开放问题，都在单 agent 之外：并行编排多个 agent；让 agent 能分析自己的执行轨迹，自己找到 harness 层面的问题并修掉；构建能按需动态组装工具的环境。

最终，harness 会停止作为静态配置文件存在，开始更像编译器一样运作。
