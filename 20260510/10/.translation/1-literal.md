# Agent 脚手架工程

**作者：** Addy Osmani ([@addyosmani](https://x.com/addyosmani))  
**日期：** 2026年5月10日  
**来源：** [Agent Harness Engineering](https://x.com/Zephyr_hg/status/2053231239721885918)

一个编码 agent 是模型加上围绕它构建的一切。脚手架工程将这套脚手架视为一个活的制品，每次 agent 出错时就对其进行收紧。

简单来说：每当 agent 失败，你就工程化一个永久解决方案，让它再也不会犯同样的错误。

在过去两年里，业界一直在争论模型：哪个最聪明，哪个写的 React 最干净，或者哪个幻觉最少。虽然这种讨论很重要，但它忽略了系统的另一半。

模型只是一个运行中的 agent 的一个输入。其余的是脚手架：提示词、工具、上下文策略、钩子、沙箱、子 agent、反馈循环，以及围绕模型包裹的恢复路径，这样它才能真正完成任务。

一个还不错的模型加上出色的脚手架，会持续胜过一个出色的模型加上糟糕的脚手架。越来越多的时候，最有趣的工程工作不在于选择模型，而在于设计围绕它的脚手架。

这个学科现在有了一个名字。@Vtrivedy10 创造了"脚手架工程"这个术语，提供了一个清晰的分解，说明脚手架究竟是什么以及每个部分存在的原因。其他行业声音——追踪新兴模式的 @dexhorthy、将 agent 失败定义为配置"技能问题"的 HumanLayer、发布长期运行应用设计指南的 Anthropic 工程团队、以及探索用户侧体验的 Birgitta Böckeler——都在汇聚到大致相同的想法上。

## 脚手架究竟是什么？

Trivedy 的核心定义承担了大部分重量：

> **Agent = 模型 + 脚手架。如果你不是模型，你就是脚手架。**

脚手架涵盖了不是模型本身的每一段代码、配置和执行逻辑。一个裸模型不是一个 agent。只有当脚手架为其提供状态、工具执行、反馈循环和可强制执行的约束时，它才会成为一个 agent。

![](images/img-01.jpg)

具体来说，脚手架包括：

- 系统提示词、CLAUDE.md、AGENTS.md、技能文件和子 agent 指令。
- 工具、技能、MCP 服务器及其技术描述。
- 捆绑的基础设施，例如文件系统、沙箱和无头浏览器。
- 用于生成子 agent、处理交接和路由模型的编排逻辑。
- 用于确定性执行的钩子和中间件，例如 lint 检查或上下文压缩。
- 用于日志、跟踪、成本和延迟计量的可观测性工具。

其核心是，一个 agent 是一个在循环中运行工具以实现目标的系统。真正的技能在于设计工具和那个循环。

虽然这代表了一个巨大的表面积，但这是你的表面积，不是模型提供商的。Claude Code、Cursor、Codex、Aider 和 Cline 都是脚手架。底层模型可能在各个平台上是相同的，但你所体验到的行为主要由脚手架决定。

## 让我们重新定义"技能问题"

当 agent 做了一些无意义的事情时，工程师通常会责怪模型，经常把这个问题归档为"等待下一个版本"来修复的事情。

脚手架工程的思维模式拒绝这种默认做法。失败通常是相当可读的。如果 agent 忽略了一个约定，就把它加到 AGENTS.md 中。如果它运行了一个破坏性命令，就写一个钩子来阻止它。如果它在一个 40 步的任务中迷失了，就把架构分成一个规划器和一个执行器。如果它持续以破损的代码结束，就把类型检查的反压信号接入循环中。

正如 HumanLayer 所说："这不是模型问题。这是配置问题。"

考虑性能基准测试：一个在现成框架中运行的领先模型，其评分通常比在自定义、高度调整的脚手架中运行的完全相同的模型低得多。将一个模型移入一个拥有更好代码库工具、更紧凑提示词和更清晰反压的环境中，可以解锁原始设置所遗留的能力。

今天的模型在理论上能做什么与你实际看到它们在做什么之间的差距，在很大程度上是一个脚手架差距。

## 棘轮：每次错误都变成一条规则

脚手架工程中最重要的习惯是将 agent 错误视为永久信号——而不是一次性的侥幸，重试后就忘记。

如果一个 agent 提交了一个带有注释掉的测试的 PR，并且意外地被合并了，那就是一个输入。AGENTS.md 的下一次迭代必须说明："永远不要注释掉测试；删除或修复它们。"下一个预提交钩子应该自动标记 diff 中的 `.skip(`。审查者子 agent 必须更新以阻止注释掉的测试。

只有当你观察到真实的失败时才应该添加约束，只有当一个有能力的模型使它们变得多余时才应该删除。一个好的系统提示词中的每一行都应该追溯到一个具体的、历史性的失败。

正因为如此，脚手架工程是一种学科，而不是一个放之四海而皆准的框架。一个特定代码库的正确脚手架完全由其独特的失败历史塑造。

## 从行为反向工作

设计脚手架最有效的方式是从期望的行为开始，并构建实现它的组件：

> 我们想要的行为 → 实现它的脚手架设计。

脚手架的每个部分都必须有一个独特的工作。如果你无法命名一个组件存在要实现的具体行为，它就应该被删除。

![](images/img-02.jpg)

**文件系统和 Git — 持久状态**

文件系统是基础。模型只能对适合其上下文窗口的内容进行操作。文件系统提供了一个读取数据的工作空间，一个卸载中间工作的地方，以及一个多个 agent 可以协调的界面。添加 Git 提供了免费的版本控制，允许 agent 跟踪进度、分支实验并回滚错误。

**Bash 和代码执行：通用工具**

大多数 agent 在 ReAct 循环上运行：推理、通过工具调用行动、观察、重复。不是为每一个可以想象的动作预先构建一个工具，而是给 agent bash 访问权限，让它即时构建所需的东西。

**沙箱和默认工具**

Bash 只有在安全运行时才有用。沙箱为 agent 提供了一个隔离的环境来运行代码、检查文件和验证工作，而不会危及主机。一个好的沙箱带有强大的默认值：预安装的语言运行时、测试 CLI 和无头浏览器，允许 agent 观察自己的工作并关闭自我验证循环。

**内存和搜索：持续学习**

模型除了训练权重和当前上下文之外没有任何知识。脚手架使用内存文件（如 AGENTS.md）来弥补这一差距，这些文件在每次会话中注入知识。对于实时信息，如新的库版本或实时数据，网络搜索和 MCP 工具直接内嵌到脚手架中。

**与上下文腐化作斗争**

随着上下文窗口填满，模型的推理能力会下降。脚手架使用三种主要技术来管理这种稀缺性：

- **压缩：** 智能地总结和卸载较旧的上下文，以防止 API 错误。
- **工具调用卸载：** 将大型工具输出（如 2,000 行日志）存储在文件系统中，同时只在上下文中保留基本的头部和尾部。
- **渐进式披露：** 只有当任务明确需要时才显示指令和工具，而不是在启动时加载所有内容。

**长时程执行**

自主、长时间运行的工作受到早期停止和差劣问题分解的困扰。脚手架通过结构化设计来对抗这种情况：

- **循环：** 拦截模型尝试退出的行为，并强制它在一个新鲜的上下文窗口中继续对抗完成目标。
- **规划：** 强制模型将目标分解为逐步的计划文件，在每一步之后通过自我验证钩子检查其工作。
- **分离：** 将生成和评估分成不同的 agent，防止模型在评估自己的工作时固有的正偏差。

**钩子是你的执行层**

钩子弥合了请求一个动作和强制执行它之间的差距。它们在特定的生命周期中运行：在工具调用之前、文件编辑之后或提交之前。钩子阻止破坏性命令，强制自动格式化以节省 token，并运行测试套件。

理想情况下，成功是无声的，失败是详细的。如果类型检查通过，agent 什么也听不到；如果失败，错误会直接注入回循环中进行自我纠正。

**规则手册和工具选择**

存储库根目录中的一个平面 markdown 文件仍然是影响力最高的配置点。然而，它必须像飞行员的检查清单一样对待，而不是风格指南。保持简短，并确保每条规则都是通过过去的失败赚来的。

同样的学科适用于工具。十个高度聚焦的工具总是胜过五十个重叠的工具。

此外，因为工具描述会填充到提示词中，恶意或草率的外部集成（如未经验证的 MCP 服务器）可以在 agent 甚至开始工作之前就向其注入坏提示词。

## 脚手架不会缩小，它会移动

随着模型改进，对脚手架的需求不会消失——它会转移。

很容易假设更好的模型会使脚手架过时。例如，最近的模型升级大幅减少了对"上下文焦虑"缓解措施的需求。但随着底线的提高，上限也在提高。以前遥不可及的任务现在在范围之内，带来了全新的失败模式。

脚手架中的每个组件都编码了关于模型无法独立完成的事情的假设。当模型改进时，过时的脚手架应该被删除，必须构建新的脚手架以到达下一个地平线。

## 训练循环呢？

在脚手架设计和模型训练之间存在一个积极的反馈循环。

今天的模型通常是在循环中带有特定脚手架的情况下进行后训练的，创造了一定程度的过拟合。模型变得极其擅长脚手架设计者优先考虑的特定动作（例如，文件系统操作、bash、子 agent 调度）。

这使脚手架成为一个活的系统，而不是一个静态的配置文件，并证明了"最好的"脚手架是专门针对你的独特任务和工作流程进行优化的。

## 脚手架即服务（HaaS）

业界正在从构建在 LLM API 上（提供补全）转向构建在脚手架 API 上（提供运行时）。SDK 现在开箱即用地提供循环、工具、上下文管理、钩子和沙箱。

现代默认做法不是从头开始构建编排，而是选择一个脚手架框架，配置其核心支柱，并将精力纯粹集中在特定领域的提示词和工具设计上。

这就是使故障排除可扩展的原因：你正在调整一个良好分解的配置界面，而不是重新发明整个 agent 架构。

## 未来走向何方

如果你看看今天顶尖的编码 agent，它们看起来比它们的底层模型更像彼此。模型不同，但脚手架模式正在趋同。业界正在迅速识别将生成性文本转变为可交付软件所需的承重脚手架。

最令人兴奋的开放问题正在超越单一 agent：并行编排多个 agent、使 agent 能够分析自己的追踪以修复脚手架级别的失败，以及构建能够即时动态组装工具的环境。

最终，这是脚手架停止作为静态配置文件并开始更像编译器行动的阶段。
