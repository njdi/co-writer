# 第二步：找问题

对照直译稿，逐处找出不符合风格定义、丢失原文语气的地方。

## 1. 翻译腔残留

- "为一个错误的'你'做优化" —— 英文 optimizing for the wrong "you" 直搬，中文别扭。可改为"它现在照着一个错位的'你'在调整"。
- "上下文预算不再流血" —— "流血"（bleeding）是英文比喻，中文读者会觉得突兀。改为"上下文预算不再白白漏掉"。
- "继续走"（Move on）多处出现，直译生硬。可按语境改为"别再纠结，往下走""到此为止"。
- "下面是真正驱动你的 Claude 的 18 个设置" —— 定语太长。拆开说。

## 2. 语气与感染力

- 原文开头几句很短促有力："Their bill creeps. Their output drifts. They blame the model." 直译"账单慢慢涨，输出慢慢漂，他们却怪模型"基本到位，但"漂"略生。可保留这种短句节奏。
- "凌晨一点去 grep Claude Code 二进制文件" —— 保留这个画面感，很好。
- 标题里的"改变一切"略浮夸，但忠于原文 change everything，保留。

## 3. 术语处理

- settings.json、Claude Code、MCP、TTL、token、hook、Project、Style 等保留英文/原词是对的，目标读者懂。
- "数据驻留税"（data residency tax）—— 这是作者的俏皮说法，保留，但首次出现可稍作交代。
- "面"（surface）—— 直译"面"中文不清楚，改为"每类用途/每个用途场景"。

## 4. 风格定义符合度

- 风格要求：像懂行朋友解释，表达清楚，避免难句。直译整体偏书面，部分句子可口语化。
- "本文里第四个未记录的设置" —— 可更自然："对了，本文承诺的第四个文档里查不到的设置，就藏在这。"
- 数字区间（4-6 周、18-25%）保留半角连字符是数据写法，没问题。

## 结论

直译信息完整，主要问题是个别英文比喻直搬、长定语、以及 "Move on" 一类口头语生硬。意译时：拆长句、换掉生硬比喻、让短句节奏更贴中文，但保留作者干脆利落、略带江湖气的口吻。代码块、设置路径、清单一律不动。
