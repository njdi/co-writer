# 直译稿问题清单

对照风格定义（懂行的朋友在解释、表达清楚、不卖弄、保留作者声音），直译稿存在以下问题：

## 1. 英文句式残留，读着像翻译腔
- “它一直能用，直到某个查询需要跨块连接事实。”——典型的 until 直译，中文应拆成两句：先说能用，再说什么时候不行。
- “这在某个查询需要连接不出现在同一个块里的事实之前都能用。”——长定语套从句，绕口，必须拆。
- “那个组织蓝图叫做本体。”——“组织蓝图”生硬，可改为“这套事先定好的结构”。
- “是一切被决定的地方”“是你把那个好处拿回来的方式”——被字句和“……的方式”是英文 where/how 的硬译。

## 2. 术语处理不一致
- chunk 一会儿“块”一会儿“chunk”，应统一为“块（chunk）”首次出现，后文用“块”。
- “sev-1 工单”“RELATES_TO”等保留原文没问题，但要确保上下文能读懂。
- agent、schema、LLM、ontology 等核心术语：agent 保留英文可接受（目标读者懂互联网），schema 可保留，ontology 首次给中文“本体”+英文。

## 3. 标点
- 直译稿里中文引号用了“”，正确。需检查代码外的中文标点全部为全角，半角只能出现在代码、英文术语、URL 中。目前基本符合。

## 4. 语气偏硬、偏书面
- “逼一个开发者去想清楚”——口语没问题，但“一个开发者”是英文 a dev 的硬译，中文直接说“逼开发者想清楚”。
- “产出废话”——可接受，保留这种口语劲儿，符合作者轻快语气。
- “没有什么新东西要学”——可以更自然：“也就没什么新东西要学”。

## 5. 需保留的作者声音
- 开头“你的 agent 记住了一切，却什么都不理解。”——这句对仗是全文钩子，必须保留力度。
- “agent 什么都没忘。是没人告诉它该关注什么。”——短促有力，保留。
- 结尾 star 的玩笑、“别忘了点 star”——保留亲切感。

## 修订方向
拆长句、去被字句、统一术语、保持开头与关键短句的节奏感，但不过度发挥、不加原文没有的内容。
