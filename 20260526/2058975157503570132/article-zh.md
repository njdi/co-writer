# 我是如何使用 Cursor 的

**作者：** lauren ([@poteto](https://x.com/poteto))  
**日期：** 2026年5月26日  
**来源：** [How I Use Cursor](https://x.com/poteto/status/2058975157503570132)

有件事我得说一下。在我去 @cursor_ai 面试之前，我其实压根没用过 Cursor。

在 Meta 的时候，Claude Code 正在爆发式地火起来。我甚至自掏腰包，给自己的业余项目买了 200 美元一个月的个人套餐。我喜欢它的简单，也喜欢那种很快就能感到自己很高产的感觉。对我来说唯一的卡点，是去打磨我自己的一套 skills，把 cc 调教成几乎任何我想要的样子。我甚至开始在它之上做自己的 agent 编排工具。

现场面试那两天，我一直用 Cursor 来做面试项目。那会儿 Cursor 3 还没发布，所以我用的是编辑器窗口（Editor Window）。我用 vscode 已经用了很多年，大部分快捷键还留在脑子里，所以重新回到 IDE 并不算难。不过说实话，最初一两个小时里，我确实挺想念 cli 的。用鼠标点来点去，感觉有点原始。但有几件事真的让我印象深刻。

第一，我当时常用的那几个模型——Opus 和 Codex——莫名地就是感觉更聪明了。而且能随时切换模型，在项目的不同部分同时用上它俩（Opus 写前端，Codex 写系统），这体验太棒了。面试之前我就已经在到处安利多模型对抗式评审，所以能在界面里原生地这么干，感觉特别自然。更妙的是，还能派生不同模型的 subagent，这样我在一次对话里就能两头的好处都占上。

第二，压缩（compaction）快得离谱。作为 cc 用户，我早就习惯了压缩要等好几分钟，所以我总是提心吊胆地盯着自己的上下文和套餐用量。结果它在 Cursor 里有多快，让我彻底惊到了。惊到什么程度呢？我基本上再也不用去看自己用了多少上下文。它就是好使。而在 cc 里，模型一压缩完，我常常觉得它整个人就变蠢了。

我注意到的第三件事，是 GUI 比 TUI 能多给你多少东西。能直接在 Cursor 自带的浏览器里打开你的 app，再用设计模式（Design Mode）改设计，这个过程很顺手，也让我开始想：专门为这事做的界面，能让 agent 编程效率高多少。

## 用 Cursor 构建 Cursor

三月底加入以来，我主要在做 Cursor 3 的 Agent Window，平时也拿它当主力工具。我仍然觉得 cc 是个很酷的产品，团队也很棒，但我发现，正是它的简单，会让人忍不住想自己再包一层抽象上去。在我上一份工作里，几乎每周都有人宣布一个新的、搭在 cc 上面的内部编排工具。

@bcherny 经常聊到一个叫"潜在需求"（latent demand）的概念：

> "产品里有个特别老的想法，叫潜在需求……你把产品做得可以随便折腾，足够开放，开放到人们能拿它去干别的事。然后你看人们怎么乱用它，再照着那个去做。"

就是这么回事！大家纷纷涌向编排工具，恰恰暴露了一个潜在需求：用 cli，等于是把你这个人，变成了那个编排的人。

但我用过的每一套 agent 工作流，关注点都搞错了。在 GUI 里跑一堆 CLI，完全没抓住重点。我真正感兴趣的，是怎么建立起对 agent 的信任。

我以前当过工程经理，很快就意识到，管 agent 这件事，跟带一支人类工程团队特别像。新人进来要先 onboard，搞清楚代码库长什么样，也得搞清楚活儿是怎么干的。他们入职时已经自带训练，带着过往攒下的 skills：怎么调试、怎么写出高质量的代码和测试、怎么沟通，等等。

agent 就像是一直处在失忆又拎不清状态里的新人。你跟他们说的话他们记不住，他们也从没真正学到什么新东西。但我们可以给他们配上规则、skills、工具和长期记忆，凑出个差不多的效果。他们有本事却很笨，而且特别好教。我把他们出错的那些时刻，都看成机会，把我懂的那些关于深入、严谨地做工程的东西，统统教给他们。

因为一旦没了严谨，agent 就会一味讨好，不择手段地去写出你要的那段代码。而且天哪，它能写、也真会写一大堆。粗糙地搞并行，只会让它们更快地堆出一堆凑数的代码。

> **lauren** [@poteto](https://x.com/poteto) · 4月26日
>
> 我越来越确信，并行编排一大批 agent，价值来自走深，而不是铺广。你应该在一个或少数几个问题上往深里钻，这样才能把拿到好结果的概率拉到最大：
>
> - 用 best of N 式的赛马，挑出最好的那个解

## 想跑得快，先走深

我确实认为，agent 编排是可以做出成果的。但前提是得深度优先。

我打算把 pstack 开源出来——这是我每天用来构建 @cursor_ai 的那套个人 skills 和工程原则。这些 skills 的雏形是我在业余项目里开始做的，之后一直在打磨。

在这里拿它：https://cursor.com/marketplace/cursor/pstack

```markdown
/add-plugin pstack
```

这些 skills 已经成了 Cursor 团队最常用的一批，所以我很高兴能分享给大家。

![](images/img-01.jpg)

Cursor 的公司排行榜。我的 skills 这周被用了 9000 次！

pstack 教 agent 借助多个模型把事情做得更严谨。我把自己观察到的各种出错方式，全都做成了 skills。整个插件的核心是 /poteto-mode，它是个高阶 skill，会根据任务给 agent 一套该照着走的 playbook。目标不是把代码行数堆到最多，恰恰相反：用最少的代码，做出最大的影响。

严谨是怎么来的？靠像有经验的工程师那样去处理问题。比如，调试有个很好的办法，就是对问题空间做二分查找。你先列几个可能出问题的假设，然后系统地一个个排除，慢慢逼近真正的根因。要是不好复现，你可以想办法人为地把 bug 逼出来。或者加点埋点、打点日志，趁程序在跑的时候看看它的状态。

这些步骤合起来就是一套 playbook，agent 可以照着它把问题彻底查清楚，而不是靠猜——你要是放任不管，它们可乐意猜了。pstack 自带了很多 skills 和 playbook，让你能用同样的严谨劲儿去做软件工程。我目前有这些方面的 playbook：

- Skill 编写和评测（evals）
- 自主工作
- Bug 修复和运行时取证
- 功能开发
- 视觉一致性和原型制作
- 还有更多

任何时候你需要严谨，就在 prompt 前面加个 /poteto-mode。比如：

```markdown
/poteto-mode 这个 pr 有个微妙的 bug，就算空闲时滚动条也会每 750ms 漂移一次。先复现，再修复并验证。
/poteto-mode 一个大列表加载要花一两秒，明明我们做了虚拟化。跑个 cpu trace，告诉我为什么。
/poteto-mode 在一个 feature flag 后面做个小功能。验证它真的能用。
/poteto-mode 做两个 markdown 渲染器原型，方便我们对比。给每一个都派生一个 agent。
/poteto-mode 把这些 skills 做成插件开源出去。别泄露任何内部东西，在临时目录里干活，先把依赖图给我看。
/poteto-mode 我要去睡了。就算 ci 不稳定，也把这个 stack 落地。我想早上起来时所有东西都已经合并了。
/poteto-mode 这个 flag 开着的时候行间距太高。第二张图才是对的。复现并修复，直到两边对上为止。
```

你也可以按需点用其他 skills：

- **/how：** 想让人给你讲讲某个子系统到底是怎么运转的。
- **/why：** 想知道某个东西当初为什么这么做。它会用你手上的 MCP，并行去查每一类证据（版本控制、问题跟踪、长文档、实时聊天、基础设施可观测性、错误跟踪、分析数据仓库）。
- **/architect：** 你马上要写跨越函数边界的代码，想先把类型和数据结构定下来。
- **/arena：** 想对同一件事并行尝试 N 次，再从每一次里挑出最好的部分。
- **/interrogate：** 想让不同的模型对抗式地评审某个东西。
- **/tdd：** 你在修 bug。先写一个会失败的测试，再写修复。
- **/unslop：** 你在清理各种 AI 写出来的东西。让它们说人话。
- **/reflect：** 长对话之后，想持续改进你的 skills。
- **/figure-it-out：** 在做点不寻常的事？给这个任务设计一套严谨、可审计的 playbook。
- **/show-me-your-work：** 想要一条能复查的决策记录。它会把每个决策记到一个你可以提交的 tsv 里。

最后，你还能用 /automate-me 做一个属于你自己的 mode skill。它会翻你最近的对话记录，照着你平时的干活方式，起草一个 your-mode skill，底层再接到 pstack 上。

pstack 配任何 agent 编程工具都能用，但在 Cursor 这类多模型工具里尤其好使。很多 skills 都用到了多模型工作流，去借每个模型各自的长处、避开它们的短处。这也是 agent 编排，只不过走的是深度优先，而不是广度优先。

agent 真正的瓶颈在于验证。它能很快写出一大堆代码，但要确认这些代码全都对，难得要命。等哪天你真能做到这一步，真正的 agent 并行——就像一座专门造软件的"暗工厂"（dark factory，无人值守、自动运转的工厂）——也许就成了。

但首先，我们得走深，得严谨。我觉得，靠的是一点点把信任调上去。

去试试 pstack，告诉我你的感受。

## 禅与软件维护的艺术

这些 skills 让我写代码时更有底气。可现在所有代码都是 agent 在写，维护反倒成了噩梦。Bug、性能问题、功能需求，照样要花时间一个个啃。而且现在这些事情多了太多！

在 Cursor，我大量用到了 Cursor 的自动化（automations）。它们是一批云端 agent，可以定时调度，也可以响应事件来跑，比如 Slack 频道里来了条新消息。我的机器人 Benny 就是个例子。我把 pstack 里那套 skills，原样也给了他。

![](images/img-02.jpg)

Benny 的多个化身

Benny 还在打磨中，但我的设想是：尽可能把软件维护的整个流程都自动化掉。思路是这样的：如果我们现在已经有底气，靠 pstack 大体上"一把搞定"问题，而且对 PR 的质量相当有把握，那反馈这一环，当然也能自动化。

这座工厂的第一道工序是分流（triage）：从同事那里收集 bug 报告的信息。我们自己天天用 Cursor，所以同事们会对发布候选版本给很多反馈。Benny 看得懂图片和视频附件，会用 pstack 的 skills 去摸代码库，要是复现步骤说不清楚，他还会找报告的人聊，把信息问明白。

![](images/img-03.jpg)

Benny 在追问更多信息

这是 bug 报告流程里很关键的一环。没有清晰的复现步骤，没搞清楚到底哪里坏了，agent 就只能瞎猜怎么修。我们得让他们清清楚楚地知道，问题到底出在哪、是怎么坏的。

分流一完成，Benny 就会建一个工单，把他查到的东西都写进去：他翻了代码、翻了最近 bug 回归的 git 历史、翻了 Slack 上关于同一个 bug 的其他消息，甚至翻了 Notion 上关于某个功能该怎么运作的设计和产品决策——这到底是个 bug，还是本来就被设计成这样的？

工单一提交，另一个 Benny 机器人就会用我做的另一个 skill 接手，那个 skill 叫 /orchestrate。

> **Cursor** [@cursor_ai](https://x.com/cursor_ai) · 5月8日
>
> 介绍一下 /orchestrate，一个会递归派生 agent、用 Cursor SDK 去啃你最有野心的任务的 skill。
>
> 我们已经用它来：
> - 自动研究我们的内部 skills，token 用量砍掉 20%，评测还更好了
> - 把内部后端的冷启动时间砍掉了 80%

![](images/img-04.jpg)

首先，他会通过"计算机使用"（computer use）来试着复现问题。Cursor Cloud Agents 可以在云端跑 Cursor 本身，在那里他们操作桌面、点击东西、敲键盘输入。这背后又用到了我做的更多 skills，靠 CDP 这类协议（或等效手段），用程序去控制我们自己的产品。

这样我们就能验证，这个 bug 报告到底能不能复现出来。如果它确实能稳定复现，他就接着试着去修。要是性能问题，Benny 还能拍下修复前后的 CPU trace 和堆快照。子规划器（subplanner）会再派生一批 worker，用 pstack 的 skills 去验证修复，对照工单看看到底修好没有。

这一轮里还会派生额外的 worker，去录一段修复前后的对比视频，最后由一个 worker 开 PR 提交评审，描述里就附上那段视频。

![](images/img-05.jpg)

一次成功复现并修复的运行

这一切都还在打磨中，要做的事还多着呢，但一想到有一支 agent 团队，能在我睡觉或者忙别的时候，有底气地帮我修 bug，我就觉得挺兴奋。让代码评审也能规模化，是另一个大方向，我觉得 Cursor 接下来会有一些很酷的功能来帮上忙。

但要建起你自己的软件工厂，关键还是信任。除非你能信任一个 agent 把一个问题从头管到尾——验证也包括在内——否则你没法把流程自动化。当你用 pstack 这类插件，给 agent 添上更多工程深度，把信任一点点调高，你就能开始啃更有野心的问题了。对那些你还信不过的 agent 硬上并行，是在白白烧 token，还会往你的代码库里塞进更多凑数的东西。

谢谢阅读！
