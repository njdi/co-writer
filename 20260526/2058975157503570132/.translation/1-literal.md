# 我是如何使用 Cursor 的

**作者：** lauren ([@poteto](https://x.com/poteto))  
**日期：** 2026年5月26日  
**来源：** [How I Use Cursor](https://x.com/poteto/status/2058975157503570132)

我需要把一件事说出来。在我面试 @cursor_ai 之前，我其实从来没有用过 Cursor。

在 Meta，Claude Code 正在爆炸式地流行起来。我甚至自己掏钱买了 200 美元一个月的个人套餐用于我的业余项目。我喜欢它的简单，喜欢我能多快就感觉到自己很高产。对我来说的卡点在于开发我自己的一套 skills，把 cc 变成几乎任何我想要的东西。我甚至开始在它之上开发我自己的 agent 编排工具。

在我的现场面试期间，我用了 Cursor 两天，用它来构建面试项目。这是在 Cursor 3 发布之前，所以我用的是编辑器窗口（Editor Window）。到这个时候我已经用了 vscode 很多年了，所以大部分键盘快捷键还在我的上下文窗口里，所以回到 IDE 上并不太难。不过我不能撒谎。在最初的一两个小时里我确实很想念 cli。点来点去感觉几乎很野蛮。但有几件事真的让我印象深刻。

首先，我当时习惯用的模型——Opus 和 Codex——感觉不知怎么的更聪明了。而且能够即时切换模型，并在我项目的不同部分同时使用它们两个（Opus 用于前端，Codex 用于系统），这太棒了。在面试之前，我就已经在大力吹捧多模型对抗式评审了，所以能在 UI 里原生地做这件事感觉非常自然。更棒的是能够派生不同模型的 subagents，所以我能在一次对话里得到两全其美。

其次，压缩（compaction）快得离谱。作为一个 cc 用户，我习惯了压缩要花好几分钟，所以我一直处于对我的上下文和套餐用量保持持续警惕的状态。所以我对它在 Cursor 里有多快感到极度震惊。震惊到我基本上从来不需要去看我用了多少上下文。它就是好用，而在 cc 里我常常觉得模型在压缩之后就变得超级蠢。

我注意到的第三件事是 GUI 相比 TUI 能提供多少东西。能够直接在 Cursor 的浏览器里打开你的 app，并用设计模式（Design Mode）做设计改动，感觉很直观，让我开始思考专门构建的 UI 能让 agent 编程变得多有效。

## 用 Cursor 构建 Cursor

自从三月底加入以来，我一直主要在做 Cursor 3 的 Agent Window，并把它作为我的日常主力。虽然我仍然认为 cc 是一个很酷的产品，有一个很棒的团队，但我注意到它的简单往往会驱使人们想要构建他们自己的抽象来包装它。在我上一份工作里，感觉每周都有一个新的、构建在 cc 之上的内部编排工具被宣布出来。

@bcherny 经常谈到这个"潜在需求"（latent demand）的想法：

> "在产品里有一个非常古老的想法叫潜在需求……你以一种可被魔改的方式构建一个产品，它足够开放、足够开放式，以至于人们可以为其他用例滥用它。然后你观察人们如何滥用它，然后你为那个去构建。"

就是这样！大家都聚拢到编排工具上，暴露了一个潜在需求：用 cli 只是让你，也就是人，成了那个编排者。

但我用过的每一个 agent 工作流都聚焦在错误的事情上。在 GUI 里运行多个 CLI 完全没抓住要点。我感兴趣的方法是建立对 agent 的信任。

作为一个前工程经理，我很快意识到，管理 agent 感觉和组建一个人类工程团队很像。新员工需要被 onboard，这样他们才理解代码库，也理解工作是怎么完成的。他们加入时已经预先训练过，带着从过往经历中获得的 skills：怎么调试，怎么写高质量的代码和测试，怎么沟通，仅举几例。

agent 就像是处于持续失忆和愚蠢状态的新员工。他们不记得你告诉他们的东西，他们也从来没真正学到任何新东西。但我们可以给他们配备规则、skills、工具和长期记忆，这能近似那种效果。他们有能力但很蠢，而且非常可教。我把他们的失败模式看作机会，去教他们我所知道的关于做深入、严谨的工程的一切。

因为当没有严谨性时，agent 会谄媚地不惜一切去写出你要求的那段代码。而且天哪，它能写、也会写很多。幼稚的并行化只会让他们更快地写出垃圾。

> **lauren** [@poteto](https://x.com/poteto) · 4月26日
>
> 我越来越确信，并行编排许多 agent 的价值来自于走深，而不是走宽。你想在单个或少数几个问题上走得更深，这样你才能最大化你得到很棒结果的机会：
>
> - best of N 风格的竞赛，去找到最佳解

## 如果你想跑得快，先走深

我确实认为 agent 编排可以被高产地完成。但我们需要深度优先。

我正在开源 pstack，这是我每天用来构建 @cursor_ai 的我个人的一套 skills 和工程原则。我在我的业余项目里开始开发这些 skills 的早期迭代版本，从那以后一直在打磨它们。

在这里获取它：https://cursor.com/marketplace/cursor/pstack

```markdown
/add-plugin pstack
```

这些 skills 已经成为 Cursor 团队最常用的一些 skills，所以我很兴奋能把它分享给你们所有人。

![](images/img-01.jpg)

Cursor 的公司排行榜。我的 skills 这周被用了 9000 次！

pstack 教 agent 使用多个模型变得更严谨。我把我观察到的所有失败模式都变成了 skills。这个插件的核心是 /poteto-mode，它是一个高阶 skill，给 agent 一个对给定任务该遵循的正确 playbook。目标不是最大化代码行数（LOC），而是相反：用最少的代码取得最大的影响。

严谨性是通过用经验丰富的工程师同样的方式来处理问题而被应用的。比如，处理调试的一个很好的方法是对问题空间做二分查找。你先有一些关于可能发生了什么的假设，然后试着系统地排除它们，直到你能更接近真正的根本原因。如果很难复现，你可能会试着人为地强制让 bug 发生。或者你可能会试着加入埋点或控制台日志，在程序运行时检查程序状态。

这些步骤构成了一个 playbook，agent 可以用它来彻底地调试问题，而不是靠猜——如果你放任他们，他们很乐意去猜。pstack 自带许多 skills 和 playbook，让你能用同样的严谨程度来处理软件工程。我目前有以下方面的 playbook：

- Skill 编写和评测（evals）
- 自主工作
- Bug 修复和运行时取证
- 功能开发
- 视觉一致性和原型制作
- 还有更多

每当你需要严谨性时，在你的 prompt 前加上 /poteto-mode 前缀。比如：

```markdown
/poteto-mode 这个 pr 有一个微妙的 bug，即使在空闲时滚动也会每 750ms 漂移一次。先复现，然后修复并验证。
/poteto-mode 一个大列表加载要花一两秒，即使我们做了虚拟化。跑一个 cpu trace，告诉我为什么。
/poteto-mode 在一个 feature flag 后面构建一个小功能。验证它真的能用。
/poteto-mode 构建两个 markdown 渲染器的原型，这样我们可以对比。为每一个派生一个 agent。
/poteto-mode 把这些 skills 作为一个插件开源。不要泄露任何内部东西，在临时目录里工作，先给我看依赖图。
/poteto-mode 我要去睡觉了。即使 ci 不稳定也把这个 stack 落地。我想在早上之前把所有东西都合并。
/poteto-mode 当这个 flag 开启时行间距太高了。第二张图是对的。复现并修复，直到它匹配为止。
```

你也可以按需可选地调用其他 skills：

- **/how：** 你想要一个关于某个子系统实际如何工作的讲解。
- **/why：** 你想知道为什么某个东西被这样构建。它使用你可用的 MCP 并行查询每个证据类别（版本控制、问题跟踪器、长文档、实时聊天、基础设施可观测性、错误跟踪、分析数据仓库）。
- **/architect：** 你即将写跨越函数边界的代码，想先把类型和数据结构定下来。
- **/arena：** 你想对同一件事做 N 次并行尝试，然后抓取每一次中最好的部分。
- **/interrogate：** 你想让不同的模型对抗式地评审某个东西。
- **/tdd：** 你在修一个 bug。先写失败的测试，然后再写修复。
- **/unslop：** 你在清理任何一种 AI 写作。让它们说人话。
- **/reflect：** 你想在长对话之后持续改进你的 skills。
- **/figure-it-out：** 在做某件不寻常的事？为这个任务设计一个严谨、可审计的 playbook。
- **/show-me-your-work：** 你想要一个可评审的决策轨迹。把决策记录到一个你可以提交的 tsv 里。

最后，你可以用 /automate-me 制作你自己的 mode skill。它挖掘你最近的记录（transcripts），从你的工作方式里起草一个 your-mode skill，并在底层通过 pstack 路由。

pstack 可以和任何 agent 编程工具一起用，但它在像 Cursor 这样的多模型工具里效果尤其好。许多 skills 使用多模型工作流，来利用每个模型独特的优势和劣势。这是 agent 编排，但应用的是深度优先而不是广度优先。

agent 的瓶颈在于验证。agent 可以快速写出大量代码。确保它全都正确则极其困难。当你能做到那一步时，真正的 agent 并行——就像软件的暗工厂（dark factory）一样——可能就成为可能。

但首先，我们需要走深，需要严谨。我认为我们靠调高信任来到达那里。

试试 pstack，让我知道你怎么想。

## 禅与软件维护的艺术

这些 skills 帮助我在写代码时更有信心地行动。但现在 agent 在写所有代码，维护代码成了一场噩梦。Bug、性能问题和功能请求仍然需要时间去处理。而现在这些东西多得多了！

我在 Cursor 大量使用 Cursor automations。它们是可以被调度的云端 agent，或者响应像 Slack 频道里的新消息这样的事件来运行。一个这样的例子是我的机器人 Benny。我给了他我在 pstack 里拥有的同样的 skills。

![](images/img-02.jpg)

Benny 的多个化身

Benny 仍然是一个进行中的工作，但我的愿景是尽可能多地自动化软件维护流程。想法是这样的：如果我们现在有信心用 pstack 大体上"一次搞定"问题，并且对 PR 质量很高有相当程度的确定性，那么我们当然也可以自动化反馈。

这个工厂的生命始于分流（triage）：从员工那里收集关于 bug 报告的信息。我们大量自用（dogfood）Cursor，所以我们从员工那里得到很多关于发布候选版本的反馈。Benny 理解图片和视频附件，使用 pstack skills 探索代码库，并在复现步骤不清楚时和报告者聊天以获取信息。

![](images/img-03.jpg)

Benny 在请求更多信息

这是 bug 报告流程的一个重要部分。没有清晰的复现步骤和对哪里坏了的理解，agent 只能去猜解决方案。我们需要给他们一个清晰的理解，明确知道它在哪里以及如何坏掉。

一旦分流完成，Benny 会用他从查看代码、查看最近 bug 回归的 git 历史、查看 Slack 里关于同一个 bug 的其他消息、甚至查看 Notion 里关于一个功能该如何工作的设计和产品决策中得到的发现，创建一个工单：这是一个 bug，还是它本来就被设计成这样工作？

工单提交之后，另一个 Benny 机器人用我创建的另一个叫 /orchestrate 的 skill 接手它。

> **Cursor** [@cursor_ai](https://x.com/cursor_ai) · 5月8日
>
> 介绍 /orchestrate，一个递归派生 agent、用 Cursor SDK 来处理你最有野心的任务的 skill。
>
> 我们已经用它来：
> - 自动研究我们的内部 skills，把 token 用量削减了 20% 同时改进了评测
> - 把我们内部后端的冷启动时间削减了 80%

![](images/img-04.jpg)

首先，他试着通过计算机使用（computer use）来复现问题。Cursor Cloud Agents 可以在云端运行 Cursor 本身，在那里他们与桌面交互、点击东西、发送键盘输入。在内部，这使用了更多我做的 skills，用像 CDP 或等效的协议以编程方式控制我们的产品。

这让我们能演示这个 bug 报告是否能被复现。如果它确实能稳定复现这个 bug，他接着就试着修它。如果是一个性能问题，Benny 可以拍摄修复前后的 CPU trace 和堆快照。子规划器（Subplanners）派生更多 worker，用 pstack skills 来验证修复，并对照工单检查它是否被修好了。

在这次运行中会派生额外的 worker 来拍一段修复前后的视频，最后一个 worker 打开 PR 供评审，描述里带着视频。

![](images/img-05.jpg)

一次成功复现并修复的运行

这一切都仍然是进行中的工作，还有大量工作要做，但我对拥有一个 agent 团队、在我睡觉或做其他事情时帮我有信心地修 bug 感到兴奋。让代码评审可扩展是另一个大领域，我认为 Cursor 会有一些很酷的即将到来的功能来帮忙。

但构建你自己的软件工厂的关键是信任。除非你能信任一个 agent 端到端地拥有一个问题，包括验证，否则你无法自动化你的流程。当你用像 pstack 这样给你的 agent 更多工程深度的插件来调高信任时，你就可以开始处理更有野心的问题。试图并行化你还不信任的 agent 是对 token 的巨大浪费，并且会给你的代码库引入更多垃圾。

谢谢阅读！
