# 如何用 Claude Code 搭一个能在你睡觉时也照样交付功能的「软件工厂」

**作者：** Rahul ([@sairahul1](https://x.com/sairahul1))  
**日期：** 2026年5月25日  
**来源：** [How to Build a Software Factory with Claude Code That Ships Features While You Sleep](https://x.com/sairahul1/status/2058832033628241931)

我一直以为自己在用 AI 写代码。

其实我只是打字打得更快而已。

这中间的区别在哪，以及那个改变了一切的「7 智能体系统」，下面就讲给你听。

收藏这条，它能帮你省下好几个月。

## 没人愿意聊的那个问题

有一个循环，看着很高效，其实并不是：

![](images/img-01.jpg)

→ 让 Claude 做个功能 → 它生成代码 → 某处坏了 → 把报错粘回去 → 它打个补丁 → 又有别处坏了 → 再问一遍

第 1 天，这感觉像变魔术。

第 30 天，你发现：盯着 AI 干活花的时间，比从前自己写代码还多。

同一段逻辑，在 3 个地方各写了一遍。

两周前你定下的约定，Claude 早忘了。

新功能一上，旧功能就坏。

测试要么没有，要么很浅。

某天你醒过来，明白过来一件事：不是 AI 不行。

是你的工作流不行。

真正的问题出在结构上。

你在 Claude Code 里敲下「做这个功能」时，其实是在要求同一个 AI 会话一个人当六个人用：

→ 产品分析师 → 架构师 → 后端工程师 → 前端工程师 → 测试工程师 → 代码评审

六个角色，全挤在同一个乱糟糟的对话里。

计划里的错误假设，会变成错误的数据库模型。

错误的数据库模型，会变成错误的 API。

错误的 API，会变成错误的 UI。

等你察觉时，错误早已扩散得到处都是。

这就是所谓的 vibe coding（凭感觉写代码）。

它有一个怎么也突破不了的上限。

## 转变：从 vibe coding 到软件工厂

真正改变一切的，是这个：

![](images/img-02.jpg)

真正的工程团队，从不在一个大对话里干活。

不同的人，各管各的事：

→ 有人把用户问题问清楚 → 有人琢磨架构 → 有人写 API → 有人做 UI → 有人想边界情况 → 有人做评审

一旦你把这些全压进一个 AI 会话，错误就会悄无声息地越积越多。

解法是：把这些活拆给一个个专门的智能体。

每个智能体只拿到：→ 一件聚焦的活 → 一块干净、属于自己的上下文 → 它真正用得上的那几样工具 → 一条条「什么不能碰」的硬规矩

结果就是一座软件工厂。

一个开发者 + 七个各司其职的智能体 = 一支配合默契的团队。

下面就是让这一切转起来的七个智能体。

## 七个智能体

### 智能体 1：代码库研究员

![](images/img-03.jpg)

开发者用 AI 时最大的错误是什么？

上来第一件事就是要代码。

AI 接过提示词，凭猜测把空缺补上，然后开始生成。

坏设计就是这时候混进来的。

代码库研究员专治这个毛病。

它唯一的活：在写任何代码之前，先把代码库摸清楚，讲明白现在是怎么运作的。

它做的事：→ 理清相关文件各自的职责 → 记录下要遵循的现有模式 → 找出已经做过的类似功能 → 标出风险（时区、多租户、重试逻辑）→ 列出哪些测试需要更新

它不能做的事：→ 改文件（只有只读权限）→ 跑任何会改动状态的命令 → 自己瞎猜——它会主动来问

工具：只有 Read、Grep、Glob。

规矩：动手之前先摸清楚，每一次都这样。

研究员永远第一个上场。

### 智能体 2：故事撰写员

![](images/img-04.jpg)

大多数功能做砸，不是因为代码写错了。

而是因为问题从一开始就没说清楚。

故事撰写员，会在任何技术决策之前，把一个粗糙的功能想法变成一个真正的用户故事。

它接收的输入：→ 你那段粗糙的功能描述 → 研究员的发现

它产出的东西：

一个用户故事：「作为一个【角色】，我想要【某种行为】，好让【某个结果】发生。」

验收标准：测试能直接验证的那种陈述。正常路径、失败路径、业务规则。

边界情况：边界、重试、多租户方面的考量。

不在范围内：明确说清楚，什么是这次不做的。

待解问题：它确实不知道的事——绝不乱猜。

它不能做的事：→ 自己编业务规则 → 写任何代码或技术设计 → 在某处确实没搞清楚时还硬往前走

工具：只有 Read。

规矩：你先读这个故事、点头批准，之后别的事才会开始。

这是个人工把关点，能省掉后面一大堆麻烦。

### 智能体 3：规格撰写员

![](images/img-05.jpg)

故事一旦批准，规格撰写员就把它变成一份技术简报。

这份简报，是后面每个构建智能体都要照着干的蓝图。

它接收的输入：→ 你批准过的用户故事 → 研究员的发现 → 你项目里的 CLAUDE.md 规则

它产出的东西：

→ 数据模型的改动（字段、类型、迁移）→ 后台流程 / 处理流程 → API 的改动（端点、请求与响应的结构）→ 前端的改动（组件、页面、hooks）→ 需要写的测试（成功、失败、边界情况）→ 风险与待解问题 → 每一个将要改动的文件

它不能做的事：→ 改任何文件 → 自己造新基础设施——它会明确把这点指出来 → 跳过租户隔离或时区方面的考量 → 留下没回答的问题

工具：只有 Read、Grep、Glob。

规矩：这份简报是第二个人工把关点。

你读过、批准了，才会有第一个文件被碰。

要是你看到「把 ID 存在内存里」这种话——那就是危险信号。

当场就拦下来。别等 10 个文件都改完了才发现。

### 智能体 4：后端构建员

![](images/img-06.jpg)

到这里，才真正开始动手做。

后端构建员负责实现功能的后端那一半——而且只做后端那一半。

它接收的输入：→ 批准过的技术简报 → 研究员的发现 → 你项目里的 CLAUDE.md

它做的东西：→ API 路由 → 服务和业务逻辑 → 数据库访问和迁移 → 后台任务 → 给自己写的每样东西配上单元测试

它不能做的事：→ 碰 React 组件、页面或客户端 hooks（那是智能体 5 的活）→ 没有指示就自己加新依赖 → 改约定范围之外的文件 → 不跑完类型检查、lint 和测试套件就停手

干完之后，它会交一份摘要：→ 加了或改了哪些文件 → 复用了哪些现成的 helper 或模式 → 哪条 CLAUDE.md 规则本来能派上用场

工具：Read、Edit、Write、Bash——但只限后端那几个文件夹。

把它们分开，正是关键所在。

后端构建员绝不可能一不小心把前端弄坏。永远不会。

### 智能体 5：前端构建员

![](images/img-07.jpg)

前端构建员负责实现 UI 那一半——而且只做 UI 那一半。

它会先读后端构建员的摘要。

这一点很重要。

后端的 API 是什么样，它就照着什么样去用。

它不自己造新端点。

要是 API 的结构对 UI 来说不合适，它会把这个对不上的地方反馈出来——而不是自己打个补丁了事。

它接收的输入：→ 批准过的技术简报 → 研究员的发现 → 后端构建员的摘要（也就是 API 契约）

它做的东西：→ React 组件和页面 → 客户端 hooks 和状态 → 加载状态和错误状态 → 给自己写的每样东西配上组件测试和单元测试

它不能做的事：→ 碰服务、API 路由、worker 或迁移（那是智能体 4 的活）→ 自己编端点或响应结构 → 没有指示就加依赖 → 不跑完类型检查、lint 和测试套件就停手

工具：Read、Edit、Write、Bash——但只限前端那几个文件夹。

两个构建员。

两块互不相干的干净上下文。

谁也别想把对方的活弄坏。

### 智能体 6：测试验证员

![](images/img-08.jpg)

两个构建员都给自己的代码写了单元测试。

但这还不够。

测试验证员只干一件事：证明这个功能真的做到了用户故事里说它该做的事。

它写的是验收测试。

不是单元测试。

是验收测试。

这种测试从外部检验功能——就像一个真实用户实际用起来的样子。

它接收的输入：→ 批准过的用户故事（连同全部验收标准）→ 批准过的技术简报 → 两个构建员的摘要

它产出的东西：→ 一个验收测试文件，把每一条验收标准都覆盖到 → 一份报告：哪些标准过了，哪些没过，哪些没法干净地覆盖

它不能做的事：→ 改任何后端或前端代码 → 给没法测试的标准编一套变通办法 → 某条标准其实没覆盖到，却把它标成「已覆盖」

要是有测试没过：说明这个功能不满足故事。

它会准确说出是哪一条标准没过。

它不去改代码。

那条标准会被退回给对应的构建员。

工具：Read、Edit、Write（只限测试文件）、Bash。

规矩：验收测试没全过之前，你这个功能就还不算做好。

### 智能体 7：实现校验员

![](images/img-09.jpg)

这个智能体，专门抓别人全都漏掉的东西。

校验员把当前的实现，跟批准过的故事和简报对照一遍——然后把差距报出来。

它从不修任何东西。

它只负责说真话。

它每次都跑的那一整套检查：

→ 故事里还没实现的验收标准 → 没有测试覆盖的失败路径 → 安全问题：少了鉴权检查、租户隔离有缺口、日志里出现密钥、把原始报错直接暴露给客户端 → 改到了约定范围之外的文件 → 跟 CLAUDE.md 或现有代码对不上的写法 → 本该复用现成 helper 的重复逻辑 → 简报里写过、却被悄悄跳过的时区或多租户考量

它的输出，永远按严重程度分组：

严重——合并前必须修。重要——合并前应该修。次要——见仁见智，由评审者拿主意。

每一条发现，都带上文件路径和行号。

要是什么问题都没有，它就直说没有。

它不会为了显得自己查得仔细，硬编出几个问题来。

工具：只有 Read、Grep、Glob。

正是这个智能体，让整座工厂值得信任。

自己给自己打分的卷子，一文不值。

而一个只看磁盘上写了什么、不管这代码当初是怎么写出来的校验员，才是诚实的。

## 这条链是怎么跑起来的

完整流程——一句提示词，就能把整条链带起来：

![](images/img-10.jpg)

你打开 Claude Code，敲下：

「给逾期未付超过 7 天的发票，做一个发票提醒功能。」

接下来，你不用再多打一个字，下面这些就会自动发生：

第 1 步 → 研究员把你的发票、支付、邮件相关的代码摸清楚，返回相关文件、现有模式和风险。

第 2 步 → 故事撰写员产出用户故事和验收标准。

⏸ 暂停：你读一遍，批准这个故事。

第 3 步 → 规格撰写员把批准过的故事变成技术简报。

⏸ 暂停：你读一遍，批准这份简报。（那个「把 ID 存在内存里」的错误，就在这里抓住。）

第 4 步 → 后端构建员实现服务、API 路由、BullMQ 任务和单元测试。返回：改了哪些文件、复用了哪些模式、所有测试全绿。

第 5 步 → 前端构建员读完后端构建员的 API 摘要，做出管理后台的 UI 卡片和提醒按钮，写好组件测试。所有测试全绿。

第 6 步 → 测试验证员为全部六条验收标准写验收测试。报告：7 条通过，1 条失败——手动触发那条没有检查租户所有权。

第 7 步 → 校验员把它揪了出来，标为「严重」，附上文件路径和行号。

→ 退回给后端构建员。补上修复。8 条验收测试全部通过。校验员再跑一遍。干净了。

⏸ 暂停：你审一遍，开 PR。

三个人工把关点。

其余的一切，全自己跑。

## 地基：在智能体能干活之前，你得先有这个

CLAUDE.md——一份能在每个会话里都活下来的记忆：

![](images/img-11.jpg)

每次你打开 Claude Code，它都是从零记忆开始的。

CLAUDE.md 解决的就是这个。

它是放在你仓库根目录下的一个 Markdown 文件，每个会话都会自动加载。

项目里那些长期不变的事实，就住在这里：

→ 你的技术栈（Next.js App Router、Node.js、Prisma、BullMQ、Resend）→ 你的命令（npm run dev、npm test、npx prisma migrate dev）→ 架构规则（「业务逻辑放在服务里。API 路由保持很薄。」）→ 不要做什么（「不要加 cron——用 BullMQ。不要把原始的支付数据记进日志。」）→ 指向更详细文档的指针（docs/billing.md、docs/architecture.md）

把它控制在 100-300 行。

每次 AI 犯了一个让你意外的错误，就问一句：要是 CLAUDE.md 里有一条规则，是不是就能拦住它？

把那条规则加进去。

几周下来，你的 CLAUDE.md 就成了一份记录——记下 AI 搞错过的每一个假设，而你的会话，会肉眼可见地越来越好。

**上下文漂移——那个沉默的杀手：**

大多数 Claude Code 会话，不是轰然崩掉的。

它们是慢慢漂走的。

一个错误假设钻进了上下文。

模型就一直在它上面接着盖。

你让 Claude 做订阅管理。

它设计成：User → Subscription。

可你记得：订阅是属于公司的，不属于用户。

要是你只说一句「不对，订阅属于公司」——Claude 会打个补丁了事。

结果你手上就同时飘着 user.subscriptionId 和 company.subscriptionId 两个东西。

规矩是：

→ 小拼写错误？就地改掉。→ 架构假设错了？把这段对话整个扔掉，重开一个，一开始就把正确的前提写进第一条提示词里。

一个心智模型从头就对的干净会话，每次都赢过一个东补西补的会话。

## 结果：到底改变了什么

用工厂之前：

→ vibe coding 的循环：提示 → 生成 → 报错 → 打补丁 → 再来一遍 → 会话上下文被噪声塞满 → 错误假设越积越多，最后攒成坏掉的功能 → 一个工程师一次只能干一件事 → 每个功能都得等那个对的人有空

用工厂之后：

→ 一条结构化的链：研究 → 故事 → 简报 → 构建 → 验证 → 校验 → 每个智能体都有一块干净的上下文，里头只放它需要的东西 → 错误假设在简报批准那一步就被抓住——不是等 10 个文件之后 → 一个工程师就能交付一个完整的纵向切片：后端、前端、测试、校验 → 团队最好的知识，沉淀在智能体里——而不是困在某个人身上

真正的转变在于：

支付方面的专家，做了一个支付集成智能体。

从此，团队里每个工程师都能交付一个跟账单沾边的功能。

不用排队等。

不用谁交接给谁。

前端负责人的那套组件模式，沉淀在前端构建员智能体里。

DevOps 工程师的那套 CI 检查，沉淀在一个 hook 里。

QA 负责人想到的那些边界情况，沉淀在测试验证员的规则里。

专家的知识，以智能体的形式分享出来。

而不再取决于某个人有没有空。

## 怎么在这个周末把你自己的工厂搭起来

一份 8 步的搭建清单：

![](images/img-12.jpg)

1. 安装 Claude Code → code.claude.com
2. 建好文件夹结构：→ .claude/agents/ → .claude/skills/feature-factory/ → .claude/skills/build-with-tests/ → .claude/hooks/
3. 写你的 CLAUDE.md（100–300 行：技术栈、命令、架构规则、不要做清单）
4. 用 Claude Code 里的 /agents 命令创建这 7 个智能体。把每个智能体的角色描述清楚，Claude 来写文件，你来审、来提交。
5. 创建 feature-factory 编排 skill。让 Claude 来写——它会读你那 7 个智能体文件，把整条链接起来。
6. 创建 build-with-tests skill，描述清楚你的团队是怎么干活的：照着现有模式来，代码和测试一起写，最后跑一遍类型检查。
7. 加一个 pre-commit hook，挡住任何包含 .env、.key、.pem 或 secrets.json 的提交。花 5 分钟，能防一场灾难。
8. 拿一个真实功能，从头到尾走完整条链。挑个小的，看它在哪儿磕磕绊绊，然后加规则。工厂会自己越调越好。

总共要花的时间：2–3 小时。

之后再跑几个功能。

跑过 3–4 个，工厂就摸清你的代码库了。

你会在盯活上花更少的时间。

把更多时间，花在决定下一个该做什么上。

## 七个智能体——速查表

→ 研究员——在动手做任何东西之前，先把代码摸清楚（只读）→ 故事撰写员——把想法变成带验收标准的用户故事（只读）→ 规格撰写员——把故事变成技术简报（只读）→ 后端构建员——做 API、服务、任务、单元测试（只限后端文件夹）→ 前端构建员——做组件、页面、hooks、UI 测试（只限前端文件夹）→ 测试验证员——对着用户故事写验收测试（只限测试文件）→ 校验员——把实现跟故事和简报对照，报出差距（只读）

3 个人工把关点：→ 批准故事 → 批准简报 → 批准 PR

其余的一切，全自己跑。

大多数用 Claude Code 的开发者，还停在 vibe coding 上。

提示 → 生成 → 打补丁 → 祈祷。

这没什么错。

只是它有个上限。

工厂不是要把你从流程里赶出去。

它只是把你从那些不需要你的环节里解放出来。

你留在那个真正需要你判断力的环节：
