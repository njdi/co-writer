# 直译稿问题清单

## 翻译腔 / 不自然

- 「同时扮演……全部同时」——英文 "All at once" 单独成句直译显得生硬，中文应合并或换说法。
- 「把所有这些塌缩进一个 AI 会话」——「塌缩」太书面，改「全塞进」。
- 「每个智能体得到」——"gets" 直译，中文说「每个智能体只负责……拥有……」更顺。
- 「在写下一行代码之前」——可保留，但「下一行」会被误读成 next line，应是「写任何代码之前」。
- 「那就是糟糕设计偷偷溜进来的时候」——可接受，但「偷偷溜进来」略生硬，改「就是这时候坏设计混了进来」。
- 「它会去问」——"it asks instead" 直译，改「它会主动问，而不是瞎猜」。
- 「拯救下游一切的人类检查点」——"saves everything downstream" 直译，改「能省掉后面一大堆麻烦的人工把关点」。
- 「红旗」——"red flag" 直译，中文用「警示信号 / 危险信号」。
- 「碰」——"Touch" 直译可口语化保留为「碰」，尚可。
- 「把不匹配作为反馈浮现出来」——"surfaces the mismatch as feedback" 太生硬，改「把这个对不上的地方反馈出来」。
- 「API 的形状」「响应形状」——"shape" 直译为「形状」不自然，改「API 的样子 / 结构」「响应结构」。
- 「一份自己打分的试卷一文不值」——可保留，挺好。
- 「把对话扔掉，带着烤进第一个提示词里的正确假设重新开始」——"baked into" 直译为「烤进」太怪，改「把对话扔掉，重开一个，一开始就把正确的前提写进第一条提示词」。
- 「住在……里」——"lives in" 反复直译为「住在」，部分可保留作拟人，但过多，宜变换：「放在 / 沉淀在」。
- 「困在『有没有空』里」——"trapped in availability" 直译生硬，改「不再取决于某个人有没有空」。
- 「它只是一个天花板」「它有一个很硬的天花板」——"ceiling" 概念要让中文读者懂，第一次出现可补「上限 / 到顶」。

## 术语处理

- vibe coding：保留英文，但首次出现可加一句通俗解释（凭感觉写代码）。
- CLAUDE.md、BullMQ、Prisma、hooks、PR、CI、lint、typecheck：技术名词保留英文。
- 「智能体」用于 agent，全文统一。

## 语气

- 整体短句风格符合原文节奏，保留。
- 标题全大写的英文小节标题已转中文，OK。

## 标点

- 注意：面向读者的中文正文一律全角标点。代码/命令（npm run dev、.env 等）内部保持半角。已基本遵守，定稿再核。
