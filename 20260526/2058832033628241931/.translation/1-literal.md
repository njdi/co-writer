# 如何用 Claude Code 搭建一个在你睡觉时也能交付功能的「软件工厂」

**作者：** Rahul ([@sairahul1](https://x.com/sairahul1))  
**日期：** 2026年5月25日  
**来源：** [How to Build a Software Factory with Claude Code That Ships Features While You Sleep](https://x.com/sairahul1/status/2058832033628241931)

我以为我在用 AI 写代码。

其实我只是在打字更快而已。

这里就是区别——以及那个改变了一切的 7 智能体系统。

收藏这条。它会为你省下好几个月。

## 没人谈论的那个问题

那个感觉很高效、其实并不高效的循环：

![](images/img-01.jpg)

→ 让 Claude 构建一个功能 → 它生成代码 → 某处坏了 → 把错误粘回去 → 它打补丁 → 又有别处坏了 → 再问一遍

第 1 天：这感觉像魔法。

第 30 天：你花在监督 AI 上的时间，比你过去自己写代码花的时间还多。

同样的逻辑出现在 3 个不同的地方。

Claude 忘了你两周前定下的约定。

新功能破坏了旧功能。

测试缺失或者很肤浅。

你醒过来意识到：不是 AI 在失败。

是你的工作流在失败。

真正的问题是结构性的。

当你在 Claude Code 里输入「构建这个功能」时，你是在要求一个 AI 会话同时扮演：

→ 产品分析师 → 架构师 → 后端工程师 → 前端工程师 → 测试工程师 → 代码评审者

全部同时。

在同一个混乱的对话里。

计划里的错误假设变成错误的数据库模型。

错误的数据库模型变成错误的 API。

错误的 API 变成错误的 UI。

等你注意到的时候，错误已经扩散到处都是了。

这叫做 vibe coding。

它有一个很硬的天花板。

## 转变：从 vibe coding 到软件工厂

真正改变一切的东西：

![](images/img-02.jpg)

真正的工程团队不会在一个大对话里工作。

不同的人负责不同的工作：

→ 有人澄清用户问题 → 有人思考架构 → 有人构建 API → 有人构建 UI → 有人思考边界情况 → 有人评审

当你把所有这些塌缩进一个 AI 会话时，错误会悄无声息地累积。

修复方法是把工作拆分到专门的智能体上。

每个智能体得到：→ 一个聚焦的工作 → 它自己干净的上下文窗口 → 只有它真正需要的工具 → 关于它不能碰什么的严格规则

结果：一个软件工厂。

一个开发者 + 七个聚焦的智能体 = 一个协调的团队。

这就是让它运转起来的七个智能体。

## 七个智能体

### 智能体 1：代码库研究员

![](images/img-03.jpg)

开发者用 AI 时犯的最大错误是什么？

把要代码当成第一步。

AI 接受提示词，做出猜测来填补空缺，然后开始生成。

那就是糟糕设计偷偷溜进来的时候。

代码库研究员修复这一点。

它唯一的工作：检查代码库并解释事情是怎么运作的——在写下一行代码之前。

它做什么：→ 映射相关文件及其角色 → 记录要遵循的现有模式 → 找到已经构建好的类似功能 → 标记风险（时区、多租户、重试逻辑）→ 列出哪些测试需要更新

它不能做什么：→ 编辑文件（只读访问）→ 运行任何修改状态的命令 → 做假设——它会去问

工具：只有 Read、Grep、Glob。

规则：构建之前先探索，每一次都是。

研究员第一个运行。永远。

### 智能体 2：故事撰写员

![](images/img-04.jpg)

大多数功能失败不是因为代码错了。

而是因为问题从来没有被清晰地定义。

故事撰写员把一个粗略的功能想法，在做任何技术决策之前变成一个真正的用户故事。

它接收的输入：→ 你粗略的功能描述 → 研究员的发现

它产出什么：

一个用户故事：「作为一个 [角色]，我想要 [行为]，以便 [结果]。」

验收标准：测试可以直接验证的陈述。正常路径。失败路径。业务规则。

边界情况：边界、重试、多租户考量。

不在范围内：什么是明确不会被构建的。

未解问题：它真的不知道的东西——绝不猜测。

它不能做什么：→ 发明业务规则 → 写任何代码或技术设计 → 在某些东西真的不清楚时往前推进

工具：只有 Read。

规则：你读这个故事并批准它，然后其他任何事情才发生。

这是拯救下游一切的人类检查点。

### 智能体 3：规格撰写员

![](images/img-05.jpg)

一旦故事被批准，规格撰写员就把它变成一份技术简报。

这是每个构建智能体都遵循的蓝图。

它接收的输入：→ 你批准的用户故事 → 研究员的发现 → 你项目的 CLAUDE.md 规则

它产出什么：

→ 数据模型变更（字段、类型、迁移）→ 后台流程 / 处理流程 → API 变更（端点、请求/响应形状）→ 前端变更（组件、页面、hooks）→ 需要的测试（成功、失败、边界情况）→ 风险与未解问题 → 每一个将会变更的文件

它不能做什么：→ 编辑任何文件 → 发明新基础设施——而是明确指出来 → 跳过租户隔离或时区考量 → 留下没回答的问题

工具：只有 Read、Grep、Glob。

规则：这份简报是第二个人类检查点。

你读它并批准它，然后才有一个文件被碰。

如果你看到「把 ID 存在内存里」——那就是你的红旗。

现在就抓住它。不要等到 10 个文件被改了之后。

### 智能体 4：后端构建员

![](images/img-06.jpg)

现在构建开始了。

后端构建员实现功能的后端那一半——而且只有后端那一半。

它接收的输入：→ 批准的技术简报 → 研究员的发现 → 你项目的 CLAUDE.md

它构建什么：→ API 路由 → 服务和业务逻辑 → 数据库访问和迁移 → 后台任务 → 为它写的所有东西写单元测试

它不能做什么：→ 碰 React 组件、页面或客户端 hooks（那是智能体 5）→ 在没有指示的情况下发明新依赖 → 修改约定范围之外的文件 → 在没有运行类型检查、lint 和测试套件之前停下

完成之后，它返回一份摘要：→ 每一个增加或编辑的文件 → 每一个复用的现有 helper 或模式 → 任何本可以有帮助的 CLAUDE.md 规则

工具：Read、Edit、Write、Bash——范围限定在后端文件夹。

分离就是重点。

后端构建员永远不可能意外破坏前端。永远。

### 智能体 5：前端构建员

![](images/img-07.jpg)

前端构建员实现 UI 那一半——而且只有 UI 那一半。

它先读后端构建员的摘要。

这很重要。

它正好按后端产出的样子消费 API。

它不发明新端点。

如果 API 的形状对 UI 来说是错的，它会把不匹配作为反馈浮现出来——而不是打补丁。

它接收的输入：→ 批准的技术简报 → 研究员的发现 → 后端构建员的摘要（API 契约）

它构建什么：→ React 组件和页面 → 客户端 hooks 和状态 → 加载和错误状态 → 为它写的所有东西写组件和单元测试

它不能做什么：→ 碰服务、API 路由、worker 或迁移（那是智能体 4）→ 发明端点或响应形状 → 在没有指示的情况下加依赖 → 在没有运行类型检查、lint 和测试套件之前停下

工具：Read、Edit、Write、Bash——范围限定在前端文件夹。

两个构建员。

两个干净的上下文窗口。

零概率一个破坏另一个的工作。

### 智能体 6：测试验证员

![](images/img-08.jpg)

两个构建员都为自己的代码写了单元测试。

那还不够。

测试验证员只做一件事：证明这个功能真的做到了用户故事说它应该做的事。

它写验收测试。

不是单元测试。

是验收测试。

这些从外部测试功能——就像真实用户体验它的方式。

它接收的输入：→ 批准的用户故事（带所有验收标准）→ 批准的技术简报 → 两个构建员的摘要

它产出什么：→ 一个验收测试文件，覆盖每一条验收标准 → 一份报告：哪些标准通过了，哪些失败了，哪些没法干净地覆盖

它不能做什么：→ 修改任何后端或前端代码 → 为不可测试的标准发明变通办法 → 在某条标准真的没被覆盖时把它标记为已覆盖

如果一个测试失败了：这个功能不满足故事。

它准确报告哪条标准失败了。

它不给代码打补丁。

那会回到对应的构建员那里。

工具：Read、Edit、Write（只有测试文件）、Bash。

规则：验收测试通过之前，你没有一个功能。

### 智能体 7：实现验证员

![](images/img-09.jpg)

这是那个抓住其他所有人漏掉的一切的智能体。

验证员把当前实现和批准的故事与简报比较——并报告差距。

它从不修复任何东西。

它只是说出真相。

它每次运行的每一项检查：

→ 故事里还没实现的验收标准 → 没有测试覆盖的失败路径 → 安全问题：缺失的鉴权检查、租户隔离缺口、日志里的密钥、暴露给客户端的原始错误 → 在约定范围之外被改的文件 → 与 CLAUDE.md 或现有代码不一致的模式 → 本应复用现有 helper 的重复逻辑 → 简报里被悄悄跳过的时区或多租户考量

输出总是按严重程度分组：

严重——合并前必须修。重要——合并前应该修。次要——基于观点，由评审者决定。

每一条发现都包含文件路径和行号。

如果没有任何问题，它就直白地这么说。

它不会为了显得彻底而发明问题。

工具：只有 Read、Grep、Glob。

这个智能体是工厂值得信任的原因。

一份自己打分的试卷一文不值。

一个只看磁盘上有什么——而不是它是怎么写的——的验证员，是诚实的。

## 这条链是怎么跑的

完整流程——一个提示词启动这一切：

![](images/img-10.jpg)

你打开 Claude Code 然后输入：

「为逾期未付超过 7 天的发票构建发票提醒。」

下面就是你不再输入任何别的东西就会发生的事：

第 1 步 → 研究员映射你的发票、支付和邮件代码。返回相关文件、现有模式、风险。

第 2 步 → 故事撰写员产出用户故事和验收标准。

⏸ 暂停：你读并批准故事。

第 3 步 → 规格撰写员把批准的故事变成技术简报。

⏸ 暂停：你读并批准简报。（就在这里抓住那个「把 ID 存在内存里」的错误。）

第 4 步 → 后端构建员实现服务、API 路由、BullMQ 任务和单元测试。返回：改了哪些文件、复用了哪些模式、所有测试都绿。

第 5 步 → 前端构建员读后端构建员的 API 摘要，构建管理后台 UI 卡片和提醒按钮，写组件测试。所有测试都绿。

第 6 步 → 测试验证员为全部六条验收标准写验收测试。报告：7 条通过，1 条失败——手动触发没有检查租户所有权。

第 7 步 → 验证员发现了它。作为「严重」报告，带文件路径和行号。

→ 回到后端构建员。应用修复。全部 8 条验收测试通过。验证员再跑一遍。干净。

⏸ 暂停：你评审并开 PR。

三个人类检查点。

其他一切自己运行。

## 基础：在智能体能工作之前，你需要这个

CLAUDE.md——在每个会话中存活下来的记忆：

![](images/img-11.jpg)

每次你打开 Claude Code，它都从零记忆开始。

CLAUDE.md 修复这一点。

它是你仓库根目录下的一个 Markdown 文件，每个会话自动加载。

它是永久项目事实居住的地方：

→ 你的技术栈（Next.js App Router、Node.js、Prisma、BullMQ、Resend）→ 你的命令（npm run dev、npm test、npx prisma migrate dev）→ 架构规则（「业务逻辑住在服务里。API 路由保持很薄。」）→ 不要做什么（「不要加 cron——用 BullMQ。不要记录原始支付载荷。」）→ 指向更深文档的指针（docs/billing.md、docs/architecture.md）

保持在 100-300 行。

每次 AI 犯了一个让你吃惊的错误，问：CLAUDE.md 里的一条规则本可以防止这个吗？

加上那条规则。

几周之后，你的 CLAUDE.md 变成一份 AI 搞错的每个假设的记录——而你的会话明显变得更好。

**上下文漂移——沉默的杀手：**

大多数 Claude Code 会话不会戏剧性地失败。

它们漂移。

一个错误假设进入上下文。

模型不断在它之上构建。

你让 Claude 构建订阅管理。

它设计成：User → Subscription。

你记得：订阅属于公司，不属于用户。

如果你只是说「不，订阅属于公司」——Claude 打补丁。

现在你有 user.subscriptionId 和 company.subscriptionId 两个都飘在那里。

规则：

→ 小拼写错误？内联改正它。→ 错误的架构假设？把对话扔掉，带着烤进第一个提示词里的正确假设重新开始。

一个有正确心智模型的干净会话，每次都胜过一个打了补丁的会话。

## 结果：真正改变了什么

工厂之前：

→ vibe coding 循环：提示 → 生成 → 错误 → 补丁 → 重复 → 会话上下文被噪声填满 → 错误假设累积成坏掉的功能 → 一个工程师一次只能做一件事 → 每个功能都在等对的人有空

工厂之后：

→ 结构化的链：研究 → 故事 → 简报 → 构建 → 验证 → 校验 → 每个智能体得到一个只有它需要的东西的干净上下文窗口 → 错误假设在简报批准时被抓住——不是在 10 个文件之后 → 一个工程师交付一个完整的垂直切片：后端、前端、测试、校验 → 团队最好的知识住在智能体里——而不是困在人里

真正的转变：

支付专家构建一个支付集成智能体。

现在团队里的每个工程师都能交付一个触及账单的功能。

不用等待。

不用交接。

前端负责人的组件模式住在前端构建员智能体里。

DevOps 工程师的 CI 检查住在一个 hook 里。

QA 负责人的边界情况住在测试验证员规则里。

专家知识，作为智能体分享出来。

不困在「有没有空」里。

## 如何在这个周末构建你自己的

8 步设置清单：

![](images/img-12.jpg)

1. 安装 Claude Code → code.claude.com
2. 创建文件夹结构：→ .claude/agents/ → .claude/skills/feature-factory/ → .claude/skills/build-with-tests/ → .claude/hooks/
3. 写你的 CLAUDE.md（100–300 行：技术栈、命令、架构规则、不要做清单）
4. 用 Claude Code 里的 /agents 命令创建 7 个智能体。描述每个智能体的角色。Claude 写文件。你评审并提交。
5. 创建 feature-factory 编排器 skill。让 Claude 写它——它读你的 7 个智能体文件并接上这条链。
6. 创建 build-with-tests skill。描述你的团队怎么构建：匹配现有模式，代码旁边一起写测试，最后跑类型检查。
7. 加一个 pre-commit hook。阻止包含 .env、.key、.pem 或 secrets.json 文件的提交。花 5 分钟。防止灾难。
8. 让一个真实功能走完整条链。挑一个小的。看它在哪里磕绊。加规则。工厂自己调优自己。

总时间：2–3 小时。

然后跑几个功能。

3–4 个之后，工厂就了解你的代码库了。

你会花更少的时间监督。

更多的时间决定接下来构建什么。

## 七个智能体——快速参考

→ 研究员——在任何东西被构建之前映射代码（只读）→ 故事撰写员——把想法变成带验收标准的用户故事（只读）→ 规格撰写员——把故事变成技术简报（只读）→ 后端构建员——构建 API、服务、任务、单元测试（只有后端文件夹）→ 前端构建员——构建组件、页面、hooks、UI 测试（只有前端文件夹）→ 测试验证员——针对用户故事写验收测试（只有测试文件）→ 验证员——把实现和故事与简报比较，报告差距（只读）

3 个人类检查点：→ 批准故事 → 批准简报 → 批准 PR

其他一切自己运行。

大多数用 Claude Code 的开发者还在 vibe coding。

提示 → 生成 → 打补丁 → 祈祷。

那没有错。

它只是一个天花板。

工厂不把你从流程里移除。

它把你从不需要你的部分里移除。

你留在你的判断力重要的那个环节：

这是对的问题吗？这是对的设计吗？这个可以安全发布吗？

智能体处理这之间的一切。

那就是把 AI 当成一个更快的键盘——

和把 AI 当成一个协调的团队——之间的区别。
