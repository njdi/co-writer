# 从零构建你自己的 LLM：五个阶段，还原 GPT 和 Claude 背后的完整流程

**作者：** Codez ([@0xCodez](https://x.com/0xCodez))  
**日期：** 2026年5月25日  
**来源：** [How to build your own LLM from scratch in 5 Stages: exact pipeline behind GPT and Claude](https://x.com/0xCodez/status/2058911661973454915)

我把大语言模型到底是怎么造出来的整个拆开了——ChatGPT、Claude、Gemini 背后那一整条流水线——然后压成了一张图。

收藏一下。读完它，你就能看清从原始的互联网文本，到一个能像助手一样答话的模型，中间究竟要走哪五步。

这不是夸张。大多数人以为造一个 LLM 关键在架构。而这里最核心的一课恰恰是：架构是其中最不重要的一环。

## 关于 LLM，几乎人人都信的一个误解

随便问个人，像 Claude 这样的模型是怎么造出来的，多半会回答"transformer"，好像秘密就藏在神经网络的设计里。

并不是。transformer 架构基本已经标准化，而且是公开发表的。各大实验室用的都是差不多的那几块积木。如果架构真是护城河，那大家手里早就都有 GPT-4 了。

有一句话能把整件事重新摆正：真正决定一个模型成败的，是数据、评估和系统——不是在架构上修修补补。最好的模型不只是"训练"出来的，而是"工程"出来的。

所以这份指南，是围绕真正要紧的东西来写的。一共五个阶段。架构只占第一阶段里的一小段，剩下那四个，才是模型真正拉开差距的地方。

## 01. 预训练——让模型先学会语言本身

一切都从一个看着特别简单的目标开始：预测下一个词。这就是所谓的自回归语言建模。给模型一串词，它去学接下来这个词最可能是什么的概率分布。

在足够多的文本上反复这么干，模型就慢慢吸收了语法、事实，还有推理的套路——没人专门教过它们，只是因为想把"下一个词"猜准，就不得不掌握这些。

![](images/img-02.jpg)

### 第一步是分词

模型还没看到文本之前，文本先被切成一个个 token。标准做法是字节对编码（BPE），它的逻辑会一路影响后面所有环节。

![](images/img-03.jpg)

### 架构——最不重要的那一环

模型就是个 transformer。这一节基本上说到这儿就完了，而这恰恰是重点。你不会靠发明一个更聪明的 transformer 取胜，你赢在另外四个阶段。

讲座用缩放曲线把这件事说死了：transformer 的常数项和斜率就是比 LSTM 好——用现成的标准件就行，别纠结，往下走。

## 02. 数据——模型真正拉开差距的地方

架构最不重要，那数据就最重要。这一步把好模型和平庸模型分了开来，也是大多数人最容易低估的一步。

流水线从 Common Crawl 开始——它是对公共互联网的一次抓取，规模大到要用 PB 来量：2500 亿个网页，一百多万 GB。可问题是，原始的网络数据特别脏。

把它变成能拿来训练的材料，要过一道道严苛的筛选。

![](images/img-04.jpg)

处理流程大致是这样：

- 从 HTML 里把正文抽出来——还得专门处理数学公式、模板套话这些特殊情况。
- 滤掉不想要的内容——色情、有害信息、个人隐私。
- 按 URL、按文档、按行去重——网络上的东西没完没了地重复（页眉、页脚、菜单）。
- 启发式过滤——按词数、异常 token、脏 token 把低质量文档剔掉。
- 用模型来过滤——预测某个页面够不够格被维基百科引用。
- 调配数据比例——把内容分门别类（代码、书籍、娱乐），再用缩放定律重新调整各领域的权重。

有句话值得记牢：数据质量胜过数量。把数据收集好，是做出一个实用 LLM 的关键所在——也是这个行当里捂得最严的秘密。

闭源的数据集，规模远不是开源能比的：LLaMA 3 用了 15 万亿个 token 来训练，GPT-4 据估计用了 13 万亿。

## 03. 缩放定律——把算力花在刀刃上

假设你有 1 万张 GPU，能用一个月。该训练一个什么样的模型？是做大一点，还是喂更多数据？瞎猜的代价是几百万美元打水漂。缩放定律能给你一个可预测的答案。

经验上的结论是：数据越多、模型越大，性能就越好，而且在开训之前，你就能根据模型的大小和数据量，把它最终的表现预测出来。

现在通行的做法，是先在小模型上把超参数调好，再顺着曲线外推，去赌那一次特别烧钱的最终大训练。

![](images/img-05.jpg)

最有名的就是 Chinchilla 给出的那个答案：每个参数配大约 20 个 token 的训练数据，在算力上最划算。不过这只算了训练那一头的账。

一旦把模型跑起来的成本——也就是推理——也算进去，这个比例就会猛涨，超过每个参数 150 个 token。你宁可拿一个更小的模型去喂多得多的数据，因为这模型造好之后，你得花钱跑它几百万次。

还有一条更上层的道理，也就是那条"惨痛的教训"（the bitter lesson）：别把事情搞复杂。把简单的事情做好，然后往大了堆。长远看，唯一真正管用的，就是把算力用起来。

## 04. 后训练——把一个会预测的模型，变成一个会答话的助手

预训练完，你手里这东西很强，可拿来聊天却没法用。它会接着把文本补全，但它压根不知道自己该回答你。

你问它一个问题，它可能反手又抛给你三个问题——单从"下一个词该接什么"来看，这么续下去完全说得通。

![](images/img-06.jpg)

### 监督微调（SFT）

你给模型看成千上万个例子——一句提问，配上一段好回答——它就学着去模仿这个套路。这叫行为克隆，正是从 GPT-3 走到 ChatGPT 的关键一步。

让人意外的是：这一步需要的数据很少。几千个例子就够了，因为 SFT 只是在教模型"一个好答案长什么样"——真正的知识，预训练那会儿早就装进去了。

Alpaca 这个项目甚至直接拿另一个 LLM 来造数据：生成了 5.2 万对指令和回答，就把一个 LLaMA 7B 调教成了一个挺能干的助手。

### RLHF——让模型对齐人的偏好

SFT 有三个毛病：它的天花板被人的能力卡住了；它会教出幻觉（让模型去克隆一个连它自己都不懂的"正确答案"，等于在教它编瞎话）；而且写出理想答案的成本很高。RLHF 换了个思路来解决——它优化的目标是人的偏好，而不是单纯的模仿。

模型一次生成两个答案，由人来挑哪个更好。这些偏好被拿去训练一个奖励模型，然后再去调模型，让它尽量把这个奖励做到最大——经典做法是用 PPO。

还有个更简单的现代替代方案叫 DPO，用普通的监督学习就能做到差不多的效果，如今已经是开源社区里的标配。

![](images/img-07.jpg)

## 05. 评估与系统——证明它真的行，也让它真的造得出来

有两件事，是把整条流水线从头到尾包起来的。少了哪一个，你都没法说自己有一个真正的模型。

### 评估：怎么衡量一个没有标准答案的东西

预训练阶段，衡量的指标是困惑度——也就是模型在多少个 token 之间"犹豫不决"。从 2017 到 2023 年，最好的模型从在大约 70 个 token 之间打转，降到了不足 10 个。可一旦经过对齐，困惑度就失灵了，于是评估转向了各种基准和对比：

- MMLU 和 HELM——覆盖很多领域、带标准答案的任务套件。MMLU 是最受认可的预训练基准。
- Chatbot Arena——让人盲测两个模型再投票，30 多万张票撑起一个 Elo 排行榜。
- AlpacaEval——让一个 LLM 去给别的 LLM 打分。它和 Chatbot Arena 的结果有 98% 的吻合度，3 分钟之内、花不到 10 美元就能跑完——但它有偏见，比如更偏爱长答案。

说句实在话：评估一个对齐过的模型，是真的难，没有哪一个数字能把它说清楚。同一个模型，光是换换提示的格式，在 MMLU 上就能从 0.637 掉到 0.488。

### 系统：让训练在现实里真能跑起来

所有人都被算力卡着脖子——GPU 又贵又稀缺，物理上还被通信速度拖着。一个 7B 的模型，啥优化都不做地硬训，光显存就得吃掉差不多 112GB。所以系统这一层根本不是可有可无，它是让这一整套东西能落地的前提：

- 低精度——用 16 位（bf16）代替 32 位，显存直接砍半，速度还更快。
- 算子融合与分块——尽量少去访问那个慢吞吞的全局内存；光是一个 FlashAttention，端到端就能快上大约 1.7 倍。
- 数据并行——把数据集切开分到多张 GPU 上（再用 ZeRO 把优化器状态分片，省显存）。
- 模型并行——把模型切开分到多张 GPU 上，要么按层切（流水线），要么按矩阵切（张量）。
- 稀疏化——专家混合（MoE）：参数更多，FLOPs 却不变，办法是每个 token 只激活其中一部分。

## 走完这一整条流水线，它到底告诉了我们什么

把这五个阶段从头再捋一遍，那个结论就没法反驳了。架构——大家最痴迷的那一块——反倒分到的笔墨最少。数据、缩放、对齐、评估、系统，才是每一个真正的决定被拍板的地方。

这也正是为什么，两个用着同一套架构的实验室，造出来的模型能天差地别。架构是共享的，而真正要紧的那些东西，全都不是。

## 那些会让 LLM 项目翻车的坑

- 死磕架构。它是整套技术栈里被抄得最多、最没区分度的部分。
- 把数据当成一种随手可得的商品。数据脏，不管你堆多少算力，上限都被它压死。质量比数量重要。
- 跳过 Chinchilla 那笔账。模型相对它的数据量做得太大，就是训练不足，纯属浪费算力。
- 停在 SFT 就不往下走了。微调出来的模型只会模仿；没有 RLHF 或 DPO，它永远学不会人到底偏好什么。
- 对齐之后还信困惑度。后训练已经把分布改掉了，困惑度也就不再说明问题了。

## 结论

一个出色的模型，不是训练出来的，是工程出来的。

大多数人还会继续相信，造 LLM 关键在架构，继续去读那些讲 transformer 的文章，也就继续错过了真正的活儿到底在哪里发生。

而真正搞懂这条流水线的人，会看得很清楚：先是语言建模，然后是干净的数据，然后是把算力缩放到最优，然后是对齐，最后是在高效的系统上做诚实的评估。五个阶段。架构，只是其中某一个阶段里的一段话而已。

挑一个你一直在绕开的阶段——多半是数据，或者评估。往深里钻。差距，就藏在那里。
