# 如何从零开始构建你自己的 LLM，分 5 个阶段：GPT 和 Claude 背后的确切流水线

**作者：** Codez ([@0xCodez](https://x.com/0xCodez))
**日期：** 2026年5月25日
**来源：** [How to build your own LLM from scratch in 5 Stages: exact pipeline behind GPT and Claude](https://x.com/0xCodez/status/2058911661973454915)

我把大语言模型实际上是怎么构建的拆解开来——ChatGPT、Claude 和 Gemini 背后的整个流水线——并压缩成了一张图。

把这个收藏起来。保存下来。读完之后你会理解从原始互联网文本到一个像助手一样回答的模型的确切五阶段路径。

这不是夸张。大多数人认为构建一个 LLM 是关于架构的。这里的核心教训是架构是最不重要的部分。

## 每个人都相信的关于 LLM 的谎言

问大多数人你怎么构建一个像 Claude 这样的模型，他们会说"transformer"。好像秘密在于神经网络设计。

不是的。transformer 架构在很大程度上是标准化的并且自由发布的。每个主要实验室都使用大致相同的构建模块。如果架构是护城河，那么每个人都会有 GPT-4。

这里有一句话重新定义了一切：在实践中是数据、评估和系统决定了一个模型的成败——而不是架构上的微调。最好的模型不只是被训练出来的。它们是被工程化出来的。

所以这份指南是围绕真正重要的东西构建的。五个阶段。架构是阶段 1 里的一个脚注。另外四个才是真实模型胜负所在。

## 01. 预训练——教模型语言本身

一切都始于一个看似简单的目标：预测下一个词。这是自回归语言建模。给定一个词序列，模型学习接下来是什么的概率分布。

在足够多的文本上做这件事，模型就会吸收语法、事实和推理模式——不是因为有人教了它们，而是因为很好地预测下一个词需要它们。

### 分词先行

在模型看到文本之前，文本被切分成 token。标准方法是字节对编码（BPE），它的逻辑塑造了下游的一切。

### 架构——最不重要的部分

模型是一个 transformer。这一节基本上就是这样了，这正是重点。你不是通过发明一个更聪明的 transformer 取胜。你在另外四个阶段取胜。

讲座用缩放曲线证明了这一点：transformer 就是比 LSTM 有更好的常数和斜率——选标准工具然后继续往前走。

## 02. 数据——模型真正被赢得的地方

如果架构最不重要，数据就最重要。这是把好模型和平庸模型区分开来的阶段，也是大多数人低估的那个。

流水线从 Common Crawl 开始——一个对公共网络的抓取，大到以 PB 衡量：2500 亿页，超过一百万 GB。但原始网络数据很脏。

把它变成训练材料是一个残酷的多步过滤。

处理流水线看起来是这样：

- 从 HTML 中提取文本——处理像数学和样板这样的特殊情况。
- 过滤不想要的内容——NSFW、有害内容、个人数据。
- 按 URL、文档和行去重——网络无休止地重复（页眉、页脚、菜单）。
- 启发式过滤——按词数、异常 token、脏 token 移除低质量文档。
- 基于模型的过滤——预测一个页面是否可能被维基百科引用。
- 数据混合——分类成类别（代码、书籍、娱乐）并用缩放定律重新加权领域。

值得刻进脑子的口头禅：数据质量胜过数量。把数据收集好是实用 LLM 的关键——也是这个领域里守护得最严的秘密。

封闭数据集让开放数据集相形见绌：LLaMA 3 在 15 万亿 token 上训练；GPT-4 估计在 13 万亿。

## 03. 缩放定律——最优地花费算力

你有 1 万个 GPU 用一个月。你训练什么模型？更大的，还是在更多数据上训练的？猜测会浪费数百万。缩放定律可预测地回答它。

经验发现：更多数据和更大模型可靠地意味着更好的性能，而且你可以在训练之前从一个模型的大小和数据预测它的性能。

现代流水线在小模型上调超参数，然后沿着曲线外推到那一次巨大的最终运行。

著名的 Chinchilla 答案：每个参数大约 20 个 token 的训练数据是计算最优的。但那只是训练成本。

一旦你算上运行模型的成本——推理——实际比率急剧上升，超过每个参数 150 个 token。你在多得多的数据上训练一个更小的模型，因为你要付钱运行它数百万次。

还有元教训，"惨痛的教训"：不要过度复杂化。做简单的事情并扩展它们。从长远来看，唯一重要的事情是利用计算。

## 04. 后训练——把一个预测器变成一个助手

预训练之后你有了某种强大但对聊天没用的东西。它补全文本，但它不知道它应该回答你。

问它一个问题，它可能用三个问题回复——一个完全合理的下一个词的延续。

### 监督微调（SFT）

你给模型展示成千上万个例子——一个提示后面跟着一个好的回应——它学会模仿那个模式。这是行为克隆，它是从 GPT-3 到 ChatGPT 的关键一步。

令人惊讶的部分：你需要很少的数据。几千个例子就够了，因为 SFT 只教好答案的格式——知识已经在预训练模型里了。

Alpaca 项目甚至用另一个 LLM 生成了它的数据：52000 个指令-回应对，用来把一个 LLaMA 7B 微调成一个有能力的助手。

### RLHF——与人类偏好对齐

SFT 有三个问题：它被人类能力束缚，它教幻觉（克隆一个模型实际上不知道的"正确"答案教它编造东西），而且理想答案很贵。RLHF 通过为偏好而不是模仿优化来修复这个问题。

模型生成两个答案。一个人挑更好的那个。那些偏好训练一个奖励模型，LLM 被优化来最大化那个奖励——经典地用 PPO。

一个更简单的现代替代方案，DPO，用普通的监督学习达到可比的质量，现在是开源社区的标准。

## 05. 评估和系统——证明它有效，让它可行

两件事包裹着整个流水线。跳过任何一个，你就没有一个真正的模型。

### 评估：测量某种开放式的东西

预训练期间指标是困惑度——模型在多少个 token 之间"犹豫"。在 2017 和 2023 年之间，最好的模型从在大约 70 个 token 之间犹豫降到少于 10 个。但困惑度在对齐之后失效，所以评估转向基准和比较：

- MMLU 和 HELM——跨许多领域有标准答案的任务套件。MMLU 是最受信任的预训练基准。
- Chatbot Arena——人类盲比两个模型并投票；30 万+ 投票驱动一个 Elo 排行榜。
- AlpacaEval——一个 LLM 评判其他 LLM。与 Chatbot Arena 98% 相关，不到 3 分钟和不到 10 美元——但它有偏见，比如偏好更长的答案。

诚实的要点：评估一个对齐的模型确实很难，没有一个单一的数字能捕捉它。同一个模型在 MMLU 上可能得 0.637 或 0.488，只取决于提示格式。

### 系统：让训练在物理上成为可能

每个人都被算力卡住——GPU 昂贵、稀缺，并且在物理上被通信速度限制。一个 7B 模型仅仅天真地训练就需要约 112GB。所以系统层不是可选的；它是让整个事情可行的东西：

- 低精度——16 位（bf16）而不是 32 位，把内存减半并提升速度。
- 算子融合和分块——最小化对全局内存的慢速访问；仅 FlashAttention 就给出约 1.7 倍的端到端加速。
- 数据并行——把数据集切分到 GPU 上（用 ZeRO 分片优化器状态以节省内存）。
- 模型并行——按层（流水线）或按矩阵（张量）把模型切分到 GPU 上。
- 稀疏性——专家混合：更多参数，相同 FLOPs，通过每个 token 只激活一个子集。

## 这整个流水线教给我们什么

回头走一遍五个阶段，论点是无可否认的。架构——每个人痴迷的部分——得到了最少的关注。数据、缩放、对齐、评估和系统才是每个真正的决定被做出的地方。

这就是为什么两个有相同架构的实验室产出截然不同的模型。架构是共享的。所有重要的东西都不是。

## 让 LLM 项目沉没的错误

- 痴迷于架构。栈里最被复制、最没有区分度的部分。
- 把数据当成一种商品。脏数据不管算力多少都给你的上限封顶。质量胜过数量。
- 跳过 Chinchilla 的数学。一个对它的数据来说太大的模型训练不足并浪费算力。
- 停在 SFT。一个微调过的模型模仿；没有 RLHF 或 DPO 它永远学不会人们偏好什么。
- 在对齐之后信任困惑度。后训练改变了分布——困惑度不再有意义。

## 结论

一个伟大的模型不是被训练出来的。它是被工程化出来的。

大多数人会继续相信构建一个 LLM 是关于架构的，继续读 transformer 讲解，继续错过真正的工作发生的地方。

理解流水线的人会清楚地看到它：语言建模，然后干净的数据，然后最优的缩放，然后对齐，然后在高效系统上诚实的评估。五个阶段。架构是其中一个阶段的一个段落。

挑一个你一直忽略的阶段——很可能是数据或评估。深入研究它。那就是差异所在。
