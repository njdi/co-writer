# Claude Skills 完整指南

**作者：** Clara Bennett ([@CodeswithClara](https://x.com/CodeswithClara))  
**日期：** 2026年4月29日  
**来源：** [The Complete Guide to Building Skills for Claude](https://x.com/Zephyr_hg/status/2049479378112041101)

过去几周我一直在深入研究 Claude Skills，我觉得大多数人还没意识到它能带来什么

如果你还不知道 Skill 是什么，这是最快的入门方式

Skill 就是一个文件夹。文件夹里有一个叫 SKILL.md 的 Markdown 文件，还可以放一些脚本、模板或参考资料。Claude 看到一个任务，发现它和你 SKILL.md 里描述的对上了，就会加载这个文件夹，按里面的指令来做

就这么简单。不需要微调，不需要向量数据库，不需要搭基础设施。只是一个文件夹，里面是 Claude 知道该什么时候去拿的指令

这件事之所以重要，是因为它解决了一个没人讨论的 LLM 问题。模型很聪明，但它不了解你的工作方式。它不知道你的品牌风格、文件命名规则、报告格式，或者你技术栈里那些奇怪的细节。结果就是你每次都要把同样的背景重新贴一遍

Skills 解决了这个问题。上下文只写一次，之后 Claude 在需要的时候自动带入

## 什么时候该建一个 Skill

不要坐下来从零开始发明一个 skill。那条路的终点是一堆精心设计却从来不用的文件夹

等到你发现自己连续三次用同样的方式提示 Claude。那就是信号。重复出现的，就是你真正需要的

我工作时会一直开着一个笔记文件，每次发现自己在对话里粘贴同样的背景、同样的指令或同样的示例，我就记下来。周末翻一下这个清单，skills 基本上自己就写出来了

## SKILL.md 的结构拆解

每个 SKILL.md 最重要的有两部分：顶部的描述，和正文的指令

描述是整个文件里最重要的一行。Claude 就是靠它来判断你的 skill 是不是跟当前任务有关。写得含糊，skill 就会一直躺在那里，永远不会被触发。写得清晰具体，它就会在该出现的时候出现

糟糕的描述："帮忙写东西"

好的描述："当用户要给创始人、投资人或运营者写开发信时使用。涵盖语气、结构和后续跟进序列"

注意区别。第二个告诉 Claude 什么时候用、什么时候不用。具体，才能让 skills 真正发挥作用

正文是你放实操手册的地方。把它想成你第一天给新员工的操作流程文档。把步骤、格式、示例、注意事项和边界情况都写进去。不要假设 Claude 知道你的偏好，全部说清楚

## 我几乎每个 Skill 都用的结构

第一行写一句目标说明，让 Claude 知道这个 skill 是干什么的

然后列出我通常会提供的输入，让 Claude 知道从我这能拿到什么

然后把实际流程写成编号步骤。这里是价值最密集的地方。要有立场，说"这样做，不要那样做"

然后放两三个输入和理想输出的示例。这是 skill 写作里最被低估的部分。示例比任何规则都能更快地教会 Claude。一组好的输入输出对，胜过十段说明文字

最后加一个叫"常见错误"的章节，列出 Claude 在这个任务上容易犯的错误。就这一个章节，帮我省掉的重复提示比其他所有东西加起来都多

## 保持 Skill 小而专注

这是我刚开始时最频繁打破的规则

一个 skill，只做一件事。如果你想做一个"搞定所有营销"的 skill，描述会变得太宽，Claude 不知道什么时候该用。拆开来做：冷邮件一个，LinkedIn 帖子一个，落地页文案一个

小 skill 可以组合，大 skill 只会垮掉

## 使用参考文件

这是 skills 真正强大起来的地方

你可以把任何文件丢进 skill 文件夹——品牌手册 PDF、样式表、批准用语列表、模板、JSON schema，都行。然后在 SKILL.md 里引用它，Claude 需要的时候会自己调用

这就是获得真正个性化输出的方式。不用在每次对话里粘贴品牌手册，放进 skill 文件夹一次，之后就不用管了。Claude 在相关时自己打开，你的输出会开始听起来像是你写的

## 让 Claude 帮你写 Skill

这是最省时间的玩法

打开一个对话，告诉 Claude 你一直在做什么任务，粘贴两三个你以前是怎么做的示例，让它帮你起草一个 SKILL.md，然后你再改

15 分钟就能有一个可用的 skill，而不是两个小时。而且 Claude 写的描述通常比你自己写的更准，因为它见过太多这样的文件了

## Skills 可以互相调用

等你积累了几个 skill，真正的魔法才开始

可以有一个收集信息的调研 skill，一个把调研变成草稿的写作 skill，还有一个输出最终文件的排版 skill。Claude 自己来编排这条链。你只需给出原始任务，skills 之间自动交接

我亲眼看到之前是不信的。有种突然拥有了一支小团队的感觉

## 关键所在

现在用 skills 玩得好的人，不是文件夹最花的那些。是那些认真留意过自己重复了什么的人

每次把同样的提示词再打一遍，时间就在往外漏。一个 skill 就是把这个漏洞堵上一次，之后再也不用管它

流程很简单：发现重复，写 SKILL.md，在真实任务上测，出错就改，然后继续

做五次，你就有了一个懂你工作方式的 Claude。做二十次，输出质量开始复利增长，那种感觉在你真正体验到之前很难形容

下次发现自己第三次在打同一段提示词时，停下来，打开一个文件夹，写个 SKILL.md。你刚给自己做了一次永久升级
