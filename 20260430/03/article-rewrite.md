# Claude Skills 完整指南

**作者：** Clara Bennett ([@CodeswithClara](https://x.com/CodeswithClara))  
**日期：** 2026年4月29日  
**来源：** [The Complete Guide to Building Skills for Claude](https://x.com/Zephyr_hg/status/2049479378112041101)

这几周我一直在研究 Claude 的 Skills，越研究越觉得多数人还没搞清楚它能做什么。

先说清楚 Skill 是什么东西。最简单的方式就是告诉你：它就是一个文件夹。

文件夹里有一个叫 SKILL.md 的文件，你在里面写好描述和指令。Claude 每次接到任务，会先看看这个描述跟眼前的事情对不对路，对上了，就把这个文件夹加载进来，按你写的指令做事。

就这样。不需要微调模型，不需要向量数据库，不需要搭任何基础设施。一个普通文件夹，一个 Markdown 文件，Claude 自己知道什么时候该去拿它。

听起来简单，但它解决的问题其实一直没人好好说清楚。

模型很聪明，但它不了解你。它不知道你的品牌语气是什么风格，你的文件用什么命名规则，报告格式有哪些要求，或者你的技术栈里有哪些奇怪的细节。每次开始一个新对话，你都要把这些背景重新贴一遍。用久了，这件事会让你很烦。

Skills 把这个问题解决掉了。背景只写一次，之后 Claude 自己在需要的时候带进来。

## 什么时候该建一个 Skill

不要坐下来从零开始想"我要建什么 skill"。这个思路会把你引进死胡同，最后做出一堆精心设计却从来不用的文件夹。

正确的方式是等。等到你连续三次用同样的方式提示 Claude，那就是信号了。重复出现的地方，就是你需要一个 skill 的地方。

我自己的习惯是工作时开着一个笔记文件，每次发现自己在对话里粘贴同样的背景、指令或示例，就记一条。周末看一遍这个清单，skills 基本上自己就浮出来了。

## SKILL.md 写什么

每个 SKILL.md 最重要的有两部分：顶部的描述，和正文的指令。

描述是整个文件里最关键的一行。Claude 就是靠它来判断这个 skill 跟当前任务有没有关系。描述写得含糊，skill 就会一直躺在那里，永远不会被用到。写得清晰，它就会在该出现的时候出现。

举个例子：

糟糕的描述："帮忙写东西"

好的描述："当用户要给创始人、投资人或运营者写开发信时使用。涵盖语气、结构和后续跟进序列"

看出区别了吗？后面那个告诉 Claude 什么时候用、什么时候不用。具体，才能让 skill 真正发挥作用。

正文就是你放操作手册的地方。把它想成一份你第一天给新员工看的流程文档。步骤、格式、示例、注意事项、边界情况，都写进去。不要假设 Claude 会猜到你的偏好，全部说清楚。

## 我几乎每个 Skill 都用的结构

分享一下我的做法，这套结构几乎适用于所有 skill。

第一行写一句目标说明，让 Claude 知道这个 skill 是干什么的。

然后列出我通常会提供的输入，让 Claude 知道每次能从我这里拿到什么。

然后把实际流程写成编号步骤。这里是干货最集中的地方，要有立场。说"这样做，不要那样做"，不要含糊。

然后放两三个输入和理想输出的对比示例。这一块是 skill 写作里最被低估的部分。示例比任何规则都能更快地教会 Claude。一组好的示例，抵得上十段说明文字。

最后加一个"常见错误"的章节，列出 Claude 在这个任务上容易犯的错误。就这一个章节，帮我省掉的重复提示，比其他所有内容加起来还多。

## 保持 Skill 小而专注

这是我刚开始时最常犯的错误。

一个 skill 只做一件事。如果你想做一个"搞定所有营销"的 skill，描述会宽到没有边，Claude 不知道什么时候该用。

拆开来做才是对的：冷邮件一个，LinkedIn 帖子一个，落地页文案一个。

小的 skill 可以组合，大的 skill 只会垮掉。

## 把参考文件放进去

这是 skills 真正开始强大的地方。

你可以把任何文件丢进 skill 文件夹——品牌手册 PDF、样式表、批准用语列表、模板、JSON schema，什么都行。然后在 SKILL.md 里引用它，Claude 需要的时候会自己去读。

这就是真正个性化输出的来源。以前每次对话都要粘贴一遍品牌手册，现在放进文件夹一次就不用管了。Claude 在相关时自己打开，你会发现输出越来越像你自己写的。

## 让 Claude 帮你写 Skill

这是最省事的一招。

打开一个对话，告诉 Claude 你一直在反复做什么任务，粘贴两三个你以前是怎么做的示例，让它帮你起草一个 SKILL.md，然后你再改。

15 分钟就能做出一个可用的 skill，而不是两个小时。而且 Claude 写的描述通常比你自己写的更准——因为它见过太多这类文件了。

## Skills 可以互相调用

等你积累了几个 skill，有趣的事情才开始发生。

比如你可以设置一个收集信息的调研 skill，一个把调研内容变成草稿的写作 skill，还有一个输出最终格式的排版 skill。Claude 自己来安排这条链路，你只需要给出原始任务，剩下的 skills 之间会自动交接。

我亲眼看到这件事发生之前，是不相信的。有种突然多了一支小团队的感觉。

## 说到底，是这么回事

现在用 skills 用得顺手的人，不是文件夹最精致的那些。是那些认真留意过自己在哪里一遍遍重复的人。

每次把同样的提示词再打一遍，时间就在往外漏。一个 skill，就是把这个漏洞堵上一次，之后再也不用管它。

流程其实很朴素：发现重复，写 SKILL.md，在真实任务上试，出错就改，然后继续。

做五次，你就有了一个懂你工作方式的 Claude。做二十次，输出质量开始复利增长——那种感觉在你亲身体验到之前，真的很难形容。

下次再发现自己第三次在打同一段提示词，停下来。打开一个文件夹，写一个 SKILL.md。你刚给自己做了一次永久升级。
