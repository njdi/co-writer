# Claude 技能构建完整指南

**作者：** Clara Bennett ([@CodeswithClara](https://x.com/CodeswithClara))  
**日期：** 2026年4月29日  
**来源：** [The Complete Guide to Building Skills for Claude](https://x.com/Zephyr_hg/status/2049479378112041101)

过去几周我深入研究了 Claude Skills，我认为大多数人仍然没有意识到它实际上能解锁什么

如果你还不知道什么是 Skill，这里是最快的理解方式

一个 Skill 就是一个文件夹。文件夹里有一个叫 SKILL.md 的 markdown 文件，还可以有一些脚本、模板或参考文档。当 Claude 看到一个任务匹配你 SKILL.md 里的描述时，它会加载这个文件夹并遵循里面的指令

就这样。不需要微调，不需要向量数据库，不需要基础设施。只是一个 Claude 知道何时去取用的指令文件夹

这之所以是大事，是因为它解决了没有人谈论的 LLM 问题。模型很聪明，但它不知道你的工作流程。它不知道你的品牌声音、你的文件命名约定、你的报告格式、你技术栈的奇怪特点。你最终每次都在重新输入相同的上下文

Skills 解决了这个问题。你只需写一次上下文，Claude 在相关时自动调用

## 如何思考何时构建一个

不要坐下来试图从头发明一个 skill。那条路会导致你永远不会使用的过度设计的文件夹

等到你发现自己连续三次以相同的方式提示 Claude。那是你的信号。重复就是需求简报

我工作时总保持一个笔记文件打开，每当我发现自己在聊天中粘贴相同的上下文、相同的指令或相同的示例时，我就把它记下来。周末看这个列表，skills 基本上自己就写出来了

## SKILL.md 的解剖

每个 SKILL.md 最重要的有两部分。顶部的描述，和正文的指令

描述是整个文件中最重要的一行。这是 Claude 读取以决定你的 skill 是否与当前任务相关的内容。如果它模糊，你的 skill 就会在那里，永远不会被触发。如果它清晰具体，它会在正确的时候激活

糟糕的描述："helps with writing"

好的描述："use this when the user asks to write a cold outreach email to a founder, vc, or operator. covers tone, structure, and follow up sequence"

注意区别。第二个告诉 Claude 什么时候用它，什么时候不用。具体性是让 skills 起作用的原因

正文是你放实际操作手册的地方。把它想成你会在第一天交给聪明新员工的标准操作程序。包括步骤、格式、示例、需要避免的事情和边缘情况。不要假设 Claude 知道你的偏好。把它们一一说明

## 我几乎在每个 skill 中使用的结构

我以一句目标陈述开头，让 Claude 知道目标

然后我列出我通常提供的输入，让 Claude 知道对我有什么期待

然后我把实际过程写成编号步骤。这是大多数价值所在。要有观点。说"做这个，不做那个"

然后我加入两三个输入和理想输出的示例。这是 skill 写作中最被低估的部分。示例比你能写的任何规则都能更快地教会 Claude。一对好的前后对比胜过十段指令

然后我添加一个叫"common mistakes to avoid"的章节，列出 Claude 在这个任务上容易出错的事情。这单个章节比其他任何东西都为我节省了更多的重新提示

## 保持 skills 小而专注

这是我刚开始时最常打破的规则

一个 skill，一个工作。如果你试图构建一个"做所有营销"的 skill，描述会变得太宽泛以至于 Claude 永远不知道何时使用它。拆分它。有一个用于冷邮件的 skill，另一个用于 linkedin 帖子，另一个用于落地页文案

小的 skills 可以组合。大的 skills 会崩溃

## 使用参考文件

这是 skills 变得真正强大的地方

你可以把任何文件放入 skill 文件夹。一个品牌指南 PDF、一个样式表、一个批准短语列表、一个模板、一个 json schema，随便什么。然后从 SKILL.md 中引用它，Claude 需要时会调用它

这就是你如何获得真正个性化输出的方法。不需要在每次聊天中粘贴你的品牌指南，你把它放在 skill 文件夹里一次然后忘掉。Claude 在相关时打开它，你的输出开始听起来像是你写的

## 让 Claude 帮你构建 skill

这是最省时间的元操作

打开一个聊天，告诉 Claude 你一直在做什么任务，粘贴你之前是如何做的两三个示例，让它为你起草一个 SKILL.md。然后编辑

你将在 15 分钟而不是两小时内得到一个可用的 skill。而且 Claude 写的描述通常比你自己写的更清晰，因为它读过数千个这样的文件

## skills 可以调用其他 skills

一旦你构建了几个 skills，真正的魔法就会出现

你可以有一个收集信息的研究 skill，一个将研究变成草稿的写作 skill，以及一个输出最终文件的格式化 skill。Claude 自己编排这个链。你只需给出原始任务，skills 就会相互交接

我直到亲眼看到才相信这一点。感觉像是拥有一个小团队

## 元要点

现在用 skills 获胜的人不是拥有最花哨文件夹的人。他们是那些关注自己重复模式的人

每次你重新提示相同的事情，你都在浪费时间。一个 skill 就是一次性堵住那个漏洞，再也不用想它

循环很简单。注意重复。写 SKILL.md。在真实任务上测试它。当它出错时编辑它。继续前进

做五次，你就有了一个知道你如何工作的个人版本的 Claude。做二十次，你的输出质量会以一种在你感受到之前真的很难解释的方式复利增长

下次你发现自己第三次输入相同的提示时，停下来。打开一个文件夹。写一个 SKILL.md。你刚给了自己一个永久升级
