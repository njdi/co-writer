# 你睡觉的时候，Claude 在帮你干活

**作者：** Khairallah AL-Awady ([@eng_khairallah1](https://x.com/eng_khairallah1))  
**日期：** 2026年5月13日  
**来源：** [How to Set Up Claude Code Routines to Automate Any Workflow (Full Course)](https://x.com/Zephyr_hg/status/2054494450009932110)

Anthropic 上线了一个功能，几乎没人在聊。

收藏起来慢慢看 :)

它叫 Claude Code Routines。

我认为这是 Anthropic 今年发布的最重要的东西。先说说为什么。

以前用 Claude Code 做自动化，有一个绕不开的前提：你的电脑得开着。/loop 可以轮询，/schedule 可以定时，但只要你关掉终端或者合上笔记本，一切就停了。

Routines 把这件事彻底改掉了。

你配置一次——写一个提示词，选一个代码库，连上需要的服务——然后它就跑在 Anthropic 的云上。可以按计划跑，可以被 API 调用，也可以由 GitHub 事件触发。你的电脑关机了没关系。终端关掉了没关系。routine 照样在跑。

这是一个很实质性的转变：从"你在用的 AI 工具"，变成"替你干活的 AI 系统"。

下面一步一步讲怎么设置，哪怕你从来没用过 Claude Code。

## 和以前的调度有什么不一样

Claude Code 之前就有 /schedule 和 /loop，那新的 routines 改变了什么？

改变的是它跑在哪里。

旧的调度命令跑在你本地的 Claude Code 会话里——依赖你的机器开着、终端开着、网络稳定。哪一条断了，自动化就挂掉。

Routines 跑在 Anthropic 的云上。它是持久的，重启了没事，关终端了没事，跑一整晚没事。同时它还直接连着你的各种工具——Slack、Linear、Google Drive、GitHub——不需要你自己去搭桥。

打个比方：旧方式像是手机提醒。提醒会响，但活还是你自己干。

Routines 是那个你睡着之后在干活、你醒来给你发汇报的员工。

## 第一步：想清楚要自动化什么

不是所有任务都适合 routines。最合适的，有这三个共同点：

**重复发生。** 按固定节奏出现的，每天、每周，或者某个事件触发的。

**说得清楚。** 你能明确描述"做完"是什么样子，没有模糊地带。

**不太需要拍板。** 不需要你的创意判断，只需要执行。

看几个早期用户已经在跑的例子：

**积压任务管理。** 每天午夜，routine 从 Linear 拉取新问题，按类型和严重程度分类，打上标签，把摘要发到 Slack。工程负责人醒来看到的是一块干净的看板，不是一锅粥。

**文档漂移检测。** 每周五，routine 扫描这周合并的 PR，找出改了 API 或接口的那些，和现有文档对比，为已经过时的文档打开更新 PR。

**部署验证。** 每次部署后由 webhook 触发，routine 跑冒烟测试，扫错误日志，把问题关联到最近的代码变更，然后向发布频道发一个"通过/不通过"的结论。

**每日代码审查。** 每天早上 9 点，routine 拿起最旧的待审 PR，检查安全问题、逻辑漏洞和代码风格，发出内联评论。

设置了三四个这样任务的人，和还在把 Claude 当聊天工具用的人，效率不在一个层面。

## 第二步：创建第一个 Routine

创建方式有两种。

**网页界面：** 打开 claude.ai/code/routines，点"New routine"。可以配置完整的选项：计划触发、API 触发、GitHub 事件触发都有。

**CLI：** 如果你已经在终端里用 Claude Code，直接输 /schedule 加上描述就行，比如：

```
/schedule daily PR review at 9am
```

不过 CLI 只能创建计划型触发器。API 触发和 GitHub 触发要用网页界面。

创建 routine 时，要定好四件事：

**提示词。** 最关键的部分。routine 是自主跑的，没有人在旁边盯着，所以提示词必须完全自洽——代理需要知道的一切都得写进去。没有"上次对话留下的上下文"，每次运行都从零开始。

**代码库。** routine 处理哪个代码库。默认有完整读取权限，可以推送到 claude/ 前缀的分支。

**连接器。** routine 能访问哪些外部服务——Slack 发更新，Linear 管理问题，Google Drive 读写文档，GitHub 监控事件和开 PR。

**触发器。** 什么时候跑、怎么跑。计划型（每小时、每晚、每周），API 触发型（用代码调用），或 GitHub 触发型（代码库里发生特定事件时触发）。

## 第三步：写一个经得起考验的提示词

这是最多人栽跟头的地方。

routine 跑起来没人盯着。提示词含糊，代理每次理解都可能不一样，结果就忽好忽坏。

好的 routine 提示词，通常包含这几块：

**角色：** 告诉 Claude 它是谁。"你是一位专注于安全性和性能的高级代码审查员。"

**任务：** 说清楚要做什么。"审查这个代码库里最旧的待审 pull request。"

**步骤：** 一步一步说怎么做。"首先读 PR 描述，然后切到该分支，读改动的文件，分析安全漏洞、逻辑错误和性能问题，为每个发现的问题写内联评论。"

**输出要求：** 说清楚结果长什么样。"在 PR 上发一条摘要评论，包括：按严重程度分类的总问题数、一段整体评估，以及明确的'批准'或'请求更改'结论。"

**出错怎么办：** 边界情况要提前想到。"如果没有待审 PR，在 Slack 的 #engineering 频道发布'今天没有待审 PR。'如果一个 PR 的改动文件超过 50 个，跳过它并说明需要人工审查。"

**限制条件：** 划清不能做的事。"不要批准有 Critical 级别问题的 PR。不要直接修改代码，只写评论。每个文件最多三条内联评论。"

提示词写得越具体，routine 的表现就越稳定。

## 第四步：搞清楚有哪些限制

Routines 能干很多事，但目前有些边界要提前知道。

**每日运行上限。** 研究预览阶段，每个账户每天 15 次额度。需要更多，在组织设置里开启额外用量，超出部分按量计费。

**Token 消耗。** Routines 和交互式 Claude Code 会话共用订阅额度。读很多文件、发多个 API 请求的复杂 routine，消耗会明显更多。

**分支安全。** 默认 Claude 只能推送到 claude/ 前缀的分支。这是保险机制——写得有问题的 routine 不会意外推到 main。没有充分的下游审查流程，别关掉这个限制。

**GitHub 事件上限。** 预览阶段，GitHub 触发的 routine 有每小时的使用上限。代码库很活跃的话，记得过滤触发条件，别让无关的噪声把额度耗完。

**计划误差。** 高峰期计划型 routine 可能会有延迟。不要搭建依赖精确时间点的工作流。

## 第五步：把几个 Routine 组成一个系统

一个 routine 有用。几个放在一起，才是一套系统。

一个小型工程团队的 routine 栈可以长这样：

**每天早上 9 点 — 每日 PR 审查**  
Claude 审查所有待审 PR，发内联评论，向 Slack 发摘要，标出今天优先要处理的事。

**每次部署后（webhook）— 部署验证**  
部署到 staging 后，Claude 跑测试套件，扫日志里的错误，几分钟内给发布频道发通过/不通过的判定。

**每晚凌晨 2 点 — 积压任务分类**  
Claude 处理当天新提交的所有问题，打标签，评优先级，生成晨会简报。

**每周五下午 5 点 — 文档检查**  
Claude 扫本周合并的 PR，找出需要更新的文档，为每个打开草稿 PR。

**每周一早上 8 点 — 技术债务报告**  
Claude 扫代码库里的 TODO 注释、过时依赖项和测试覆盖率缺口，生成一份按工作量排序的清单。

每个 routine 配置需要 10 到 15 分钟。整套栈大概一个下午搞定。节省的时间，每周都在滚雪球。

## 第六步：看日志，持续改进

每次 routine 运行都会生成日志。要养成看的习惯。

重点看这几件事：

- 输出是否稳定达标？如果不是，提示词哪里写得模糊？
- 某些运行花了很长时间？可能范围定得太宽了。
- 遇到了报错？在提示词里加上对应的错误处理。
- 产生了太多噪声？收紧限制条件。

还有一个新功能值得一提：Dreaming。5 月 6 日的 Code with Claude 活动上刚宣布的。开启后，Claude 会在两次运行之间回顾自己过去的会话，找规律，自动调整下次的策略。

事情就是这么有意思——routines 运行越多，它自己会越来越聪明。

## 这个功能适合谁

Routines 是给这样的人设计的：

- 已经在用 Claude Code，或者愿意学
- 工作流里有清晰、可重复的任务
- 想把每周吃掉几小时的运营杂务自动掉
- 代码库托管在 GitHub 上

如果你是非技术用户，带计划任务的 Claude Cowork 是更合适的起点。Routines 是进阶工具。

但如果你是开发者、工程经理、DevOps 工程师，或者技术型创始人——routines 节省的时间，会比 Anthropic 出过的其他任何功能都多。

## Routines 和 GitHub Actions 的区别

很多人会问：GitHub Actions 免费，为什么要用 Claude routines？

问题在于，GitHub Action 是一段脚本。每一步你来写，每个条件你来定，每个边界情况你来处理。它完全按照你写的走，不多也不少。

Claude routine 是一个代理。你给它一个目标，它自己想怎么达到。它能应对意外，能推理问题，能验证自己的结果。

GitHub Action 跑完 linter，告诉你什么报错了。Claude routine 读懂报错，理解原因，提出修复方案，然后直接开一个带修正的 pull request。

这是两个完全不同的东西。脚本按规则走，代理解决问题。

简单的事——跑测试、格式检查、发通知——GitHub Actions 够用。但凡需要判断、分析、随机应变的，routines 是另一个档次。

## 今天就能用的五个配方

这五个配置，一小时内就能搭好：

**配方 1：晨间站会机器人**  
计划：每天上午 8:30  
提示词："检查 GitHub 代码库昨天推送的所有提交。检查 Linear 里新的和有更新的问题。检查 Slack #engineering 里提到阻塞点的消息。整理一份站会简报，分三部分：已完成、进行中、被阻塞。将简报发到 Slack 的 #daily-standup 频道。"

**配方 2：依赖项审计员**  
计划：每周一上午 6 点  
提示词："扫描 package.json 和 requirements.txt 里的所有依赖项。用网络检查每个依赖项的已知漏洞。找出落后当前版本超过两个主要版本的依赖项。生成一份带严重程度评级的优先报告，如果发现 Critical 漏洞，打开一个 GitHub issue。"

**配方 3：变更日志生成器**  
触发器：GitHub 事件——新发布标签推送  
提示词："检测到新发布标签时，读取自上个标签以来的所有提交，将每个提交分类为功能、修复、改进或杂务，在 CHANGELOG.md 中生成格式化的变更日志并打开一个 PR。"

**配方 4：测试覆盖率监控器**  
计划：每晚凌晨 1 点  
提示词："跑测试套件，按模块计算覆盖率。与 coverage-config.json 里的基准对比。如果任何模块的覆盖率比基准低超过 2%，打开一个 GitHub issue，注明具体模块、旧覆盖率、新覆盖率，以及可能导致下降的提交。"

**配方 5：PR 描述检查员**  
触发器：GitHub 事件——新 PR 打开  
提示词："有新 PR 打开时，检查描述是否满足模板要求：必须有摘要部分、测试部分，如果有 UI 改动还要有截图部分。如果缺少任何部分，发一条礼貌的评论，请作者在审查前补全。"

每个配方不超过 10 分钟能配好。加在一起，每个月能为团队省几十小时。
