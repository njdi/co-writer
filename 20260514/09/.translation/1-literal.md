# 如何设置 Claude Code Routines 来自动化任何工作流（完整课程）

**作者：** Khairallah AL-Awady ([@eng_khairallah1](https://x.com/eng_khairallah1))  
**日期：** 2026年5月13日  
**来源：** [How to Set Up Claude Code Routines to Automate Any Workflow (Full Course)](https://x.com/Zephyr_hg/status/2054494450009932110)

Anthropic 发布了一个功能，几乎没有人在讨论它。

收藏并保存这篇文章 :)

它叫做 Claude Code Routines（例行程序）。

它可能是 Anthropic 今年发布的最重要的功能。

原因如下。

到目前为止，每一个 Claude Code 自动化操作都需要你的笔记本电脑保持开启状态。你可以用 /loop 来轮询变化，用 /schedule 设置定期任务。但是一旦你关闭终端或关闭笔记本电脑，一切就停止了。

Routines 彻底解决了这个问题。

一个 routine 是一种 Claude Code 自动化操作，你只需配置一次——一个提示词、一个代码库、一组连接器——然后它就会在 Anthropic 的云基础设施上运行。按计划、通过 API 调用或由 GitHub 事件触发。

你的笔记本电脑可以关机。你的终端可以关闭。routine 照样运行。

这是从"你使用的 AI 工具"到"为你工作的 AI 系统"的转变。

下面是如何设置它的详细步骤，即使你以前从未使用过 Claude Code。

## 为什么 Routines 与其他功能不同

Claude Code 本来就有调度功能。那么改变了什么？

区别在于基础设施。

旧的 /schedule 和 /loop 命令在你本地的 Claude Code 会话中运行。它们依赖于你的机器开机、终端打开，以及网络连接稳定。如果其中任何一个出问题，自动化就会中断。

Routines 运行在 Anthropic 的云端。它们是持久的自主代理，能够在重启、终端关闭和隔夜运行中继续存活。它们可以直接访问你的代码库和连接器——Slack、Linear、Google Drive、GitHub——无需你自己管理任何这些。

把旧系统想象成手机上的一个提醒。它会响，但你仍然需要自己做工作。

Routines 就是那个在你睡觉时做工作、你醒来时给你发摘要的员工。

## 第一步：决定要自动化什么

最好的 routines 自动化的任务是：

**重复性的** — 它们按可预测的计划发生（每天、每周，或由事件触发）。

**定义明确的** — 你可以明确地描述"完成"是什么样子，没有歧义。

**低判断性的** — 该任务不需要你独特的创意思维或决策能力。它需要的是执行。

以下是早期用户目前正在运行的模式：

**积压任务管理** — 每天午夜，routine 从 Linear 拉取新问题，按类型和严重程度分类，分配标签，并将摘要发布到 Slack 频道。工程负责人醒来时看到的是一个干净、有组织的看板。

**文档漂移检测** — 每个周五，routine 扫描过去一周合并的 pull request，找出任何更改了 API 或接口的 PR，对照文档进行交叉引用，并为已过期的文档打开更新 PR。

**部署验证** — 每次部署后由 webhook 触发，routine 对新构建运行冒烟测试，扫描错误日志中的回归问题，将任何问题与最近的代码更改关联起来，并向发布频道发布通过/不通过的判定。

**每日代码审查** — 每天早上 9 点，routine 拿起最旧的待审查 PR，检查安全问题、逻辑错误和风格违规，并发布内联评论。

设置了三四个这样的 routine 的人，其运营层次与仍在将 Claude 作为聊天工具使用的人完全不同。

## 第二步：创建你的第一个 Routine

创建 routine 有两种方式。

**通过网页界面：** 前往 claude.ai/code/routines 并点击"New routine"。这提供了完整的配置选项——计划触发器、API 触发器和 GitHub 事件触发器。

**通过 CLI：** 如果你已经在终端中使用 Claude Code，输入 /schedule 后跟描述。例如：

```
/schedule daily PR review at 9am
```

CLI 只创建基于计划的触发器。对于 API 和 GitHub 触发器，你需要使用网页界面。

当你创建 routine 时，需要配置四件事：

**提示词** — 这是最关键的部分。由于 routine 是自主运行的，提示词需要完全自洽。代理需要知道的一切都必须包含在提示词中。没有"来自之前对话的上下文"。每次运行都从零开始。

**代码库** — routine 处理的是哪个代码库。默认情况下，它获得完整读取权限，并可以推送到以 claude/ 为前缀的分支。

**连接器** — routine 可以访问哪些外部服务。Slack 用于发布更新，Linear 用于读取和管理问题，Google Drive 用于读取和写入文档，GitHub 用于监控事件和打开 PR。

**触发器** — routine 何时以及如何运行。计划型（每小时、每晚、每周），API 触发型（你以编程方式调用它），或 GitHub 触发型（当你的代码库中发生特定事件时触发）。

## 第三步：写一个无懈可击的提示词

这是大多数人失败的地方。

routine 在没有你监视的情况下运行。如果提示词模糊，代理每次都会以不同的方式解释它，你会得到不一致的结果。

最好的 routine 提示词遵循以下结构：

**角色定义：** "你是一位专注于安全性和性能的高级代码审查员。"

**任务定义：** "审查此代码库中最旧的待审查 pull request。"

**逐步流程：** "首先，阅读 PR 描述。然后检查分支。阅读已更改的文件。分析安全漏洞、逻辑错误和性能问题。为每个发现的问题写内联评论。"

**输出规范：** "在 PR 上发布摘要评论，包括：发现的总问题数（按严重程度）、一段总体评估，以及明确的批准/请求更改判定。"

**错误处理：** "如果没有待审查的 PR，在 Slack 的 #engineering 频道发布'今天没有待审查的 PR。'如果一个 PR 有超过 50 个已更改的文件，跳过它并发布说明它需要人工审查。"

**约束：** "永远不要批准有严重程度为 Critical 的问题的 PR。永远不要直接修改任何代码——只添加评论。每个文件最多三条内联评论，以避免噪声。"

你的提示词越精确，你的 routine 就越可靠。

## 第四步：了解限制

Routines 很强大，但有一些你需要了解的约束。

**每日运行上限：** 在研究预览期间，每个账户每天可以运行 15 次 routine。如果需要更多，在组织设置中启用额外使用——额外运行按量计费。

**Token 消耗：** Routines 从与交互式 Claude Code 会话相同的订阅限制中提取。读取许多文件并进行多个 API 调用的复杂 routine 将比简单的 routine 消耗更多的 token。

**分支安全：** 默认情况下，Claude 只能推送到以 claude/ 为前缀的分支。这是一项安全措施——写得不好的 routine 不会意外推送到 main 分支。除非你在下游有健全的审查流程，否则不要禁用这个限制。

**GitHub 事件上限：** 在预览期间，GitHub 触发的 routine 有每个 routine 和每个账户的每小时上限。如果你的代码库非常活跃，过滤哪些事件触发 routine，以避免在噪声上浪费运行次数。

**计划依赖性：** 计划 routine 在指定时间运行，但在高需求时期可能会有偏差。不要构建依赖于精确到秒的计时的工作流。

## 第五步：构建 Routine 栈

一个 routine 是有用的。一栈 routine 是一个系统。

以下是一个小型工程团队完整 routine 栈的样子：

**早上（9 点）— 每日 PR 审查**  
Claude 审查所有待审查的 PR，发布内联评论，并向 Slack 发送摘要，包含今天需要关注的事项的优先列表。

**部署后（webhook）— 部署验证**  
每次部署到 staging 时，Claude 运行测试套件，扫描日志中的错误，并在几分钟内向发布频道发布通过/不通过的判定。

**每夜（凌晨 2 点）— 积压任务分类**  
Claude 处理当天提交的所有新问题，添加标签，分配优先级分数，并创建晨会简报文档。

**每周（周五下午 5 点）— 文档检查**  
Claude 扫描本周合并的 PR，找出需要更新的文档，并为每个文档打开草稿 PR。

**每周（周一上午 8 点）— 技术债务报告**  
Claude 扫描代码库中的 TODO 注释、已废弃的依赖项和测试覆盖率缺口。生成一份按估计工作量排名的技术债务项目列表。

每个 routine 需要 10-15 分钟来设置。整个栈需要一个下午。节省的时间每周都在复利增长。

## 第六步：监控和改进

每次 routine 运行都会生成日志。仔细查看它。

寻找规律：

- routine 是否持续产生良好的输出？如果没有，提示词的哪个部分是模糊的？
- 在某些运行中是否花费了太长时间？你可能需要缩小范围。
- 它是否遇到了错误？在提示词中添加明确的错误处理。
- 它是否产生了太多噪声？收紧约束条件。

新的"Dreaming"功能——于 5 月 6 日的 Code with Claude 活动上宣布——将这一点推向更远。启用 Dreaming 后，Claude 会在两次运行之间回顾自己过去的 routine 会话，识别哪些做法有效、哪些没有效果，并自我改进下次的方法。

你的 routines 运行得越多，就真的会变得越来越聪明。

## 这适合谁（以及不适合谁）

Routines 是为开发者和技术型运营人员构建的，他们：

- 已经使用 Claude Code 或愿意学习
- 工作流中有遵循清晰模式的可重复任务
- 希望自动化每周吞噬数小时的运营开销
- 在 GitHub 上托管代码库

如果你是寻找 AI 自动化的非技术用户，带有计划任务的 Claude Cowork 是更好的起点。Routines 是高级用户工具。

但如果你是开发者、工程经理、DevOps 工程师或技术创始人——routines 将为你节省比 Anthropic 推出的任何其他功能更多的时间。

## Routines 与 GitHub Actions：有什么区别

许多开发者会问：当 GitHub Actions 是免费的时候，为什么要为 Claude routines 付费？

答案是 GitHub Action 是一个脚本。你编写每一步。你定义每一个条件。你自己处理每一个边缘情况。它完全按照你编写的代码执行，不多也不少。

Claude routine 是一个代理。你给它一个目标。它决定如何达到目标。它适应意外情况。它推理问题。它验证自己的工作。

GitHub Action 运行一个 linter 并告诉你什么失败了。Claude routine 读取错误，理解为什么失败，提出修复方案，并打开一个包含修正的 pull request。

这是本质上不同的类别。脚本遵循规则。代理解决问题。

对于简单的自动化——运行测试、检查格式、发送通知——GitHub Actions 是合适的。对于任何需要判断、分析或适应能力的事情，routines 处于另一个层次。

## 你今天就可以复制的常用 Routine 配方

以下是五种你可以在接下来一小时内设置的 routine 配置：

**配方 1：晨间站会机器人**  
计划：每天上午 8:30  
提示词："检查 GitHub 代码库中昨天推送的所有提交。检查 Linear 中的新问题和已更新的问题。检查 Slack #engineering 中提到障碍的任何消息。编制一份包含三个部分的站会简报：已完成的工作、正在进行的工作和被阻塞的工作。将简报发布到 Slack 的 #daily-standup 频道。"

**配方 2：依赖项审计员**  
计划：每周一上午 6 点  
提示词："扫描 package.json 和 requirements.txt 中的所有依赖项。使用网络检查每个依赖项的已知漏洞。识别任何落后当前版本超过两个主要版本的依赖项。创建一份带有严重程度评级的优先报告，如果发现任何严重漏洞，打开一个 GitHub 问题。"

**配方 3：变更日志生成器**  
触发器：GitHub 事件——新的发布标签被推送  
提示词："当新的发布标签被推送时，读取自上一个标签以来的所有提交。将每个提交分类为功能、修复、改进或杂务。在 CHANGELOG.md 中生成格式化的变更日志并打开一个 PR。"

**配方 4：测试覆盖率监控器**  
计划：每晚凌晨 1 点  
提示词："运行测试套件。按模块计算覆盖率百分比。与 coverage-config.json 中的覆盖率基准进行比较。如果任何模块的覆盖率比其基准低超过 2%，打开一个 GitHub 问题，包含具体模块、旧覆盖率、新覆盖率以及可能导致下降的提交。"

**配方 5：PR 描述执行者**  
触发器：GitHub 事件——新的 PR 被打开  
提示词："当新的 PR 被打开时，检查描述是否满足我们的模板要求：必须包含摘要部分、测试部分，以及如果涉及 UI 更改，还需要截图部分。如果缺少任何部分，发布一条礼貌的评论，请求作者在审查前更新描述。"

每个配方配置不超过 10 分钟。合在一起，每个月可以为团队节省数十个小时。

## 结论

旧方式：醒来，打开终端，开始 Claude Code 会话，输入命令，等待结果，继续下一个任务。明天重复。

新方式：一次性配置 routines，让它们在 Anthropic 的云端运行，醒来时查看结果。

这不是理论上的改进。人们已经在运行 routine 栈，让它们在一夜之间处理整个运营工作流。

"将 Claude 用作聊天机器人的人"和"让 Claude 全天候自主工作的人"之间的差距每周都在扩大。

Routines 是你跨越到另一边的方式。

大多数人读完这篇文章会想"我应该什么时候设置一下。"那些今天真正创建第一个 routine 的人，下周就会有一个系统在运行，每个月为他们节省数小时。
