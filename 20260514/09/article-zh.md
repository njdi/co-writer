# 如何设置 Claude Code Routines 来自动化任何工作流（完整课程）

**作者：** Khairallah AL-Awady ([@eng_khairallah1](https://x.com/eng_khairallah1))  
**日期：** 2026年5月13日  
**来源：** [How to Set Up Claude Code Routines to Automate Any Workflow (Full Course)](https://x.com/Zephyr_hg/status/2054494450009932110)

Anthropic 发布了一个功能，几乎没有人在聊这件事。

收藏起来慢慢看 :)

它叫 Claude Code Routines。

可能是 Anthropic 今年发布的最重要的功能。

原因如下。

以前，每一个 Claude Code 自动化任务都需要你的电脑保持开机。你可以用 /loop 轮询变化，用 /schedule 设置定时任务。但只要你关掉终端或合上笔记本，一切就停了。

Routines 把这个问题彻底解决了。

一个 routine，就是你配置一次的 Claude Code 自动化任务——一个提示词、一个代码库、一组连接器——然后它就跑在 Anthropic 的云上。可以按计划跑，可以通过 API 调用，也可以由 GitHub 事件触发。

你的电脑可以关机。终端可以关掉。routine 照样在跑。

这是从"你用的 AI 工具"到"为你干活的 AI 系统"的转变。

下面讲清楚怎么设置，哪怕你从没用过 Claude Code。

## 为什么 Routines 和其他功能不一样

Claude Code 本来就有调度功能。那改变的是什么？

是基础设施。

旧的 /schedule 和 /loop 命令跑在你本地的 Claude Code 会话里。依赖你的机器开着、终端开着、网络稳定。哪个条件不满足，自动化就挂掉。

Routines 跑在 Anthropic 的云上。它们是持久的自主代理，重启也好、关终端也好、跑一整晚也好，都不会中断。它们能直接访问你的代码库和各种连接器——Slack、Linear、Google Drive、GitHub——不需要你自己去管这些。

旧方式就像手机上的一个提醒。它会响，但活还是要你自己干。

Routines 是那个你睡觉时在干活、你醒来时给你发汇报的员工。

## 第一步：决定要自动化什么

最适合用 routines 自动化的任务，有这三个特征：

**重复性的** — 按可预测的节奏发生（每天、每周，或由某个事件触发）。

**定义清晰的** — 你能清楚说出"完成"是什么样子，没有歧义。

**不需要太多判断的** — 不需要你的创意思维或决策，只需要执行。

以下是早期用户已经在跑的几种模式：

**积压任务管理** — 每天午夜，routine 从 Linear 拉取新问题，按类型和严重程度分类，打上标签，并把摘要发到 Slack 频道。工程负责人醒来时看到的是一块干净、有序的看板。

**文档漂移检测** — 每个周五，routine 扫描过去一周合并的 PR，找出那些改动了 API 或接口的提交，和现有文档交叉比对，并为已经过时的文档打开更新 PR。

**部署验证** — 每次部署后由 webhook 触发，routine 对新构建跑冒烟测试，扫描错误日志里的回归问题，把问题关联到最近的代码变更，然后向发布频道发一个"通过/不通过"的结论。

**每日代码审查** — 每天早上 9 点，routine 拿起最旧的待审 PR，检查安全问题、逻辑错误和代码风格，然后发出内联评论。

设置了三四个这样的 routine 的人，运作效率已经和那些还在把 Claude 当聊天工具用的人不在一个层面上了。

## 第二步：创建你的第一个 Routine

创建 routine 有两种方式。

**通过网页界面：** 访问 claude.ai/code/routines，点击"New routine"。这里可以配置完整的选项——计划触发器、API 触发器和 GitHub 事件触发器。

**通过 CLI：** 如果你已经在终端里用 Claude Code，输入 /schedule 加上描述就行。比如：

```
/schedule daily PR review at 9am
```

CLI 只能创建基于计划的触发器。API 和 GitHub 触发器需要用网页界面。

创建 routine 时，需要配置四件事：

**提示词** — 最关键的一部分。routine 是自主运行的，提示词必须完全自洽。代理需要知道的一切都要写进去。没有"从上一次对话带来的上下文"，每次运行都是全新开始。

**代码库** — routine 处理哪个代码库。默认获得完整读取权限，可以推送到以 claude/ 为前缀的分支。

**连接器** — routine 可以访问哪些外部服务。Slack 用来发更新，Linear 用来读取和管理问题，Google Drive 用来读写文档，GitHub 用来监控事件和开 PR。

**触发器** — routine 什么时候、用什么方式运行。计划型（每小时、每晚、每周），API 触发型（你用代码调用它），或 GitHub 触发型（代码库里发生特定事件时自动触发）。

## 第三步：写一个经得起考验的提示词

这是大多数人卡住的地方。

routine 在没人盯着的情况下运行。提示词如果含糊，代理每次的理解都可能不一样，结果就会忽好忽坏。

好的 routine 提示词，结构是这样的：

**角色定义：** "你是一位专注于安全性和性能的高级代码审查员。"

**任务定义：** "审查这个代码库里最旧的待审 pull request。"

**逐步流程：** "首先读 PR 描述，然后切到该分支，读改动的文件，分析安全漏洞、逻辑错误和性能问题，并为每个发现的问题写内联评论。"

**输出要求：** "在 PR 上发一条摘要评论，包括：按严重程度分类的总问题数、一段整体评估，以及明确的'批准'或'请求更改'结论。"

**错误处理：** "如果没有待审 PR，在 Slack 的 #engineering 频道发布'今天没有待审 PR。'如果一个 PR 的改动文件超过 50 个，跳过它并说明需要人工审查。"

**限制条件：** "不要批准有 Critical 级别问题的 PR。不要直接修改任何代码，只写评论。每个文件最多三条内联评论，避免噪声太多。"

提示词越精确，routine 就越稳定。

## 第四步：了解限制

Routines 很强大，但有些限制需要提前知道。

**每日运行上限：** 研究预览阶段，每个账户每天有 15 次 routine 运行额度。需要更多的话，在组织设置中开启额外用量——超出部分按量计费。

**Token 消耗：** Routines 和交互式 Claude Code 会话共用同一个订阅额度。读很多文件、发起多个 API 调用的复杂 routine，消耗的 token 会比简单的多很多。

**分支安全：** 默认情况下，Claude 只能推送到以 claude/ 为前缀的分支。这是一道保险——写得有问题的 routine 不会意外推到 main 分支。除非你下游有完善的审查流程，否则不要关掉这个限制。

**GitHub 事件上限：** 在预览阶段，GitHub 触发的 routine 有每个 routine 和每个账户的每小时上限。如果你的代码库很活跃，记得过滤哪些事件才触发 routine，别让无关的噪声把运行次数耗光。

**计划误差：** 计划型 routine 会在指定时间运行，但高峰期可能有延迟。不要搭建那种依赖精确到秒的时间节点的工作流。

## 第五步：搭建 Routine 栈

一个 routine 有用。一组 routine 才是系统。

一个小型工程团队的完整 routine 栈长这样：

**早上（9 点）— 每日 PR 审查**  
Claude 审查所有待审 PR，发布内联评论，并向 Slack 发送摘要，列出今天优先需要关注的事项。

**部署后（webhook）— 部署验证**  
每次部署到 staging，Claude 跑测试套件，扫描日志里的错误，几分钟内向发布频道发出通过/不通过的判定。

**每夜（凌晨 2 点）— 积压任务分类**  
Claude 处理当天新提交的所有问题，打标签，评定优先级，并生成晨会简报文档。

**每周（周五下午 5 点）— 文档检查**  
Claude 扫描本周合并的 PR，找出需要更新的文档，为每个文档打开草稿 PR。

**每周（周一上午 8 点）— 技术债务报告**  
Claude 扫描代码库中的 TODO 注释、已废弃的依赖项和测试覆盖率缺口，生成一份按预估工作量排序的技术债务清单。

每个 routine 设置需要 10 到 15 分钟。整个栈大概需要一个下午。节省下来的时间，每周都在复利积累。

## 第六步：监控和改进

每次 routine 运行都会生成日志。仔细看。

找规律：

- routine 是否稳定产出好结果？如果没有，提示词哪里写得模糊？
- 某些运行是否花了太长时间？可能需要缩小范围。
- 是否遇到了报错？在提示词里加上明确的错误处理。
- 是否产生了太多噪声？收紧限制条件。

新的"Dreaming"功能——5 月 6 日的 Code with Claude 活动上宣布的——把这件事推进了一步。开启 Dreaming 后，Claude 会在两次运行之间回顾自己过去的 routine 会话，识别哪些做法有效、哪些没效果，并自动调整下次的策略。

routines 运行越多，就真的会越来越聪明。

## 这适合谁（以及不适合谁）

Routines 是为这样的开发者和技术型运营人员设计的：

- 已经在用 Claude Code，或者愿意学
- 工作流里有遵循清晰模式的可重复任务
- 想把每周吞掉好几小时的运营杂务自动化
- 代码库托管在 GitHub 上

如果你是寻找 AI 自动化方案的非技术用户，带有计划任务功能的 Claude Cowork 是更合适的起点。Routines 是给进阶用户的工具。

但如果你是开发者、工程经理、DevOps 工程师，或者技术型创始人——routines 能为你节省的时间，比 Anthropic 发布的其他任何功能都多。

## Routines 和 GitHub Actions 有什么区别

很多开发者会问：GitHub Actions 是免费的，为什么还要为 Claude routines 付钱？

答案是：GitHub Action 是一段脚本。你写每一步，你定义每一个条件，你自己处理每一个边界情况。它完全按照你写的代码执行，一分不多。

Claude routine 是一个代理。你给它一个目标，它自己决定怎么达到。它能适应意外情况，能推理问题，能验证自己的工作。

GitHub Action 跑完 linter 告诉你什么失败了。Claude routine 读懂报错，理解为什么失败，提出修复方案，再打开一个带修正的 pull request。

这是完全不同的两个类别。脚本遵守规则。代理解决问题。

简单的自动化——跑测试、检查格式、发通知——GitHub Actions 够用。但凡需要判断、分析或随机应变的事，routines 是另一个档次。

## 今天就能用的 Routine 配方

以下五个配置，你在接下来一小时内就能搞定：

**配方 1：晨间站会机器人**  
计划：每天上午 8:30  
提示词："检查 GitHub 代码库昨天推送的所有提交。检查 Linear 里新的和有更新的问题。检查 Slack #engineering 里提到阻塞点的消息。整理一份站会简报，分三部分：已完成、进行中、被阻塞。将简报发到 Slack 的 #daily-standup 频道。"

**配方 2：依赖项审计员**  
计划：每周一上午 6 点  
提示词："扫描 package.json 和 requirements.txt 里的所有依赖项。用网络检查每个依赖项的已知漏洞。找出落后当前版本超过两个主要版本的依赖项。生成一份带严重程度评级的优先报告，如果发现 Critical 漏洞，打开一个 GitHub issue。"

**配方 3：变更日志生成器**  
触发器：GitHub 事件——新的发布标签被推送  
提示词："检测到新的发布标签时，读取自上个标签以来的所有提交，将每个提交分类为功能、修复、改进或杂务，在 CHANGELOG.md 中生成格式化的变更日志并打开一个 PR。"

**配方 4：测试覆盖率监控器**  
计划：每晚凌晨 1 点  
提示词："跑测试套件，按模块计算覆盖率。与 coverage-config.json 里的基准对比。如果任何模块的覆盖率比基准低超过 2%，打开一个 GitHub issue，注明具体模块、旧覆盖率、新覆盖率，以及可能导致下降的提交。"

**配方 5：PR 描述检查员**  
触发器：GitHub 事件——新 PR 被打开  
提示词："有新 PR 打开时，检查描述是否满足模板要求：必须包含摘要部分、测试部分，如果有 UI 改动还需要截图部分。如果缺少任何部分，发一条礼貌的评论，请作者在审查前补全。"

每个配方配置不超过 10 分钟。放在一起，每个月能为团队节省几十小时。

## 说到底

旧方式：醒来，打开终端，开一个 Claude Code 会话，输命令，等结果，继续下一个任务。明天再重复一遍。

新方式：一次性配置好 routines，让它们在 Anthropic 的云上跑，你醒来查看结果就行。

这不是纸面上的改进。已经有人在跑一整套 routine 栈，让它们在一夜之间处理完所有运营工作了。

"把 Claude 当聊天工具用的人"和"让 Claude 全天候自主干活的人"之间的差距，每周都在拉大。

Routines 就是你跨过去的那步。
