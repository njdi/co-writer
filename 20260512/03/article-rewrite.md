# 自动改进的软件

**作者：** Ashpreet Bedi ([@ashpreetbedi](https://x.com/ashpreetbedi))  
**日期：** 2026年5月12日  
**来源：** [Auto-Improving Software](https://x.com/Zephyr_hg/status/2053885390717890757)

编码 agent 改变了我们构建软件的方式。现在，它们开始改变我们改进软件的方式。

今天我想分享一个 agent 平台——编码 agent 可以在上面自行构建、运行，并改进其他 agent。

---

整个 agent 开发生命周期，被五条指令覆盖：

- **Create（创建）**：搭建新 agent 的脚手架。
- **Improve（改进）**：根据 agent 自身的规格对其进行强化。
- **Extend（扩展）**：为现有 agent 添加新能力。
- **Hill Climb（爬坡）**：运行评测套件，诊断失败，修复在范围内的问题。
- **Review（审查）**：扫描仓库，检查文档、代码和配置之间的漂移。

"Improve → Hill Climb"这个循环，在几乎无需人工干预的情况下，持续迭代改进我的 agent。很难想象手动做这些是什么感觉。

顺便说一下，这个自动改进循环之所以可行，是因为整个环境从一开始就是为此设计的。agent 代码、traces、日志、评测套件和正在运行的软件，全部在同一个地方。这样，编码 agent 才能端到端地把整个流程跑起来。

## 之所以能工作，是因为我们控制了技术栈

大多数软件无法自动改进，根本原因是：输入和输出分散在各种工具里。要跑一个自动改进循环，编码 agent 必须把来自三个不同工具的数据拼凑在一起，每个工具都有各自的认证方式和操作习惯。

理论上可行。但实际摩擦太大，根本跑不起来。

我的代码库是专门为自动改进设计的。举个例子：Claude Code 可以测试一个 agent，然后通过读取 sessions、traces 和日志来判断 PASS 或 FAIL。如果失败，它就去编辑那个 agent，然后重新运行。

三件事让这成为可能。

**每个操作都作为 API 暴露。** 运行 agent、读取 session、运行评测——每个关键操作都可以用 cURL 或 Bash 触发。

**数据是同位存储的。** Sessions 和 traces 存在 Postgres 数据库里。编码 agent 可以触发一次运行，直接读取输出，不需要离开自己的环境。

**日志优先于一切。** 整个平台在本地 Docker 上运行。编码 agent 读取实时日志，按需更新。测试 → 审查的循环大概是 5 秒。日志是解锁一切的那个实时反馈。

说到底，agent 平台是第一类软件——操作、数据和迭代工具彼此足够近，让编码 agent 可以端到端测试、修改代码、再测试，直到 agent 变好。也就是说，托管这个循环的平台，本身就是循环首先改进的对象。

## Agent 开发生命周期

接下来我展示一下 Claude Code 是怎么运行我的 agent 平台的。

### 1. 创建一个 agent

要创建新 agent，我打开 Claude Code，输入：

> Run create-new-agent.md in a new branch.

Claude 会先问几个问题：这个 agent 要做什么？需要哪些工具？然后它通过 MCP 搜索 Agno 文档，找到合适的工具包，生成 agent 文件，在 `app/main.py` 里注册，重启容器，用 cURL 跑冒烟测试。从一条指令到一个可用的 agent，5 到 10 分钟。

因为平台把所有事情都处理好了，我开始构建以前根本不会费心去做的东西：一个汇总隔夜积压 Slack 消息的 agent，一个起草每周更新的 agent，一个在仓库里突出显示重要问题的 agent。这些都不值得花超过一天的时间。但它们全都适合在喝咖啡的时间里完成。

### 2. 改进一个 agent

要改进现有 agent，我输入：

> Run improve-agent.md on code-search agent.

Claude 读取 agent 的 INSTRUCTIONS，从中派生出 8 到 12 个探针。有些测主路径，有些测边缘情况，有些测工具选择。还有几个是对抗性的：提示注入、格式错误的输入、试图让 agent 偏离目标的尝试。

它通过 cURL 对实时容器逐一运行探针，读取响应，从容器日志里读取工具调用，根据 INSTRUCTIONS 实际承诺的内容来判断 PASS 或 FAIL。

针对每次失败，它选择相应的修复手段：收紧规则、添加规则、替换工具、提高 `num_history_runs`——选择适合这个失败模式的那种。编辑 `agents/<slug>.py`，热重载，只重新运行失败的探针。

然后迭代。上限五轮，如果全部通过则提前停止。

除了启动任务，不需要我做任何事。以前这要花整整一天手动点来点去，现在完全自动化了。

### 3. 扩展一个 agent

要为现有 agent 添加能力，我输入：

> Run extend-agent.md on code-search agent.

Extend 是在我主导下运行的。我描述一个变更：添加工具、优化提示词、修复 bug。Claude 来执行。Agno 文档 MCP 已经加载，所以工具包的研究基于真实的 API。

Claude 做出修改，跑冒烟测试。每次迭代都是一个小的、经过验证的步骤，变更精准，在隔离状态下测试。

### 4. Hill Climb（爬坡）

随着时间推移，eval 越积越多。如果手动逐一修复失败，会很繁琐。我只需输入：

> Run eval-and-improve.md.

Hill Climb 跑评测套件，诊断每次失败，修复在范围内的问题。失败类型对应修复位置：INSTRUCTIONS 中缺少的规则、幻觉、触发了错误的工具、过于具体的评分标准。针对每次失败，Claude 选对修复手段，编辑，只重新运行失败的用例。全部变绿之后，再跑一遍完整套件确认没有回归。

评测套件只有两个文件。`evals/cases.py` 声明用例，每个用例是一个输入加一个评分标准，还有可选的预期工具调用。基于 Agno 的 `AgentAsJudgeEval` 和 `ReliabilityEval` 构建。

Improve 捕获分布外的失败，Hill Climb 确保分布内的用例继续通过。两者配合得很好。

### 5. Review（审查）

仓库主要由编码 agent 管理，移动很快。为了让一切保持同步，我输入：

> Run review-and-improve.md.

Claude 扫描整个仓库，检查文档、代码和配置之间的漂移。磁盘上的每个 agent 文件都应该在 `app/main.py` 里注册。代码读取的每个环境变量都应该在 `example.env` 和 `AGENTS.md` 里。Markdown 里的每个路径都应该还存在。每个脚本都应该做到它声称的事情。

机械性漂移被原地自动修复：重命名的文件、`example.env` 里缺失的条目、架构图里遗漏的新 agent。更大的问题被标记，附带建议的下一步。

最好在发布前或重构后跑一遍。这类工作对人类来说很繁琐，但对能读取仓库每个文件的编码 agent 来说，是小事一桩。

文档与代码之间的偏离，历来是生产软件的隐性成本。现在它不再花费任何代价了。

## 为什么选择 Agent 平台？

Agent 平台是检验这个模式的完美场景。

**全新开始。** 平台相对较新，可以从一开始就专门为编码 agent 而设计。

**工作流程清晰。** 我们知道怎么改进一个 agent：运行它，读日志，对响应评分，编辑，再运行。

**循环真的有用。** 对普通软件来说，优化一个 API 端点没太大意义。但对 agent 来说，每轮改进都是真实的、可衡量的，并且会带来实际价值。

正确搭好平台，你就可以在上面构建任何 agent：用 create 工作流从想法到 agent，用 improve 工作流强化它，用 extend 工作流添加新能力，用 eval 把它锁定，然后 Hill Climb。

用 review-and-improve 工作流保持整个仓库同步。

手动做这些，几乎是不可能的。
