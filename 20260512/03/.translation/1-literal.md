# 自动改进的软件

**作者：** Ashpreet Bedi ([@ashpreetbedi](https://x.com/ashpreetbedi))  
**日期：** 2026年5月12日  
**来源：** [Auto-Improving Software](https://x.com/Zephyr_hg/status/2053885390717890757)

编码 agent 改变了我们构建软件的方式。现在它们正在改变我们改进软件的方式。今天我将分享一个 agent 平台，编码 agent 可以在上面自行构建、运行和改进 agent。

整个 agent 开发生命周期由五条指令覆盖：

- **Create（创建）**：搭建一个新 agent 的脚手架。
- **Improve（改进）**：根据 agent 自身的规格对其进行强化。
- **Extend（扩展）**：为现有 agent 添加新能力。
- **Hill Climb（爬坡）**：运行评测套件，诊断失败，修复在范围内的问题。
- **Review（审查）**：扫描仓库，检查文档、代码和配置之间的漂移。

"Improve → Hill Climb"循环以极少的监督递归地改进我的 agent。很难想象手动做这些事情。

顺便说一下，这个自动改进循环之所以可能，是因为环境是为此设计的。agent 代码、traces、日志、评测套件和正在运行的软件全部在同一个地方，这样编码 agent 就可以端到端地运行。

## 之所以能工作，是因为我们控制了技术栈

大多数软件无法自动改进，因为其输入和输出分散在各种工具中。要运行自动改进循环，编码 agent 必须将来自三个不同工具的数据拼凑在一起，每个工具都有自己的认证方式和做事方式。

理论上可行。实际上，摩擦太大。

我的代码库是专门为自动改进而设计的。例如，claude code 可以测试一个 agent，然后通过读取 sessions、traces 和日志来判断 PASS 或 FAIL。如果 agent 失败，它会编辑该 agent 并重新运行。

三件事使这成为可能：

**每个操作都作为 API 暴露。** 运行 agent、读取 session、运行评测。每个关键操作都可以用 cURL 或 bash 运行。

**数据是同位存储的。** Sessions 和 traces 存储在我们的 Postgres 数据库中。编码 agent 可以触发一次运行并读取其输出，而无需离开自己的环境。

**日志优先于一切。** 整个平台在本地的 Docker 上运行。编码 agent 读取实时日志并根据需要进行更新。测试 → 审查循环约为 5 秒。日志是解锁一切的实时反馈循环。

agent 平台是第一类软件，其中操作、数据和迭代工具彼此足够近，使得编码 agent 可以端到端测试、修改代码，然后再次测试，直到 agent 改进。这意味着托管循环的平台是循环首先改进的东西。

## Agent 开发生命周期

接下来我将展示 Claude Code 如何运行我的 agent 平台。

### 1. 创建一个 agent

要创建一个新 agent，我打开 Claude Code 并输入：

> Run create-new-agent.md in a new branch.

Claude 首先会问几个关于 agent 应该做什么、需要哪些工具的问题。然后它通过 MCP 搜索 Agno 文档以找到合适的工具包，生成 agent 文件，在 `app/main.py` 中注册它，重启容器，并通过 cURL 进行冒烟测试。从指令到 agent，5-10 分钟。

因为平台处理了所有事情，我开始构建以前根本不会费心去做的 agent。一个汇总隔夜 Slack 消息的 agent，一个起草我每周更新的 agent，一个在仓库中突出显示重要问题的 agent。这些都不会撑过一个多天的项目。它们全部适合在喝咖啡的时间里完成。

### 2. 改进一个 agent

要改进现有 agent，我输入：

> Run improve-agent.md on code-search agent.

Claude 读取 agent 的 INSTRUCTIONS 并从中派生出 8-12 个探针。有些是主路径探针，有些是边缘情况，有些是工具选择测试。还有几个对抗性的：提示注入、格式错误的输入、试图让 agent 偏离目标的尝试。

它通过 cURL 对实时容器运行每个探针。读取响应。从容器日志中读取工具调用。根据 INSTRUCTIONS 实际承诺的内容判断 PASS 或 FAIL。

对于每次失败，它选择一个调节手段：收紧规则、添加规则、替换工具、提高 `num_history_runs`。无论什么适合这个失败模式。它编辑 `agents/<slug>.py`，热重载，并只重新运行失败的探针。

然后它迭代。上限为五轮。如果一切通过则提前停止。

除了启动任务之外，不需要我的任何输入。这以前需要一整天手动点来点去，现在完全自动化了。

### 3. 扩展一个 agent

要为现有 agent 添加能力，我输入：

> Run extend-agent.md on code-search agent.

Extend 在我主导的情况下运行。我描述一个变更：添加工具、优化提示词、修复 bug。Claude 执行。Agno 文档 MCP 已加载，因此工具包研究基于真实的 API。

Claude 进行更改。运行冒烟测试。每次迭代都是一个小的、经过验证的步骤。变更保持精准，并在隔离状态下测试。

### 4. Hill Climb（爬坡）

随着时间推移，我们积累了很多 eval，如果手动修复失败将会很可惜。我只需输入：

> Run eval-and-improve.md.

Hill Climb 运行评测套件，诊断每次失败，并修复在范围内的问题。失败类型映射到修复位置：INSTRUCTIONS 中缺少的规则、幻觉、触发了错误的工具、过于具体的评分标准。对于每次失败，Claude 选择正确的调节手段，编辑，并只重新运行失败的用例。一旦一切变绿，它重新运行完整套件以捕获回归。

评测套件只有两个文件。`evals/cases.py` 声明用例。每个用例是一个输入加上一个评分标准（正确响应应该是什么样子）以及可选的预期工具调用。基于 Agno 的 `AgentAsJudgeEval` 和 `ReliabilityEval` 构建。

Improve 捕获分布外失败。Hill Climb 确保分布内用例继续通过。两者配合得非常好。

### 5. Review（审查）

因为仓库主要由编码 agent 管理，它移动很快。为了让一切保持同步，我输入：

> Run review-and-improve.md.

Claude 扫描整个仓库，检查文档、代码和配置之间的漂移。磁盘上的每个 agent 文件都应该在 `app/main.py` 中注册。代码读取的每个环境变量都应该在 `example.env` 和 `AGENTS.md` 中。Markdown 文档中的每个路径都应该仍然存在。每个脚本都应该做到它声称的事情。

机械性漂移会被原地自动修复：重命名的文件、`example.env` 中缺少的条目、架构图中缺少的新 agent。更大的问题会被标记，附带推荐的下一步。

最好在发布前或重构后运行。这类工作对人类来说很繁琐，但对能读取仓库中每个文件的编码 agent 来说则微不足道。

文档和代码之间的漂移一直是生产软件的税。现在它不再花费任何成本。

## 为什么选择 Agent 平台？

Agent 平台是这个模式的完美试验场。

**全新开始。** Agent 平台相对较新，可以从一开始就为编码 agent 而设计。

**工作流程清晰。** 我们知道如何改进一个 agent：运行它，读取日志，对响应评分，编辑，再次运行。

**循环确实有用。** 对于普通软件，优化一个 API 端点实际上没什么意义。对于 agent，每轮改进都是真实的、可衡量的，并且会增加价值。

正确设置平台，你就可以在上面构建任何 agent：使用 create 工作流从想法到 agent，使用 improve 工作流强化 agent，使用 extend 工作流添加新能力，用 eval 锁定它们，然后对它们进行 Hill Climb。

使用 review-and-improve 工作流保持整个仓库同步。

手动做这些几乎是不可能的。
