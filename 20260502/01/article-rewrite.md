# 做个小东西，就够了

**作者：** Zara Zhang ([@zarazhangrui](https://x.com/zarazhangrui))  
**日期：** 2026年5月2日  
**来源：** [How to build something small](https://x.com/Zephyr_hg/status/2050445806428438734)

大多数人教你怎么做大东西。我想聊聊怎么做小东西。

我自己做过几个小项目，全是业余时间玩出来的：

1. [Frontend Slides](https://github.com/zarazhangrui/frontend-slides)（GitHub 16k stars）：用 AI 编程代理生成漂亮 HTML 幻灯片的工具
2. [Follow Builders](https://github.com/zarazhangrui/follow-builders)（GitHub 3.6k stars）：每天从一份精选的 AI 构建者名单里，给你推送相关推文和播客
3. [Tab Out](https://github.com/zarazhangrui/tab-out)（GitHub 1.1k stars）：一个 Chrome 插件，把新标签页变成任务控制台，让你一眼看清所有打开的标签，随手就能关掉

# 为什么要做小东西？

## 1. 大东西都是从小东西长出来的

定位决定一切。

很多产品没能活下去，不是因为太专注，而是因为太贪——想覆盖所有人，结果对谁都不够有用。你变成一个没有特点的东西，用户不如直接用 Claude。

## 2. 为好玩而做，为学习而做

别把软件太当回事。

以前，写软件主要是为了赚钱，严肃且功利。软件稀缺，成本高，这完全合理。

但现在不一样了。软件便宜，而且到处都是。你完全可以纯粹为了好玩去做一个东西。Vibe coding 已经是我的周末爱好，有些小项目做出来更像艺术品。那种"做个没什么用的东西"——完全可以。

还有一件事：学东西，最好的方式就是去做。传统教育是先学再做；AI 时代反过来了，你是做着做着就学会了。

## 3. 做有立场的产品

软件是一种自我表达。你做一款东西，就是把自己的一小块思维方式嵌进去——你怎么看问题，你怎么做事。

现在 AI 能很快做出任何东西。那问题就来了：凭什么选你的，不选别人的？

答案是：你的产品得有性格。要让人看了之后有一种"哈，我以前没这么想过，但好像确实是这样"的感觉。他们用你的，是因为产品背后的那个观点，和他们想到一块儿去了。

一个通用的生产力工具，是没有这种吸引力的。用户不如直接用 Claude。

举个例子：做 AI 信息聚合工具的多了去了，但我的 [Follow Builders](https://github.com/zarazhangrui/follow-builders) 能打动人，原因只有一个——它背后那句话："关注构建者，不关注网红。"大家认同这个立场。

## 4. 现在你负担得起了

以前，做小东西是奢侈品。软件开发成本太高，你得组团队、说服别人、向委员会汇报，一圈走下来什么都没了。

现在是你和一个 AI 编程代理。代理不需要被说服，你有什么奇思妙想，它都会认认真真帮你做出来。

所以，去做那些在大厂产品评审会上会被直接毙掉的东西。

# 怎么找产品想法？

## 1. 你没那么独特

常有人跟我说："我想解决一个痛点，但怕太小众，没人用。"

我的回答可能不太好听：你没你以为的那么独特。

不管你的痛点多小众，全世界至少有 1 万个人有同样的困扰。不是 1000 万，但 1 万肯定有。

以前，1 万人的市场不值得做。现在，1 万用户的副业项目，已经算很成功了。

你以为极其小众的需求，也许没那么小众。你只需要找到那 1 万个和你一样"奇怪"的人。

## 2. 从技术出发

我的大多数项目，并不是先有痛点的，而是先有技术。

每天刷 X，我会把看到的有意思的新技术和 API 收藏起来。然后定期开一段"玩耍时间"，把这些新东西扔给 AI 编程代理，一起想能做什么。很好玩。

[Tab Out](https://github.com/zarazhangrui/tab-out) 就是这么来的。起点是我看到有人做了一个读 Chrome 浏览历史的东西，才知道原来浏览历史存在本地数据库里。于是我和 Claude 聊能用它做什么，反复讨论、反复迭代，最后做出了一个帮你关标签页的浏览器插件。

# 怎么和模型协作？

## 1. 用最好的模型

听起来反常识，但用好模型往往比用差模型更省钱。

差模型做不对，你得反复改，花更多 token 和时间兜圈子。算下来，用好模型其实更划算。

## 2. 把它当联合创始人，不是执行工具

我很少一上来就给详细需求，让代理去执行。我更常做的是，先把问题描述清楚，然后问代理的看法。我们互相碰想法，一起把方案想出来。

这是共创，不是执行。

**带来问题，不是解决方案。**

## 3. 尽管试，token 随便烧

标准可以高得不讲道理。一直推，看看代理能走多远。各种奇怪的想法都可以试，看它怎么应对。

跟人类协作，你不敢这么干。跟代理协作，这个自由要充分用好。

## 4. 发布前先做减法

LLM 很会加东西，但不太会减东西。它们常常加进来一堆功能，乍看合理，真正用起来却没什么用。

作为人，你最重要的工作就是做减法：简化，删掉，只留核心。

做小的，做精的。软件是讲故事的媒介。去讲一个好故事。
