# 如何做小东西

**作者：** Zara Zhang ([@zarazhangrui](https://x.com/zarazhangrui))  
**日期：** 2026年5月2日  
**来源：** [How to build something small](https://x.com/Zephyr_hg/status/2050445806428438734)

大多数人教你怎么做大东西。但我觉得，我们更应该学的是怎么做小东西。

我做过几个小东西，都是业余时间玩出来的：

1. [Frontend Slides](https://github.com/zarazhangrui/frontend-slides)（GitHub 16k stars）：让你用 AI 编程代理生成漂亮 HTML 幻灯片的工具
2. [Follow Builders](https://github.com/zarazhangrui/follow-builders)（GitHub 3.6k stars）：每天给你推送来自一份精选 AI 构建者名单的推文和播客
3. [Tab Out](https://github.com/zarazhangrui/tab-out)（GitHub 1.1k stars）：一个 Chrome 插件，把你的新标签页变成任务控制台，一眼看清所有标签，随手就能关掉

# 为什么要做小东西？

## 1. 大东西都是从小东西长出来的

定位决定一切。大多数产品死掉，不是因为太窄，而是因为想覆盖太多人。

想对所有人都有用，最后往往对谁都没用。你变成一个没有特点、平平无奇的东西，然后被 Claude 直接取代。

## 2. 为好玩而做，为学习而做

别把软件太当回事。

过去，写软件是件严肃的事，大家主要是为了赚钱。软件稀缺且昂贵，这没什么问题。

但现在软件便宜又随手可得，你完全可以纯粹为了好玩去做。Vibe coding 已经是我的周末爱好，有些小项目做出来更像艺术品。我们得改掉对软件的"稀缺心态"——做个没什么用的东西，完全没问题。

学东西，最好的方式就是去做。传统教育是先学再做，AI 时代反过来了：你做着做着就学会了。

## 3. 做有立场的产品

软件是一种自我表达。你做一款软件，就是把自己的一小块思维方式嵌进去——你看问题的角度，你做事的方法。

现在 AI 什么都能做，而且很快。那人们凭什么选你的软件，不选别人的？

你的产品得有性格。要让人看了有一种"哈，我以前没这么想过，但好像确实是这样"的感觉。他们用你的，是因为产品背后的观点和他们对上了。

如果你做的只是一个通用的生产力工具，用户不如直接用 Claude。

举个例子：做 AI 信息聚合工具的一大堆，但我的 [Follow Builders](https://github.com/zarazhangrui/follow-builders) 能打动人，是因为大家认同它背后的那句话："关注构建者，不关注网红。"

## 4. 你现在负担得起了

AI 之前，做小东西是奢侈品。软件开发成本太高，你得组团队、说服别人、向委员会汇报。

现在只有你和一个 AI 编程代理。代理不需要被说服，它会老老实实把你的任何奇思妙想做出来。所以，去做那些在任何大厂产品评审会上都会被直接毙掉的东西吧。

# 怎么找产品想法？

## 1. 你没那么独特

很多人来找我说："我有个很想解决的痛点，但担心太小众，没人用。"

我想直接告诉他们：这话可能不太好听，但你没你以为的那么独特。

不管你有多小众的痛点，全世界至少有 1 万个人和你一样。也许不是 1000 万，但 1 万肯定有。

以前，总可寻址市场只有 1 万的产品，不值得做。现在，1 万用户的副业项目，已经算很成功了。

你以为极其小众的需求，可能没那么小众。你只需要找到那 1 万个和你一样"奇怪"的人。

## 2. 从技术出发

我大多数项目，并不是先有痛点的。我是先有技术的。我每天刷 X，把觉得有意思的新技术和 API 收藏起来。然后定期开一段"玩耍时间"，把这些新东西扔给 AI 编程代理，一起头脑风暴能做什么。很好玩。

比如 [Tab Out](https://github.com/zarazhangrui/tab-out) 的起点，是我看到有人做了一个读取 Chrome 浏览历史的东西，才知道原来浏览历史是存在本地数据库里的。于是我和 Claude 聊能用它做什么，反复讨论迭代，最后做出了一个帮你关标签页的浏览器插件。

# 怎么和模型协作？

## 1. 用最好的模型

听起来反常识，但用好模型往往比用差模型更省钱。差模型做不对，你得反复修，花更多 token 和时间去兜圈子。

## 2. 把它当联合创始人，不是执行工具

我很少一上来就给详细的产品需求，让代理去执行。我更常做的是，把问题完整描述一遍，然后问代理的看法。我们互相碰想法，一起把方案想出来。这是共创，不是执行。

带来问题，不是解决方案。

## 3. 尽管试，token 随便烧

标准可以高得不讲道理。一直推，看看代理能走多远。试各种奇怪的想法，看它怎么应对。跟人类协作，你不敢这么干；跟代理协作，充分用好这个自由。

## 4. 发布前先做减法

LLM 很会加东西，但不太会减东西。它们常常加进来一堆功能，乍看合理，用起来没用。作为人，你最重要的工作就是做减法：简化，删掉，只留核心。

做小的，做精的。软件是讲故事的媒介。去讲一个好故事。
