# 充分利用 Codex

**作者：** jason ([@jxnlco](https://x.com/jxnlco))  
**日期：** 2026年5月21日  
**来源：** [Getting the most out of Codex](https://x.com/Zephyr_hg/status/2057153744630890620)

大多数开发者最初使用编码代理是为了写代码：检查一个仓库、做一个 diff、运行测试、开一个拉取请求。

那仍然是 Codex 的重心所在。但计算机上的很多工作早已由代码中介完成：执行 shell 命令、浏览网页、调用 API、导出文档、响应事件、触发自动化。随着这些界面对 Codex 可用，它开始感觉不那么像狭义上的编码助手，而更像一个完成计算机工作的系统。

Codex 应用让这种转变变得具体。一个线程可以保持上下文、使用工具、呈现产物，并跨提示词持续下去，而不是在每次交流后重置。

充分利用 Codex 意味着把这些能力一起使用：

- 保持上下文的持久线程
- 在用户仍在循环中时的语音、引导和排队
- 让 Codex 在仓库之外行动的浏览器、计算机使用、MCP 服务器和连接器
- 在用户离开时继续工作的线程自动化和目标
- 侧边面板，用户可以在那里审阅代码、文档、幻灯片和其他产物

## 持久线程

> **持久线程：** 跨重复会话保持工作上下文的长期运行的 Codex 线程。

固定线程是把持久线程放在手边的一种方式。它们对于循环出现的工作流很有用，例如：

- 一个幕僚长线程
- 一个发布线程
- 一个文档审阅线程
- 一个专门用于外部监控的线程

这些是持久的工作空间，不是简短的聊天。Codex 可以随时间反复回到它们，保留先前的决策、偏好和工作上下文，否则就需要从头重建。

固定线程的快捷键让这变得实用。Command-1 到 Command-9 直接跳进保存的线程。

## 语音输入

语音输入很有价值，因为它在一个想法被压缩成打磨过的散文之前，捕捉到了它粗糙的版本。

Codex 有内置的语音输入。它对于那些自然到可以说出口但打字很别扭的模糊起点尤其好用：

> 我想有个叫 Ben 的人在 Slack 里提过这个。
> 我不记得细节了。
> 请去查一下。

对于一个能搜索、收集上下文并回报的代理，那往往就足够了。

它也很适合在任务还没完全成形之前做两三分钟的想法倾倒。

转录稿也以同样的方式运作。一份原始的会议转录或口述的计划笔记往往比一份简短的摘要提供更好的源材料，因为它保留了不确定性、强调和未完成的思路。

## 引导和排队

当语音与对一个活动任务的明确控制配对时，它会变得更有用。

> **引导：** 在当前步骤完成之前，用新的方向打断一个进行中的 Codex 任务。

引导在代理朝错误方向前进、需要在完成前修正时很有用。例如在一次网站审阅中，用户可以在侧边面板上标注界面的同时打断工作：

- 把这个弄小一点
- 这两个元素之间的间距感觉不对
- 这段文案是错的

> **排队：** 添加 Codex 在当前步骤完成后要做的工作。

排队不一样。它不打断进行中的任务。它把下一个任务加到队列里。用户可能会说：

> 工作完成后，把预览链接发给 Slack 里的审阅者。

引导改变 Codex 现在正在做什么。排队改变接下来应该发生什么。两者都让用户在工作展开时贴近它。

## 工具和触及范围

一旦一个线程有了连续性，下一个问题就是它能在什么上面行动。Codex 可以分层向外移动：

- `$browser` 用于侧边面板里的应用内浏览器，Codex 可以在那里检查和标注网页界面
- `@chrome` 用于已登录的浏览器状态和基于 Chrome 的工作流
- `@computer` 用于只通过桌面 GUI 才存在的工作

`$browser` 适合侧边面板的浏览器审阅。`@chrome` 适合依赖用户 Chrome 上下文的已登录浏览器工作。`@computer` 适合只通过桌面 GUI 存在的任务。

MCP 服务器和连接器把同样的想法延伸到工作流的其余部分。Slack、Gmail 和 Calendar 之所以重要，是因为许多重要任务在成为代码之前，首先以消息、收件箱条目或日程问题的形式出现。

技能让重复的工作流可复用。一旦一个工作流被证明有用，就把它打包成一个技能，这样 Codex 就能再次运行它，而无需从头重新学习这套流程。

## 在任何地方工作

Codex 移动应用改变了用户必须坐在桌前的情况。一个任务可以在文件、权限和本地设置已经存在的 Mac 上开始，然后在用户从手机上查看时继续。

那在一些小时刻里很重要。有人可以在 Codex 运行一个较长任务时离开桌子，从外面回答一个问题、批准下一步，或者在回来之前重新引导线程。本地环境留在原地；用户不必如此。

## 自动化

自动化按计划运行 Codex 工作。当循环出现的工作应该从一个工作空间全新开始时，使用计划自动化，例如每日报告或定期仓库检查。当计划应该回到一个有运行上下文的活动对话时，使用线程自动化。

> **线程自动化：** 按计划回到同一个 Codex 线程的心跳式循环唤醒。

固定线程很有用，但它们仍然等待用户回来。一个线程自动化可以每隔几分钟或几小时检查某件事，持续到满足一个条件，并随时间调整节奏。

一个幕僚长线程可能每 30 分钟运行一次：

> 每 30 分钟，检查 Slack 和 Gmail 里需要我关注的未回复消息。
> 帮我排出最重要的优先级。
> 如果有人问我一个问题，尽你所能深入研究答案并替我起草一份回复，但不要发送。

当用户回来时，收集上下文这一昂贵的部分往往已经做完。人类仍然决定什么被发送出去。

线程自动化也适合反馈循环。一个线程自动化可以盯着拉取请求评论、Google Docs 评论或 Slack 回复，并在用户离开时让周围的工作继续推进。

设想一个动画工作流，一位审阅者在 Slack 里分享了一个视频。一个线程自动化可以按计划检查这个线程，在评论到来时渲染一个更新的版本，并在同一个线程里回复并标记审阅者。如果某个集成无法完成最后的上传，桌面自动化可以通过 GUI 完成这一步。

这个循环横跨用于反馈的 Slack、用于渲染的代码库，以及用于最终上传的桌面自动化。

## 目标

当任务有一条代理可以持续推进的真正的终点线时，目标最为强大。一个弱的目标是：

> **目标：** 有一条终点线、代理可以随时间持续努力达成的较长期运行的 Codex 任务。

> 实现这个 Markdown 文件里的计划。

一个更强的目标有一个可衡量的成功标准。

例如，一位工程师可能通过建立新目录、定义目标并把终点线说清楚，来把一个内部工具从 Python 迁移到 Rust：新实现在单元测试通过之前不算完成。

一个目标把持续执行和一个验证器结合起来。用户定义结果、停止条件，以及说明 Codex 是否在接近的信号。

有用的验证器包括：

- 一个测试套件
- 一个基准测试
- 一个 bug 复现
- 一个验证矩阵
- 一个必须持续通过的端到端工作流

雄心很重要，但没有验证，它只是一个愿望。

## 侧边面板

侧边面板把工作放在产生它的对话旁边。用户不必导出一个产物再切换上下文，而是可以就地审阅它。输出可能是代码，但也可能是一份幻灯片、一个 PDF、一个浏览器页面、一张表格，或者一路上创建的另一个产物。

它尤其擅长支持四项工作：

1. 检查产物
2. 标注需要改的地方
3. 操作网页界面
4. 审阅更改

侧边面板让用户就地审阅 Markdown、电子表格、数据表、文档和幻灯片。他们可以检查、标记和修订产物，而不打断循环。

![](images/img-01.jpg)

**标注**

幻灯片或 PDF 可以在产生它的线程旁边保持打开，随时可供直接审阅和修复。

![](images/img-02.jpg)

**Codex 里的表格**

应用内浏览器让 Codex 检查一个渲染好的页面、控制它，并直接在被审阅的界面上响应标注。对一个页面或产物的评论留在工作循环内部，而不是变成一次单独的交接。

网页同时成为输出和控制界面。Codex 可以构建一个产物，在侧边面板里打开它、检查它、调试它，并就地不断打磨同一个对象。

![](images/img-03.jpg)

这些界面尤其好用：

- index.html 用于轻量级静态产物
- Storybook 用于 UI 审阅
- Remotion Studio 用于程序化动画
- 基于浏览器的幻灯片用于演示
- 数据应用用于分析工作流

一个单独的 index.html 文件可以成为一个持久的交互式产物，无需服务器。线程自动化也可以随时间刷新静态产物，这样当用户回来时线程就有新东西等着。

## 共享记忆

长期运行的线程在它们共享任何单个对话之外的记忆时会变得更有用。

> **共享记忆：** 存储在单个线程之外的持久上下文，这样未来的工作可以从某个明确且可审阅的东西恢复。

一个持久的模式是把持久线程锚定在一个 Obsidian 库里。在实践中，那意味着一个由纯文件组成的文件夹，便于检查、编辑、移动并长期保存。团队可以把那个文件夹存在云存储、Git、Dropbox、Google Drive，或者适合他们工作流的另一个同步层里。

一个库可能长这样：

```
vault/
├── TODO.md
├── people/
├── projects/
├── agent/
└── notes/
```

在顶层，AGENTS.md 可以定义 Codex 在更多地了解人、项目、决策和未决事项时应该如何更新那个工作空间。

不要照搬某个确切的库结构。教会代理持久上下文应该住在哪里、要保留什么上下文，以及什么时候不要制造无谓的变动。

一份实用的 AGENTS.md 可能会说：

```
- 把 ~/vault 当作持久的工作记忆。
- 偏好规范的笔记，而非笔记泛滥。
- 明确地分流 TODO、人、项目、每日摘要和草稿笔记。
- 保留决策、阻碍、负责人、日期和有用的链接。
- 如果没有任何有意义的变化，不要折腾这个库。
```

仓库装代码。库装滚动的上下文：涉及的人、什么变了、什么被阻塞、什么需要跟进，以及什么本来会在会话之间消失。

重要的上下文不应该只活在一份对话转录里。把它写下来，放在下一个线程能接着用的地方。

Codex 在设置 > 个性化 > 记忆里也有第一方的记忆功能。它们为偏好、循环出现的工作流和已知陷阱提供一个本地回忆层。它们补充而不是取代明确写下的上下文。Chronicle 朝同一个方向推进，帮助 Codex 从近期屏幕上下文构建记忆。

## 从代码向外

Codex 仍然从代码开始。但代码周围越来越多的工作如今可以通过同一个系统触及：MCP 服务器、浏览器界面、桌面控制、线程自动化和可审阅的产物。

那改变了控制模型。引导打断进行中的工作。排队排上下一个任务。线程自动化在用户走开时让一个线程保持活动。目标加上一条 Codex 可以持续努力达成的具体终点线。

Codex 如今可以把一个工作流从指令带到执行再带到产物审阅，即使工作离开了仓库。
