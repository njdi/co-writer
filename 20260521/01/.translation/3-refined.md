# 充分利用 Codex

**作者：** jason ([@jxnlco](https://x.com/jxnlco))  
**日期：** 2026年5月21日  
**来源：** [Getting the most out of Codex](https://x.com/Zephyr_hg/status/2057153744630890620)

大多数开发者第一次用编码智能体，都是拿它来写代码：看看一个仓库、改出一份 diff、跑一遍测试、提一个 PR。

这至今仍是 Codex 的重心。但计算机上的很多活儿，本来就要靠代码来完成：执行 shell 命令、浏览网页、调用 API、导出文档、响应事件、触发各种自动化。当这些场景一个个向 Codex 开放，它就越来越不像狭义上的那种编码助手，而更像一个帮你把电脑上的活儿都干完的系统。

Codex 应用把这种转变落到了实处。一个线程能保住上下文、调用工具、把成果摆出来，还能跨多轮提示词一直接着干，而不是每聊完一轮就清零重来。

想充分用好 Codex，关键是把下面这些能力一起用上：

- 能保住上下文的持久线程
- 人还在场时的语音、引导和排队
- 让 Codex 走出仓库去干活的浏览器、电脑操作、MCP 服务器和各种连接器
- 在你离开时接着把活儿往下推的线程自动化和目标
- 侧边面板——你可以在这里审阅代码、文档、幻灯片和其他成果

## 持久线程

> **持久线程：** 能跨多次会话保住工作上下文、长期运行的 Codex 线程。

把持久线程随手放在手边，一个办法是固定它们。固定线程很适合那些会反复出现的工作，比如：

- 一个当幕僚长用的线程
- 一个管发布的线程
- 一个做文档审阅的线程
- 一个专门盯外部动态的线程

这些是长期的工作空间，不是聊两句就散的对话。Codex 可以隔段时间就回来接着用，先前的决策、偏好和工作上下文都还在，不必每次从头重建。

固定线程的快捷键让这一切更顺手。Command-1 到 Command-9，一按就直接跳进对应的线程。

## 语音输入

语音输入的价值在于：它能在一个想法还没被打磨成成形文字之前，就把它最粗糙的样子先抓下来。

Codex 内置了语音输入。对那些说出来很自然、打字却很别扭的模糊念头，它尤其好用：

> 我记得好像有个叫 Ben 的人在 Slack 里提过这事。
> 具体细节我忘了。
> 你去查查。

对一个能搜索、能收集上下文、还能回头汇报的智能体来说，这点信息往往就够了。

它也很适合在任务还没成形时，花两三分钟把想法一股脑倒出来。

转录稿也是同样的道理。一份原始的会议记录或口述的计划草稿，往往比一段简短摘要更好用，因为它把那些不确定、强调的语气和没说完的思路都保留了下来。

## 引导和排队

当语音再配上对当前任务的直接掌控，它会更有用。

> **引导：** 在当前这一步还没做完时，给一个进行中的 Codex 任务塞进新方向。

当智能体走偏了、得在做完之前赶紧纠偏，引导就派上用场。比如审阅一个网站时，你可以一边在侧边面板上圈圈点点，一边直接打断它：

- 这个再做小一点
- 这两个元素之间的间距看着不太对
- 这段文案写错了

> **排队：** 给 Codex 安排好当前这一步做完之后再接着做的活儿。

排队是另一回事。它不打断手头正在做的任务，而是把下一件事排进队列。你可能会说：

> 这事做完之后，把预览链接发给 Slack 里的审阅者。

引导改的是 Codex 此刻在做什么，排队改的是接下来该做什么。两者都让你在工作展开的过程中一直跟得很近。

## 工具和能触及的范围

线程有了连续性之后，下一个问题就是：它能在什么东西上动手。Codex 可以一层层往外延伸：

- `$browser` 对应侧边面板里的应用内浏览器，Codex 可以在这里查看和标注网页
- `@chrome` 对应已登录的浏览器状态和基于 Chrome 的工作流
- `@computer` 对应那些只能通过桌面图形界面才能做的活儿

`$browser` 适合在侧边面板里审阅网页，`@chrome` 适合那种依赖你 Chrome 登录状态的浏览器工作，`@computer` 则适合只能靠桌面图形界面完成的任务。

MCP 服务器和连接器把同样的思路延伸到工作流的其他环节。Slack、Gmail、日历之所以重要，是因为很多要紧的任务在变成代码之前，最先是以一条消息、一封邮件或一个日程问题的形式冒出来的。

技能让重复的工作流可以反复用。一个工作流一旦被证明好用，就把它打包成技能，这样 Codex 下次直接就能跑，不用从头再学一遍这套流程。

## 随时随地工作

有了 Codex 移动应用，你不必非守在桌前不可。一个任务可以在 Mac 上开个头——文件、权限、本地环境都现成——然后等你拿起手机时接着往下走。

这在一些小时刻里很关键。Codex 跑一个耗时较长的任务时，你大可离开座位，在外面回个问题、批准下一步，或者在回来之前先把线程的方向掰一掰。本地环境一直在原地待着，你不用守着。

## 自动化

自动化会按设定的时间自动跑 Codex 的活儿。如果一件反复要做的事每次都该从一个工作空间从头开始——比如每日报告或定期的仓库检查——就用计划自动化；如果它该回到一段还带着上下文的对话里接着干，就用线程自动化。

> **线程自动化：** 像心跳一样按节奏定时唤醒、每次都回到同一个 Codex 线程的机制。

固定线程很好用，但它们还是得等你回来。线程自动化则可以每隔几分钟或几小时就主动查一下某件事，一直查到满足某个条件为止，还能随着情况调整节奏。

一个当幕僚长用的线程，可能每 30 分钟跑一次：

> 每 30 分钟，查一下 Slack 和 Gmail 里需要我处理、还没回的消息。
> 帮我理出哪些最要紧。
> 如果有人问我问题，尽量帮我把答案研究透，替我起草一份回复，但先别发。

等你回来时，收集上下文这件最费工夫的事往往已经办妥了。该发什么，仍然由人来定。

线程自动化也很适合反馈循环。它可以盯着 PR 评论、Google Docs 评论或 Slack 回复，在你离开的时候让相关工作一直往前推。

设想一个动画工作流：一位审阅者在 Slack 里发了段视频。线程自动化可以按节奏来查这个对话，等评论来了就渲染出一版新的，再在同一个对话里回复并 @ 上审阅者。要是某个集成搞不定最后的上传，桌面自动化可以通过图形界面把这一步收尾。

这个循环串起了三头：负责反馈的 Slack、负责渲染的代码库，还有负责最终上传的桌面自动化。

## 目标

当一个任务有一条实打实、智能体能一直朝它使劲的终点线时，目标最能见威力。下面这种就是弱目标：

> **目标：** 有明确终点线、智能体能长期朝它持续努力的 Codex 任务。

> 把这个 Markdown 文件里的计划实现出来。

更强的目标，会带一个能衡量的成功标准。

举个例子，一位工程师想把一个内部工具从 Python 迁到 Rust，他可以先建好新目录、定下目标，再把终点线说死：新版本要等单元测试全过了，才算做完。

一个目标，是把"持续执行"和"一个验证器"绑在一起。你来定结果、定停在哪里，再定一个能说明 Codex 是不是在靠近的信号。

好用的验证器包括：

- 一套测试
- 一个基准测试
- 一个 bug 复现
- 一张验证矩阵
- 一条必须一直跑得通的端到端流程

有雄心是好事，可光有雄心、不去验证，那就只是空想。

## 侧边面板

侧边面板把成果摆在产生它的那段对话旁边。你不用先导出再切来切去，就地就能审阅。产出可能是代码，也可能是一份幻灯片、一个 PDF、一个网页、一张表格，或者一路上顺手做出来的别的东西。

它尤其擅长四件事：

1. 查看成果
2. 标出要改的地方
3. 操作网页
4. 审阅改动

侧边面板让你就地审阅 Markdown、电子表格、数据表、文档和幻灯片。你可以查看、标注、修改，全程不打断手头的活儿。

**标注**

幻灯片或 PDF 可以一直开在产生它的线程旁边，随时直接审阅、随时修。

![](images/img-02.jpg)

**Codex 里的表格**

应用内浏览器让 Codex 查看一个渲染好的页面、操控它，还能直接在你正在审的界面上回应标注。对页面或成果的评论就留在工作流里，不用再单独交接一趟。

于是网页既是输出，也是操控的界面。Codex 可以做出一个成果，在侧边面板里打开、查看、调试，就地把同一个东西一遍遍打磨好。

![](images/img-03.jpg)

下面这些界面尤其好用：

- index.html，适合轻量的静态成果
- Storybook，适合审阅 UI
- Remotion Studio，适合用程序做动画
- 基于浏览器的幻灯片，适合做演示
- 数据应用，适合分析类的工作流

一个 index.html 文件，就能变成一个长期可用的交互式成果，连服务器都不用。线程自动化还能时不时刷新这些静态成果，这样你回来时，线程里就有新东西等着了。

## 共享记忆

长期运行的线程，如果能把记忆放到单个对话之外共享出来，会更有用。

> **共享记忆：** 存在单个线程之外的持久上下文，让以后的工作能从一个明确、看得见的地方接着往下做。

一个经得起时间考验的做法，是把持久线程锚定在一个 Obsidian 库里。说白了，就是一个装着纯文本文件的文件夹，方便查看、编辑、搬动，也方便长期留着。团队可以把这个文件夹放进云存储、Git、Dropbox、Google Drive，或者任何合自己工作流的同步方式里。

一个库大概长这样：

```
vault/
├── TODO.md
├── people/
├── projects/
├── agent/
└── notes/
```

在最顶层，AGENTS.md 可以规定：随着 Codex 越来越了解相关的人、项目、决策和没了结的事，它该怎么更新这个工作空间。

别照抄某一套现成的库结构。要做的是教会智能体：持久上下文该放在哪、哪些上下文要留下来，以及什么时候别瞎折腾。

一份实用的 AGENTS.md 可能会这么写：

```
- 把 ~/vault 当作长期的工作记忆。
- 宁可几篇规范的笔记，也别让笔记到处泛滥。
- 把 TODO、人、项目、每日小结、临时笔记，明确地各归各位。
- 保留好决策、阻碍、负责人、日期和有用的链接。
- 要是没什么实质变化，就别去折腾这个库。
```

仓库装代码，库装不断累积的上下文：都有谁参与、什么变了、什么卡住了、什么还得跟进，以及那些本来会在一次次会话之间悄悄消失的东西。

要紧的上下文，不该只活在一份对话记录里。把它写下来，放在下一个线程能接着用的地方。

Codex 在"设置 > 个性化 > 记忆"里也有自带的记忆功能。它给偏好、常用工作流和已知的坑提供了一个本地的回忆层。它是对你写下来的上下文的补充，而不是替代。Chronicle 也朝同一个方向使劲，帮 Codex 从最近的屏幕内容里攒出记忆。

## 从代码往外走

Codex 仍然从代码起步。但围着代码的那些活儿，如今越来越多都能通过同一个系统够得着了：MCP 服务器、浏览器界面、桌面操控、线程自动化，还有可以审阅的成果。

这改变了掌控的方式。引导，打断手头进行中的工作；排队，排上下一个任务；线程自动化，在你走开时让线程一直活着；目标，给 Codex 立一条它能一直朝它使劲的具体终点线。

如今 Codex 可以把一个工作流，从下指令、到执行、再到审阅成果，一路带下去——哪怕这活儿早已走出了仓库。
