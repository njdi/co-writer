# 上手 Codex：让结果更好的几条成熟做法

刚开始用 Codex，或者第一次接触编程智能体，多少都会有点摸不着头脑：同样一个工具，有人用得顺手，有人却老是拿到一堆不太对劲的结果。差别往往不在工具本身，而在那些不起眼的习惯上。这篇就来讲讲这些习惯，从写提示、做规划，一路讲到验证、MCP、技能和自动化。这些做法在 [CLI](https://developers.openai.com/codex/cli)、[IDE 扩展](https://developers.openai.com/codex/ide) 和 [Codex 应用](https://developers.openai.com/codex/app) 里都通用。

说到底，用 Codex 最舒服的心态，是别把它当成一个用完即走的助手，而是当成一个队友——你会一点点摸清它的脾气，把它调教得越来越合手。

整件事大致是这么个顺序：先给它对的任务上下文，再用 `AGENTS.md` 把那些长期管用的指导写下来，把它配置成贴合你的工作流，用 MCP 接上外部系统，把反复要做的事变成技能，最后把已经跑顺的流程交给自动化。下面一条条来看。

## 第一次就用好：上下文和提示

先说个让人安心的事：哪怕你的提示写得不怎么样，Codex 也已经强到够用了。很多时候你几乎不用做准备，把一个难题直接丢过去，照样能得到不错的结果。

这就有意思了——既然提示不完美也行，那把提示写清楚还有什么意义？意义在于「稳」。清晰的[提示](https://developers.openai.com/codex/prompting)不是拿到价值的前提，但它能让结果更可靠，尤其是在那种代码库很大、或者出错代价很高的任务里。

如果你正对着一个又大又复杂的仓库，最能帮上你的，就是把任务的上下文给对，再把你想做的事讲出个结构来。一个好用的默认套路，是在提示里写清楚四件事：

- **目标：** 你想改什么、想做什么？
- **上下文：** 这个任务牵扯到哪些文件、文件夹、文档、示例或报错？你可以用 @ 把某些文件直接提进来。
- **约束：** 该遵守哪些标准、架构、安全要求或惯例？
- **做到什么算完：** 任务结束时，什么该成立？比如测试通过了、行为变了，或者那个 bug 不再出现了。

把这四样讲明白，Codex 就不容易跑偏，也少做些没必要的猜测，最后产出的东西你审查起来也省心。

还有一个开关值得留意：推理级别。任务有多难，就给多高的级别，再多试几次，看哪种最对你的胃口——不同的人、不同的活，合适的设置都不一样。

- 低：用于更快、范围清楚的任务
- 中或高：用于更复杂的改动或调试
- 超高：用于那种又长、又得反复推理的重活

最后一个小窍门：想更快地把上下文交出去，可以在 Codex 应用里用语音听写，把要做的事说出来，比一个字一个字敲快多了。

## 难任务，先做规划

不是每个任务都该上来就写代码。如果一件事复杂、含糊，或者连你自己都讲不太清楚，那就先让 Codex 把计划想好，再动手。

有几种办法都挺管用。

**最省事的是规划模式。** 对大多数人来说，这就够用了，效果也好。它会让 Codex 先去收集上下文、问你几个澄清的问题，在真正动手前先拿出一个像样的计划。用 `/plan` 或 <kbd>Shift</kbd>+<kbd>Tab</kbd> 就能切换。

**还有一招，是让 Codex 反过来问你。** 有时候你心里大概知道想要什么，可就是说不清。这时候别硬憋，让它来问你。告诉它别客气，尽管质疑你的假设，把那个模糊的念头一点点逼成具体的东西，然后再开始写。

**如果你的活更复杂，可以用 PLANS.md 模板。** 对那些跑得久、步骤多的工作，你可以让 Codex 照着一个 `PLANS.md` 或执行计划模板来做。细节可以看[执行计划指南](https://developers.openai.com/cookbook/articles/codex_exec_plans)。

## 用 `AGENTS.md` 把指导沉淀下来

一个提示套路一旦试出来好用，下一件事就是别再每次手动重打一遍了。这正是 [AGENTS.md](https://developers.openai.com/codex/guides/agents-md) 该上场的地方。

你可以把它想成一份写给智能体看的 README，格式是开放的。它会自动加载进上下文，所以是你和团队写下「希望 Codex 在这个仓库里怎么干活」的最好地方。

一份好的 `AGENTS.md`，大致会写到这些：

- 仓库的结构，还有哪些目录重要
- 这个项目怎么跑起来
- 构建、测试和 lint 命令
- 工程上的惯例，以及对 PR 的要求
- 有哪些约束，哪些事不要做
- 「做完」到底指什么，又该怎么验证

不知道从哪写起也没关系。CLI 里有个 `/init` 斜杠命令，能在当前目录帮你快速搭出一份起步用的 `AGENTS.md`。这是个好起点，但别原样照搬，改一改，让它对得上你们团队实际怎么构建、测试、审查和发布代码。

`AGENTS.md` 还能分层来写：放在 `~/.codex` 里的全局文件，管你的个人默认；放在仓库里的文件，管团队共享的标准；放在子目录里的文件，管更局部的规则。规则是这样的——哪份文件离你当前目录更近、更具体，就听哪份的。

有一点要记住：写得实用，比写得全更重要。一份简短、准确的 `AGENTS.md`，远胜过一份满是空话的长文件。先从基础写起，等你发现某个错误老是重复出现，再去添新规则。万一它越长越大，就让主文件保持简洁，把规划、代码审查、架构这些内容拆到各自的 markdown 文件里，再从主文件引过去。

## 把 Codex 配置好，它才稳定

想让 Codex 在不同会话、不同界面之间表现得一个样，配置是主要的抓手。模型选哪个、推理用多大力、沙盒模式、审批策略、配置档案、MCP 设置——这些都可以先定好默认值。

一个不错的起步分法是这样：

- 个人默认值，放在 `~/.codex/config.toml`（路径是设置 → 配置 → 从 Codex 应用打开 config.toml）
- 某个仓库专属的行为，放在 `.codex/config.toml`
- 命令行覆盖，只留给一次性的场合（如果你用 CLI 的话）

[`config.toml`](https://developers.openai.com/codex/config-basic) 就是你写下长期偏好的地方，像 MCP 服务器、配置档案、多智能体设置、功能开关都在这儿。你可以自己改，也可以让 Codex 替你改。

这里还有两个开关，关系到安全，值得专门说一下。Codex 自带操作系统级别的沙盒：审批模式管的是它什么时候得先问过你、才能跑命令；沙盒模式管的是它能不能在目录里读写、又能碰哪些文件。

如果你是新手，建议先用默认权限，把这两个开关都收紧。等你确实有需要了，再针对那些信得过的仓库、或者特定的工作流，一点点放松。顺带提一句，CLI、IDE 和 Codex 应用其实共用同一套配置层，想再深入了解，可以翻翻[示例配置](https://developers.openai.com/codex/config-sample)页面。

## 靠测试和审查，把可靠性提上来

很多人让 Codex 改完代码就收工了，其实这一步往往最可惜。更好的做法是让它接着往下走：该写测试就写测试，把相关检查跑一遍，确认结果对了，再在你点头之前自己先审一遍。

这一整套循环，Codex 都能替你做——前提是它知道「好」长什么样。这份标准，要么写在提示里，要么写在 `AGENTS.md` 里。它大概包括这些：

- 为这次改动写或更新测试
- 把对应的测试套件跑起来
- 检查 lint、格式和类型
- 确认最终行为跟你要的对得上
- 审一遍 diff，把 bug、回归和有风险的写法挑出来

具体怎么审也有几种趁手的方式。在 Codex 应用里把 diff 面板打开，就能直接在本地[审查改动](https://developers.openai.com/codex/app/review)；点某一行还能留下反馈，这条反馈会作为上下文带进下一轮。CLI 里则有个 `/review` 命令，能让你：

- 对着一个基础分支审，做 PR 那样的审查
- 审还没提交的改动
- 审某一个提交
- 用你自己写的审查指令来审

如果你和团队有一份 `code_review.md`，再在 `AGENTS.md` 里把它引过来，那 Codex 审查时也会照着这份走。对那些想让审查标准在各个仓库、各个人之间都保持一致的团队来说，这一招很顶用。

说到底，Codex 不该只会生成代码。指令给对了，它一样能帮你**测试、检查和审查**这些代码。要是你用 GitHub Cloud，还能让它直接给你的 PR 跑[代码审查](https://developers.openai.com/codex/integrations/github)——在 OpenAI，100% 的 PR 都是 Codex 审的。你可以开自动审查，也可以在 @Codex 的时候再让它来。

## 用 MCP 接上外部上下文

有时候 Codex 需要的信息根本不在仓库里，而你又不想一遍遍把它复制粘贴进提示。这种时候，MCP 就派上用场了——它能让 Codex 直接接上你本来就在用的工具和系统。

说白了，[模型上下文协议](https://developers.openai.com/codex/mcp)（也就是 MCP）就是一个把 Codex 接到外部工具和系统的开放标准。下面这些情况，特别适合用它：

- 需要的上下文不在仓库里
- 数据经常在变
- 你想让 Codex 直接去调一个工具，而不是靠你粘贴指令
- 你需要一套能在多个用户或项目之间反复用的集成

技术上，Codex 同时支持 STDIO 和带 OAuth 的 Streamable HTTP 服务器。在 Codex 应用里，去设置 → MCP 服务器，就能看到自定义的和推荐的服务器。很多时候你都不用自己折腾，开口问一句，Codex 就能帮你把要用的服务器装上。当然，你也可以在 CLI 里用 `codex mcp add`，填上名称、URL 和其他细节，自己加。

不过有个分寸要把握：只在一个工具能真正打通一条工作流时，才把它加进来。别一上来就把你用过的每个工具都接上。先挑一两个——挑那种能明显省掉你经常要手动跑一遍的工具——再慢慢往外扩。

## 把反复要做的事变成技能

一条工作流一旦变得可重复，就别再靠长提示或没完没了的来回了。这时候该请出[技能](https://developers.openai.com/codex/skills)：把指令、上下文，还有 Codex 该一致照做的那些逻辑，一起打包进一个 SKILL.md 文件。技能在 CLI、IDE 扩展和 Codex 应用里都能用。

写技能有几条要点。第一，让每个技能只盯着一件事。先从 2 到 3 个具体用例做起，把输入和输出定清楚，再把描述写明白——这个技能是做什么的、什么时候该用它，连用户真正会说出口的那种触发用语也写进去。

第二，别一开始就想着把每种边角情况都覆盖到。先挑一个有代表性的任务，把它打磨好，再把这条流程变成技能，然后慢慢改。脚本或额外的资源，也是同理，只在能让它更可靠时才加。

那到底什么时候该做技能呢？有句话挺好记：如果你总在重复用同一个提示，或者总在纠正同一条流程，那它八成就该做成技能了。这类反复出现的活尤其合适，比如：

- 日志分类
- 起草发布说明
- 对着清单审 PR
- 迁移规划
- 遥测或事故总结
- 标准的调试流程

真要动手，从 `$skill-creator` 这个技能入手最合适，它能帮你搭出第一个版本。迭代的时候先把它留在本地；等它成熟到可以拿出去广泛分享了，再打包成一个[插件](https://developers.openai.com/codex/plugins/build)。再强调一遍：技能里最要紧的部分之一就是描述，它得说清楚这个技能做什么、什么时候用。

放在哪也有讲究——个人技能存在 `$HOME/.agents/skills` 里；团队共享的技能可以签入仓库里的 `.agents/skills`，这对新队友入职特别友好。

## 把重复的活交给自动化

一条工作流一旦稳定下来，下一步就能更省心了：让 Codex 在后台替你跑。在 Codex 应用里，[自动化](https://developers.openai.com/codex/app/automations)能让你给一个反复出现的任务挑好项目、提示、节奏和执行环境。

具体操作是在 Codex 应用的「自动化」标签里建一个。你可以选它在哪个项目里跑、跑什么提示（提示里还能调用技能）、按什么节奏跑，甚至能选它是在一个专门的 git 工作树里跑，还是在你本地环境里跑。想了解工作树是怎么回事，可以看 [git 工作树](https://developers.openai.com/codex/app/worktrees)。

适合交出去的活，比如：

- 总结最近的提交
- 扫一遍可能的 bug
- 起草发布说明
- 检查 CI 失败
- 产出站会总结
- 按计划跑一些可重复的分析流程

这里有一句话能帮你分清技能和自动化：技能定义「怎么做」，自动化定义「什么时候做」。所以顺序很重要——如果一条流程还得你不停在旁边引导，那就先把它做成技能；等它变得可预测了，再交给自动化，这时候才能真正让你事半功倍。还有一点容易被忽略：自动化别只用来执行，也可以用来反思和维护，比如回看最近的会话，把反复遇到的麻烦总结出来，再慢慢去改进提示、指令和整套设置。

## 用会话控制，管好跑得久的活

最后说说会话。Codex 的会话可不只是聊天记录，它们是一条条工作线程，会随着时间不断攒下上下文、决策和动作。正因为这样，管好它们，对结果质量影响很大。

论方便，还是 Codex 应用的界面管线程最省事——你能把线程固定住，也能创建工作树。如果你用 CLI，下面这些[斜杠命令](https://developers.openai.com/codex/cli/slash-commands)会特别趁手：

- `/experimental`：开关实验性功能，并写进你的 `config.toml`
- `/resume`：接着一段保存过的对话往下聊
- `/fork`：开一条新线程，同时把原来的记录留着
- `/compact`：线程变长、你想要一份前面内容的摘要版时用。顺带一提，Codex 本来也会自动帮你压缩对话
- `/agent`：在跑并行智能体、想在几条活动线程间切换时用
- `/theme`：挑一个语法高亮主题
- `/apps`：在 Codex 里直接用 ChatGPT 应用
- `/status`：看看当前会话是个什么状态

管线程还有个朴素的原则：每一块完整的活，配一条线程。如果几件事还属于同一个问题，那留在同一条线程里通常更好，因为它把推理的来龙去脉都给你留着。只有当工作真的分了岔，才去 fork。另外，那些有明确边界的活，可以用 Codex 的[子智能体](https://developers.openai.com/codex/concepts/subagents)工作流卸下来——让主智能体专心啃核心问题，把探索、测试、分类这类杂活交给子智能体。

## 常见的坑

第一次用 Codex，有几个坑很多人都踩过，先给你列出来，绕开就好：

- 把长期规则一股脑塞进提示，而不是挪进 `AGENTS.md` 或某个技能
- 不让智能体看到自己干的活——没交代清楚构建和测试命令到底该怎么跑
- 多步骤的复杂任务，跳过了规划
- 还没摸清工作流，就把电脑的完全权限交了出去
- 在同一批文件上跑实时线程，却不用 git 工作树
- 一个反复出现的任务，手动都还没跑稳，就急着做成自动化
- 把 Codex 当成一个要你一步步盯着的东西，而不是让它跟你的活并行推进
- 一个项目用一条线程，而不是一个任务一条线程——这么干，时间一长，上下文越来越臃肿，结果也越来越差
