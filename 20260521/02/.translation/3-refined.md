# 上手 Codex：让结果更好的几条成熟做法

如果你刚开始用 Codex，或者刚接触编程智能体，这份指南能帮你更快地拿到更好的结果。它讲的是那些让 Codex 真正好用的核心习惯，覆盖 [CLI](https://developers.openai.com/codex/cli)、[IDE 扩展](https://developers.openai.com/codex/ide) 和 [Codex 应用](https://developers.openai.com/codex/app)，从写提示、做规划，一直到验证、MCP、技能和自动化。

用 Codex 的最佳方式，是别把它当成一次性的助手，而是当成一个队友——你会随着时间慢慢调教它、让它越来越合手。

可以这样理解整个过程：先给它正确的任务上下文，再用 `AGENTS.md` 写下长期有效的指导，把 Codex 配置成贴合你的工作流，用 MCP 接上外部系统，把反复要做的事变成技能，最后把已经稳定的流程交给自动化。

## 第一次就用好：上下文和提示

哪怕你的提示并不完美，Codex 也已经强到够用了。很多时候，你几乎不用做什么准备，把一个难题直接丢给它，照样能得到不错的结果。清晰的[提示](https://developers.openai.com/codex/prompting)不是拿到价值的前提，但它确实能让结果更稳——尤其是在大型代码库，或者那些容错率低的任务里。

如果你面对的是一个又大又复杂的仓库，最能帮上你的，就是给 Codex 正确的任务上下文，再把你想做的事讲清楚结构。

一个不错的默认做法，是在提示里写清楚四件事：

- **目标：** 你想改什么、想做什么？
- **上下文：** 这个任务涉及哪些文件、文件夹、文档、示例或报错？你可以用 @ 把某些文件提进来当上下文。
- **约束：** Codex 该遵守哪些标准、架构、安全要求或惯例？
- **完成的标准：** 任务做完时，什么该成立？比如测试通过了、行为变了，或者某个 bug 不再复现了。

这样 Codex 不容易跑偏，少做无谓的猜测，产出的东西也更好审查。

任务有多难，就选多高的推理级别，并多试试哪种最适合你的工作流。不同的人、不同的任务，合适的设置都不一样。

- 低：用于更快、范围清楚的任务
- 中或高：用于更复杂的改动或调试
- 超高：用于那种又长、又需要智能体反复推理的重活

想更快地把上下文交给它，可以试试在 Codex 应用里用语音听写，把要做的事说出来，而不是一个字一个字地敲。

## 难任务，先做规划

如果任务复杂、含糊，或者你自己也很难讲清楚，那就让 Codex 先规划，再动手写代码。

有几种办法都挺管用：

**用规划模式：** 对大多数人来说，这是最简单也最有效的选择。规划模式会让 Codex 先收集上下文、问几个澄清问题，在动手实现前先拿出一个更靠谱的计划。用 `/plan` 或 <kbd>Shift</kbd>+<kbd>Tab</kbd> 来切换。

**让 Codex 反过来问你：** 如果你大致知道自己想要什么，但说不太清楚，那就让它先来问你。告诉它质疑你的假设，把模糊的想法一点点逼成具体的东西，然后再开始写代码。

**用 PLANS.md 模板：** 对更进阶的工作流，你可以让 Codex 按一个 `PLANS.md` 或执行计划模板来做那些跑得久、步骤多的活。更多细节见[执行计划指南](https://developers.openai.com/cookbook/articles/codex_exec_plans)。

## 用 `AGENTS.md` 把指导沉淀下来，反复用

一个提示模式一旦好用，下一步就是别再每次手动重复它了。这正是 [AGENTS.md](https://developers.openai.com/codex/guides/agents-md) 该上场的地方。

可以把 `AGENTS.md` 看成写给智能体的 README，格式开放。它会自动加载进上下文，是你和团队写下「希望 Codex 在这个仓库里怎么干活」的最佳地方。

一份好的 `AGENTS.md` 应该涵盖：

- 仓库的结构和重要目录
- 怎么运行这个项目
- 构建、测试和 lint 命令
- 工程惯例和对 PR 的要求
- 约束，以及哪些事不要做
- 「做完」到底指什么，又该怎么验证

CLI 里的 `/init` 斜杠命令，能在当前目录快速搭出一份起步用的 `AGENTS.md`。这是个很好的起点，但你应该改一改，让它贴合团队实际怎么构建、测试、审查和发布代码。

`AGENTS.md` 可以分好几个层级来写：放在 `~/.codex` 里的全局文件，用作个人默认；放在仓库里的文件，用作团队共享的标准；放在子目录里的文件，用作更局部的规则。哪份文件离你当前目录更近、更具体，就听哪份的。

把它写得实用些。一份简短、准确的 `AGENTS.md`，比一份满是空泛规则的长文件有用得多。先从基础写起，等你发现某个错误反复出现，再加新规则。

如果 `AGENTS.md` 开始变得太大，就让主文件保持简洁，把规划、代码审查、架构这类内容拆到各自的 markdown 文件里，再从主文件引用过去。

## 把 Codex 配置好，表现才稳定

配置，是让 Codex 在不同会话、不同界面之间表现一致的主要手段之一。比如，你可以给模型选择、推理力度、沙盒模式、审批策略、配置档案和 MCP 设置都定好默认值。

一个不错的起步模式是：

- 个人默认值放在 `~/.codex/config.toml`（设置 → 配置 → 从 Codex 应用打开 config.toml）
- 仓库专属的行为放在 `.codex/config.toml`
- 命令行覆盖只用在一次性的场合（如果你用 CLI）

[`config.toml`](https://developers.openai.com/codex/config-basic) 是你写下长期偏好的地方，比如 MCP 服务器、配置档案、多智能体设置和功能开关。你可以直接改它，也可以让 Codex 替你改。

Codex 自带操作系统级别的沙盒，有两个关键开关可以由你控制。审批模式决定 Codex 什么时候要先征得你同意才能跑命令；沙盒模式决定 Codex 能不能在目录里读写，以及它能碰哪些文件。

如果你刚接触编程智能体，就先用默认权限。默认把审批和沙盒都收紧，等到确实需要了，再针对你信得过的仓库或特定工作流去放松。

要知道，CLI、IDE 和 Codex 应用共用同一套配置层。想了解更多，看[示例配置](https://developers.openai.com/codex/config-sample)页面。

## 靠测试和审查，把可靠性提上来

别让 Codex 改完就完事。该写测试时让它写测试，让它跑相关检查、确认结果，并在你接受之前先审查一遍。

这一整套循环 Codex 都能替你做，前提是它知道「好」长什么样。这份标准，可以来自提示，也可以来自 `AGENTS.md`。

它可以包括：

- 为这次改动写或更新测试
- 跑对应的测试套件
- 检查 lint、格式或类型
- 确认最终行为跟需求对得上
- 审查 diff，揪出 bug、回归或有风险的写法

在 Codex 应用里把 diff 面板打开，就能直接在本地[审查改动](https://developers.openai.com/codex/app/review)。点某一行就能留下反馈，这些反馈会作为上下文带进 Codex 的下一轮。

这里有个好用的选择，是斜杠命令 `/review`，它给你几种审查代码的方式：

- 对着一个基础分支审查，做 PR 风格的审查
- 审查还没提交的改动
- 审查某一个提交
- 用你自定义的审查指令

如果你和团队有一份 `code_review.md`，并在 `AGENTS.md` 里引用它，那 Codex 审查时也会照着这份指导走。对那些希望审查行为在各个仓库、各个贡献者之间都保持一致的团队来说，这是个很强的做法。

Codex 不该只会生成代码。给对了指令，它还能帮你**测试、检查和审查**这些代码。

如果你用 GitHub Cloud，可以设置让 Codex 给你的 PR 跑[代码审查](https://developers.openai.com/codex/integrations/github)。在 OpenAI，100% 的 PR 都由 Codex 审查。你可以开启自动审查，也可以在 @Codex 的时候让它来审。

## 用 MCP 接入外部上下文

当 Codex 需要的上下文不在仓库里时，就用 MCP。它能让 Codex 接上你本来就在用的工具和系统，省得你一遍遍把实时信息复制粘贴进提示里。

[模型上下文协议](https://developers.openai.com/codex/mcp)，也就是 MCP，是一个把 Codex 接到外部工具和系统的开放标准。

以下情况适合用 MCP：

- 需要的上下文不在仓库里
- 数据经常变
- 你想让 Codex 直接调用一个工具，而不是靠你粘贴指令
- 你需要一套能在多个用户或项目间重复使用的集成

Codex 同时支持 STDIO 和带 OAuth 的 Streamable HTTP 服务器。

在 Codex 应用里，去设置 → MCP 服务器，能看到自定义的和推荐的服务器。很多时候，Codex 能帮你把需要的服务器装上，你只管开口问就行。你也可以在 CLI 里用 `codex mcp add` 命令，填上名称、URL 和其他细节，加上你自己的服务器。

只在某个工具能真正打通一条工作流时，才把它加进来。别一上来就把你用的每个工具都接上。先从一两个工具开始——它们能明显省掉你经常要手动跑的某个循环——再慢慢往外扩。

## 把反复要做的事变成技能

一个工作流一旦变得可重复，就别再靠长提示或反复来回了。用[技能](https://developers.openai.com/codex/skills)，把指令、上下文，还有 Codex 该一致照做的支持逻辑，一起打包进一个 SKILL.md 文件。技能在 CLI、IDE 扩展和 Codex 应用里都能用。

让每个技能只盯着一件事。先从 2 到 3 个具体用例做起，把输入和输出定清楚，再把描述写明白：这个技能是做什么的、什么时候该用它。把用户真正会说的那种触发用语也写进去。

别一开始就想着把每种边角情况都覆盖到。先挑一个有代表性的任务，把它打磨好，再把这条工作流变成技能，然后慢慢改进。脚本或额外的资源，只在能提升可靠性时才加。

有一条经验法则：如果你总在重复用同一个提示，或者总在纠正同一条工作流，那它八成该做成技能了。

技能特别适合那些反复出现的活，比如：

- 日志分类
- 起草发布说明
- 对着清单审查 PR
- 迁移规划
- 遥测或事故总结
- 标准的调试流程

要搭一个技能的第一个版本，从 `$skill-creator` 技能入手最合适。迭代的时候，先把第一个版本留在本地。等它成熟到可以广泛分享了，再打包成一个[插件](https://developers.openai.com/codex/plugins/build)。技能里最要紧的部分之一就是描述，它要说清楚这个技能做什么、什么时候用。

个人技能存在 `$HOME/.agents/skills` 里；团队共享的技能可以签入仓库里的 `.agents/skills`。这对新队友入职特别有帮助。

## 把重复的活交给自动化

一个工作流一旦稳定下来，你就能安排 Codex 在后台替你跑它。在 Codex 应用里，[自动化](https://developers.openai.com/codex/app/automations)让你给一个反复出现的任务，挑好项目、提示、节奏和执行环境。

一个任务对你来说一旦变得重复，你就能在 Codex 应用的「自动化」标签里建一个自动化。你可以选它在哪个项目里跑、跑什么提示（提示里可以调用技能）、按什么节奏跑。你还能选它是在一个专门的 git 工作树里跑，还是在你本地环境里跑。想了解更多，看 [git 工作树](https://developers.openai.com/codex/app/worktrees)。

适合交给自动化的，比如：

- 总结最近的提交
- 扫一遍可能的 bug
- 起草发布说明
- 检查 CI 失败
- 产出站会总结
- 按计划跑一些可重复的分析流程

有一句话挺好记：技能定义「怎么做」，自动化定义「什么时候做」。如果一条工作流还得你不停引导，那就先把它做成技能。等它变得可预测了，自动化才能让你事半功倍。

自动化别只用来执行，也用来反思和维护：回看最近的会话，把反复遇到的麻烦总结出来，再慢慢去改进提示、指令或整套工作流的设置。

## 用会话控制，管好跑得久的活

Codex 的会话不只是聊天记录。它们是一条条工作线程，会随着时间不断积累上下文、决策和动作——所以管好它们，对结果质量影响很大。

Codex 应用的界面最方便管理线程，因为你可以把线程固定住，还能创建工作树。如果你用 CLI，下面这些[斜杠命令](https://developers.openai.com/codex/cli/slash-commands)特别有用：

- `/experimental`：切换实验性功能，并写进你的 `config.toml`
- `/resume`：恢复一段保存过的对话
- `/fork`：开一条新线程，同时把原来的记录留着
- `/compact`：当线程变长，你想要一份前面上下文的摘要版时用。顺带一提，Codex 本来也会自动帮你压缩对话
- `/agent`：当你在跑并行的智能体，想在几个活动线程之间切换时用
- `/theme`：选一个语法高亮主题
- `/apps`：在 Codex 里直接用 ChatGPT 应用
- `/status`：查看当前会话的状态

每一块完整的活，保持一条线程。如果这些工作还属于同一个问题，那留在同一条线程里通常更好，因为它把推理的来龙去脉都保留了下来。只有当工作真的分了岔，才去 fork。

用 Codex 的[子智能体](https://developers.openai.com/codex/concepts/subagents)工作流，把有明确边界的活从主线程上卸下来。让主智能体专心对付核心问题，把探索、测试、分类这类活交给子智能体。

## 常见的坑

第一次用 Codex 时，有几个常见的坑要避开：

- 把长期规则一股脑塞进提示，而不是把它们挪进 `AGENTS.md` 或某个技能
- 不让智能体看到自己干的活——没交代清楚构建和测试命令到底该怎么跑
- 在多步骤的复杂任务上跳过规划
- 还没摸清工作流，就把电脑的完全权限交给 Codex
- 在同一批文件上跑实时线程，却不用 git 工作树
- 一个反复出现的任务，手动都还没跑稳，就急着做成自动化
- 把 Codex 当成一个要你一步步盯着的东西，而不是让它跟你自己的活并行推进
- 一个项目用一条线程，而不是一个任务用一条线程——这么干，时间一长，上下文会越来越臃肿，结果也越来越差
