# 上手 Codex 以及获得更好结果的成熟实践

如果你刚接触 Codex 或者编程智能体，这份指南会帮你更快获得更好的结果。它涵盖了让 Codex 在 [CLI](https://developers.openai.com/codex/cli)、[IDE 扩展](https://developers.openai.com/codex/ide) 和 [Codex 应用](https://developers.openai.com/codex/app) 中更有效的核心习惯，从提示和规划到验证、MCP、技能和自动化。

当你把 Codex 当作一个你随着时间配置和改进的队友，而不是一次性的助手时，它表现得最好。

一个有用的思考方式是：从正确的任务上下文开始，用 `AGENTS.md` 提供持久的指导，配置 Codex 以匹配你的工作流，用 MCP 连接外部系统，把重复的工作变成技能，并把稳定的工作流自动化。

## 强力的第一次使用：上下文和提示

即使你的提示并不完美，Codex 已经足够强大到有用。你常常可以用最少的设置把一个难题交给它，仍然得到很好的结果。清晰的[提示](https://developers.openai.com/codex/prompting)并不是获得价值的必要条件，但它确实让结果更可靠，尤其是在更大的代码库或更高风险的任务中。

如果你在一个大型或复杂的仓库中工作，最大的解锁是给 Codex 正确的任务上下文，以及一个清晰的结构来说明你想要完成什么。

一个好的默认做法是在你的提示中包含四样东西：

- **目标：** 你想要改变或构建什么？
- **上下文：** 哪些文件、文件夹、文档、示例或错误对这个任务重要？你可以用 @ 提及某些文件作为上下文。
- **约束：** Codex 应该遵循哪些标准、架构、安全要求或惯例？
- **完成标准：** 在任务完成之前应该有什么是真的，比如测试通过、行为改变，或者一个 bug 不再复现？

这帮助 Codex 保持聚焦、做更少的假设，并产出更容易审查的工作。

根据任务的难度选择一个推理级别，并测试什么最适合你的工作流。不同的用户和任务用不同的设置效果最好。

- 低，用于更快、范围明确的任务
- 中或高，用于更复杂的改动或调试
- 超高，用于长的、智能体式的、推理密集的任务

为了更快地提供上下文，试着在 Codex 应用里用语音听写来口述你想让 Codex 做什么，而不是打字。

## 困难任务先规划

如果任务复杂、模糊或难以描述清楚，让 Codex 在开始编码之前先规划。

有几种方法效果很好：

**用规划模式：** 对大多数用户来说，这是最简单、最有效的选项。规划模式让 Codex 收集上下文、问澄清性问题，并在实现之前构建一个更强的计划。用 `/plan` 或 <kbd>Shift</kbd>+<kbd>Tab</kbd> 切换。

**让 Codex 采访你：** 如果你对想要的东西有一个大致的想法，但不确定如何很好地描述它，让 Codex 先问你问题。告诉它挑战你的假设，把模糊的想法变成具体的东西，然后再写代码。

**用 PLANS.md 模板：** 对于更高级的工作流，你可以配置 Codex 遵循一个 `PLANS.md` 或执行计划模板，用于运行时间更长或多步骤的工作。更多细节见[执行计划指南](https://developers.openai.com/cookbook/articles/codex_exec_plans)。

## 用 `AGENTS.md` 让指导可复用

一旦一个提示模式有效，下一步就是不再手动重复它。这就是 [AGENTS.md](https://developers.openai.com/codex/guides/agents-md) 的用武之地。

把 `AGENTS.md` 想成给智能体的开放格式 README。它自动加载到上下文中，是编码你和你的团队希望 Codex 在一个仓库里如何工作的最佳位置。

一个好的 `AGENTS.md` 涵盖：

- 仓库布局和重要的目录
- 如何运行项目
- 构建、测试和 lint 命令
- 工程惯例和 PR 期望
- 约束和不要做的规则
- 完成意味着什么，以及如何验证工作

CLI 中的 `/init` 斜杠命令是在当前目录中搭建一个起步 `AGENTS.md` 的快速启动命令。它是一个很好的起点，但你应该编辑结果以匹配你的团队实际如何构建、测试、审查和发布代码。

你可以在不同层级创建 `AGENTS.md` 文件：一个全局 `AGENTS.md` 用于个人默认值，位于 `~/.codex`，一个仓库级别的文件用于共享标准，以及子目录中更具体的文件用于本地规则。如果在离你当前目录更近的地方有一个更具体的文件，那个指导胜出。

保持它实用。一个简短、准确的 `AGENTS.md` 比一个充满模糊规则的长文件更有用。从基础开始，然后只在你注意到重复的错误后才添加新规则。

如果 `AGENTS.md` 开始变得太大，保持主文件简洁，并引用任务特定的 markdown 文件，用于规划、代码审查或架构这类东西。

当 Codex 第二次犯同样的错误时，让它做一个复盘并更新 `AGENTS.md`。指导保持实用，并基于真实的摩擦。

## 配置 Codex 以保持一致性

配置是让 Codex 在不同会话和界面之间表现得更一致的主要方式之一。例如，你可以为模型选择、推理努力、沙盒模式、审批策略、配置档案和 MCP 设置设定默认值。

一个好的起步模式是：

- 把个人默认值放在 `~/.codex/config.toml` 中（设置 → 配置 → 从 Codex 应用打开 config.toml）
- 把仓库特定的行为放在 `.codex/config.toml` 中
- 只在一次性的情况下使用命令行覆盖（如果你用 CLI）

[`config.toml`](https://developers.openai.com/codex/config-basic) 是你定义持久偏好的地方，比如 MCP 服务器、配置档案、多智能体设置和功能标志。你可以直接编辑它，或者让 Codex 为你更新它。

Codex 自带操作系统级别的沙盒，并有两个你可以控制的关键旋钮。审批模式决定 Codex 何时请求你的许可来运行一个命令，沙盒模式决定 Codex 能否在目录中读或写，以及智能体可以访问哪些文件。

如果你刚接触编码智能体，从默认权限开始。默认保持审批和沙盒收紧，然后只在需求清晰之后才为受信任的仓库或特定工作流放松权限。

注意 CLI、IDE 和 Codex 应用都共享同样的配置层。在[示例配置](https://developers.openai.com/codex/config-sample)页面了解更多。

尽早为你的真实环境配置 Codex。许多质量问题其实是设置问题，比如错误的工作目录、缺少写入权限、错误的模型默认值，或缺少工具和连接器。

## 用测试和审查提高可靠性

不要止步于让 Codex 做一个改动。让它在需要时创建测试、运行相关检查、确认结果，并在你接受之前审查工作。

Codex 可以为你做这个循环，但只有在它知道"好"是什么样子的时候。那个指导可以来自提示或 `AGENTS.md`。

那可以包括：

- 为改动写或更新测试
- 运行正确的测试套件
- 检查 lint、格式或类型检查
- 确认最终行为匹配请求
- 审查 diff，找 bug、回归或有风险的模式

在 Codex 应用中切换 diff 面板，直接在本地[审查改动](https://developers.openai.com/codex/app/review)。点击某一行来提供反馈，这些反馈会作为上下文喂给 Codex 的下一轮。

这里一个有用的选项是斜杠命令 `/review`，它给你几种审查代码的方式：

- 针对一个基础分支审查，做 PR 风格的审查
- 审查未提交的改动
- 审查一个提交
- 使用自定义审查指令

如果你和你的团队有一个 `code_review.md` 文件并从 `AGENTS.md` 引用它，Codex 也可以在审查期间遵循那个指导。对于希望审查行为在不同仓库和贡献者之间保持一致的团队来说，这是一个强有力的模式。

Codex 不应该只是生成代码。有了正确的指令，它还可以帮助**测试它、检查它和审查它**。

如果你用 GitHub Cloud，你可以设置 Codex 来为你的 PR 运行[代码审查](https://developers.openai.com/codex/integrations/github)。在 OpenAI，Codex 审查 100% 的 PR。你可以启用自动审查，或者在你 @Codex 时让 Codex 被动地审查。

## 用 MCP 获取外部上下文

当 Codex 需要的上下文在仓库之外时使用 MCP。它让 Codex 连接到你已经使用的工具和系统，这样你就不必一直把实时信息复制粘贴到提示里。

[模型上下文协议](https://developers.openai.com/codex/mcp)，或 MCP，是一个连接 Codex 到外部工具和系统的开放标准。

在以下情况使用 MCP：

- 需要的上下文在仓库之外
- 数据频繁变化
- 你想让 Codex 使用一个工具，而不是依赖粘贴的指令
- 你需要一个跨用户或项目的可重复集成

Codex 同时支持 STDIO 和带 OAuth 的 Streamable HTTP 服务器。

在 Codex 应用中，前往设置 → MCP 服务器，查看自定义和推荐的服务器。通常，Codex 可以帮你安装需要的服务器。你只需要问。你也可以在 CLI 中用 `codex mcp add` 命令，用名称、URL 和其他细节添加你的自定义服务器。

只在工具解锁一个真实工作流时才添加它。不要一开始就把你用的每个工具都接进来。从一两个明显能去掉你已经经常做的手动循环的工具开始，然后从那里扩展。

## 把可重复的工作变成技能

一旦一个工作流变得可重复，就别再依赖长提示或重复的来回。用[技能](https://developers.openai.com/codex/skills)把指令打包到一个 SKILL.md 文件、上下文和 Codex 应该一致应用的支持逻辑里。技能在 CLI、IDE 扩展和 Codex 应用中都能用。

让每个技能聚焦于一个工作。从 2 到 3 个具体用例开始，定义清晰的输入和输出，并把描述写成说明这个技能做什么以及何时使用它。包含用户实际会说的那种触发短语。

不要一开始就试图覆盖每个边缘情况。从一个有代表性的任务开始，让它运行良好，然后把那个工作流变成一个技能，并从那里改进。只在脚本或额外资产能提高可靠性时才包含它们。

一个好的经验法则是：如果你不断重用同一个提示或纠正同一个工作流，它大概应该变成一个技能。

技能对以下这类反复出现的工作特别有用：

- 日志分类
- 发布说明起草
- 针对清单的 PR 审查
- 迁移规划
- 遥测或事故总结
- 标准调试流程

`$skill-creator` 技能是开始搭建技能第一个版本的最佳起点。在你迭代时把第一个版本保持在本地。当它准备好广泛分享时，把它打包成一个[插件](https://developers.openai.com/codex/plugins/build)。技能最重要的部分之一是描述。它应该说明这个技能做什么以及何时使用它。

个人技能存储在 `$HOME/.agents/skills` 中，共享的团队技能可以签入仓库内的 `.agents/skills`。这对入职新队友特别有帮助。

## 为重复的工作使用自动化

一旦一个工作流稳定了，你就可以安排 Codex 在后台为你运行它。在 Codex 应用中，[自动化](https://developers.openai.com/codex/app/automations)让你为一个反复出现的任务选择项目、提示、节奏和执行环境。

一旦一个任务对你来说变得重复，你就可以在 Codex 应用的自动化标签里创建一个自动化。你可以选择它在哪个项目里运行、它运行的提示（你可以调用技能）以及它运行的节奏。你还可以选择自动化是在一个专用的 git 工作树里运行还是在你的本地环境里运行。在 [git 工作树](https://developers.openai.com/codex/app/worktrees)了解更多。

好的候选包括：

- 总结最近的提交
- 扫描可能的 bug
- 起草发布说明
- 检查 CI 失败
- 产出站会总结
- 按计划运行可重复的分析工作流

一个有用的规则是：技能定义方法，自动化定义计划。如果一个工作流仍然需要很多引导，先把它变成一个技能。一旦它可预测，自动化就成了一个力量倍增器。

把自动化用于反思和维护，不只是执行。审查最近的会话，总结重复的摩擦，并随着时间改进提示、指令或工作流设置。

## 用会话控制组织长期运行的工作

Codex 会话不只是聊天历史。它们是随时间积累上下文、决策和动作的工作线程，所以好好管理它们对质量有很大影响。

Codex 应用 UI 让线程管理最容易，因为你可以固定线程并创建工作树。如果你用 CLI，这些[斜杠命令](https://developers.openai.com/codex/cli/slash-commands)特别有用：

- `/experimental` 切换实验性功能并添加到你的 `config.toml`
- `/resume` 恢复一个保存的对话
- `/fork` 创建一个新线程，同时保留原始的记录
- `/compact` 当线程变长，你想要更早上下文的总结版本时。注意 Codex 确实会自动为你压缩对话
- `/agent` 当你运行并行智能体并想在活动智能体线程之间切换时
- `/theme` 选择一个语法高亮主题
- `/apps` 在 Codex 中直接使用 ChatGPT 应用
- `/status` 检查当前会话状态

每个连贯的工作单元保持一个线程。如果工作仍然是同一个问题的一部分，待在同一个线程里通常更好，因为它保留了推理的轨迹。只在工作真正分叉时才 fork。

用 Codex 的[子智能体](https://developers.openai.com/codex/concepts/subagents)工作流，从主线程卸下有边界的工作。让主智能体聚焦于核心问题，用子智能体做探索、测试或分类这类任务。

## 常见错误

第一次使用 Codex 时要避免的几个常见错误：

- 用持久规则塞满提示，而不是把它们移到 `AGENTS.md` 或一个技能里
- 不让智能体看到它的工作，没有给出如何最好地运行构建和测试命令的细节
- 在多步骤和复杂任务上跳过规划
- 在你理解工作流之前给 Codex 对你电脑的完全权限
- 在同样的文件上运行实时线程，而不用 git 工作树
- 在一个反复出现的任务手动可靠之前就把它变成自动化
- 把 Codex 当作你必须一步步盯着的东西，而不是与你自己的工作并行使用它
- 每个项目用一个线程，而不是每个任务用一个线程。这随着时间导致臃肿的上下文和更差的结果
