# 使用 Claude Code：HTML 出人意料地好用

**作者：** Thariq ([@trq212](https://x.com/trq212))  
**日期：** 2026年5月9日  
**来源：** [Using Claude Code: The Unreasonable Effectiveness of HTML](https://x.com/Zephyr_hg/status/2052809885763747935)

Markdown 已经成了 AI 助手和我们沟通最常用的格式。它简单、到处能用，有基本的富文本支持，你自己也容易改。Claude 现在甚至很擅长在 markdown 里用 ASCII 画图。

但随着 AI 能做的事越来越多，我开始觉得 markdown 成了一种束缚。超过一百行的 markdown 文件，我基本读不下去。我想要更丰富的视觉效果、颜色、图表，也想方便地把内容分享给别人。

而且我越来越少自己去编辑这些文件了，更多是把它们当成需求文档、参考资料、头脑风暴的输出。就算要改，通常也是让 Claude 去改——这就把 markdown 最大的优点也消掉了。

所以我开始把 HTML 当成默认的输出格式，Claude Code 团队里也有越来越多的人在这样做。这篇文章说说原因。

（想先看例子的话，这里有很多：https://thariqs.github.io/html-effectiveness，看完记得回来读正文）

# 为什么选 HTML？

## 能装的信息多得多

![Image](images/img-01.jpg)

跟 markdown 比，HTML 能表达的东西要丰富得多。文档结构、标题、加粗这些当然没问题，但除此之外，它还能表示：

- 表格数据
- 带 CSS 的视觉样式
- SVG 插图
- 代码片段（用 script 标签）
- 用 HTML + JavaScript + CSS 实现的交互
- 用 SVG 和 HTML 画的工作流
- 用绝对定位和 canvas 表达的空间关系
- 图片

说实话，我觉得几乎没有什么信息是 Claude 能读懂但没法用 HTML 高效表达的。这就意味着，它是模型向你传递深度内容的极佳载体，你读起来也很方便。

相比之下，如果没法用 HTML，模型只能在 markdown 里做些比较笨的事情——比如用 ASCII 画图，或者我最喜欢举的例子，用 unicode 字符来"近似"表示颜色，就像下面这张 Claude Code 的截图。

![Image](images/img-02.jpg)

*Claude Code 试图在 markdown 里表示颜色*

## 读起来清楚多了

![Image](images/img-03.jpg)

Claude 能做的事越来越复杂，写出来的需求文档和计划也越来越长。在实际使用中，我发现超过 100 行的 markdown 我基本不会认真读，更别提让团队里的人去看了。

但 HTML 文档就好很多。Claude 可以用标签页、插图、链接把内容组织得便于导航，还可以做成移动端自适应，在不同设备上有不同的阅读体验。

## 分享更方便

Markdown 文件不太好分享，大多数浏览器不能直接渲染，通常只能作为附件发出去。

HTML 就简单了，只要把文件传到某个地方（比如 S3），就能分享链接。同事在哪里都能打开，直接查阅。

同样一份需求文档、报告或 PR 说明，做成 HTML 之后，真正被人读到的概率会高出很多。

## 可以双向交互

![Image](images/img-04.jpg)

HTML 文档是可以交互的。比如你可以让 Claude 在里面加上滑块或旋钮，用来调整设计参数，或者试试算法的不同选项。还可以让它提供一个"复制为提示"的按钮，把你在文档里调整好的内容一键转成可以粘贴到 Claude Code 的指令。

我关于 playgrounds 的文章里有更多双向交互的例子：https://x.com/trq212/status/2017024445244924382

## 能读取更多上下文

为什么要用 Claude Code 来生成 HTML，而不是 ClaudeAI 或 Claude Design 呢？很重要的一个原因是 Claude Code 能读取的信息多得多。

比如写这篇文章的时候，我让 Claude Code 读遍我的代码文件夹，找出所有我生成过的 HTML 文件，分类整理，然后做成一个带图表的 HTML 文件来展示每种类型。你在这篇文章里看到的图表，就是这样来的。

除了文件系统，Claude Code 还可以通过 MCP（比如 Slack、Linear 等）、浏览器（Chrome 里的 Claude）和 git 历史来获取额外的上下文。

## 就是好玩

和 Claude 一起做 HTML 文档更有意思，让我感觉自己真的参与了创作过程，光这一点就够了。

## 怎么上手

我有点担心有人看完这篇文章，马上去做一个 /html 技能什么的。这也不是不行，但我想说的是，其实根本不需要那么麻烦。直接让 Claude "做一个 HTML 文件"或者"做一个 HTML 输出"就行了。

关键是想清楚你要这个文件做什么、你打算怎么用它。也许以后你会做一个专门的技能，但现在我建议从头开始提示，先摸摸感觉，看看在不同场景下怎么用。

# 使用场景

讲得更具体一点——我在不同场景下做过很多 HTML 文件。可以在这里看到所有例子：https://thariqs.github.io/html-effectiveness/，下面先做个概览。

## 需求梳理、规划与探索

HTML 是 Claude 深入分析问题的好画布。开始处理一个新问题时，我不再只是做一个 markdown 计划，而是期望做出一整套 HTML 文件。比如，我可能先让 Claude Code 做头脑风暴，生成几种不同方向的探索。然后选一个深入下去，可能做一些原型图或代码片段。最后感觉差不多了，让它写一份实施计划。对计划满意之后，新建一个会话，把所有这些文件一起传进去，开始实施。

验证的时候，我也会让验证代理读这些文件，这样它对"要达到什么目标"就有更全面的理解。

![Image](images/img-05.jpg)

**示例提示：**

- 我不确定引导页该走哪个方向。生成 6 种截然不同的方案，布局、语气、信息密度都要有差异，把它们排在一个 HTML 文件的网格里，方便我并排对比。每种方案标注它在做的权衡。
- 在一个 HTML 文件里写一份完整的实施计划，要有原型图，要有数据流，还要附上我可能需要看的关键代码片段。做得好读好消化。

**适用场景：**

- 探索代码的不同实现方式
- 对比多种视觉设计方案

## 代码审查与理解

Markdown 文件不太适合展示代码。但用 HTML 就可以渲染 diff、注释、流程图、模块依赖关系等。用它来理解代理写的代码，做代码审查，或者给 PR 的审阅者解释改动。我发现这比 GitHub 默认的 diff 视图好用得多——现在我每次提 PR 都会附上一个 HTML 代码解释文件。

![Image](images/img-06.jpg)

**示例提示：**

帮我审查这个 PR，做一个描述它的 HTML 文件。我对流式处理和背压逻辑不太熟，重点讲这部分。渲染出实际的 diff，在旁边加内联注释，按严重程度用颜色区分，还有其他能帮助说清楚概念的东西都加上。

**适用场景：**

- 提交 PR
- 审查 PR
- 理解某个代码主题

## 设计与原型

Claude Design 本身就是基于 HTML 的，因为 HTML 在表达设计上非常灵活——哪怕你最终的目标界面不是 HTML 也一样。Claude 可以先在 HTML 里画出设计草图，再用你选择的语言实现，无论是 React、Swift 还是别的什么。

也可以用 HTML 来原型化交互效果，比如动画、操作反馈。让 Claude 加上滑块、旋钮，直到调出你想要的感觉。

![Image](images/img-07.jpg)

**示例提示：**

我想做一个新的结账按钮原型，点击时先播放一个动画，然后快速变成紫色。做一个带多个滑块和选项的 HTML 文件，让我可以尝试不同的动画效果，加一个复制按钮，把调好的参数复制出来。

**适用场景：**

- 做设计系统相关的输出物
- 调整组件
- 可视化组件库
- 原型化动画效果

## 报告、研究与学习

Claude Code 特别擅长把多个数据源的信息整合起来，生成可读性很强的报告。你可以让 Claude 检索你的 Slack、代码库、git 历史、互联网，然后为你自己、管理层或团队生成报告。

可以做成一份长文档，也可以做成交互式的说明，甚至是幻灯片。让 Claude 用 SVG 画图表，更直观。

比如我写 prompt caching 相关文章时，就让 Claude 读了 git 历史，给我生成了一份 HTML 深度研究文件，汇总了所有关于 prompt caching 的改动。

![Image](images/img-08.jpg)

**示例提示：**

我搞不懂我们的限流器到底是怎么工作的。读一下相关代码，做一个单页 HTML 解释器：一张 token bucket 流程图，3-4 个关键代码片段附上注释，底部加一个"注意事项"。针对只读一遍的人来优化。

**适用场景：**

- 解释某个功能是怎么运作的
- 搞清楚某个概念
- 给上司的每周进展报告
- 给管理层的事故复盘报告
- SVG 插图、流程图、技术图表等

# 自定义编辑界面

有时候，光靠文字很难说清楚你想要什么。这时候，我会让 Claude 给我做一个一次性的编辑器，专门为手头这件事服务。不是正式的产品，不是可以复用的工具，就是一个 HTML 文件，专门针对这一份数据。

关键是最后一定要有个导出：一个"复制为 JSON"或"复制为提示"的按钮，把我在界面里做的所有操作转回可以粘贴到 Claude Code 的内容。

![Image](images/img-09.jpg)

**示例提示：**

- 我需要重新整理这 30 个 Linear 任务的优先级。做一个 HTML 文件，每个任务是一张可拖动的卡片，分成"现在""下一步""以后""砍掉"四列。按你的判断先排好。加一个"复制为 markdown"按钮，导出最终排序，每列附一行理由。
- 这是我们的功能开关配置。做一个表单编辑器，按模块分组，显示开关之间的依赖关系，如果我开启了一个前置条件未满足的开关就警告我。加一个"复制变更"按钮，只显示改动的部分。
- 我在调整这个系统提示。做一个并排编辑器：左边是可编辑的提示，变量插槽高亮显示；右边是三个示例输入，实时渲染填入变量后的效果。加上字符/token 计数和复制按钮。

**适用场景：**

- 对任何内容重新排序、分类或分桶（任务、测试用例、反馈）
- 编辑结构化配置（功能开关、环境变量、带约束的 JSON/YAML）
- 调整提示词、模板或文案，实时预览效果
- 整理数据集，批准/拒绝条目，打标签，导出选择结果
- 标注文档、对话记录或 diff，并导出注释
- 选择难以用文字描述的值：颜色、缓动曲线、裁剪区域、cron 表达式、正则表达式

## 常见问题

我把这套做法讲给很多人听，反复被问到几个问题。

**token 消耗不是更多吗？**  
是的，markdown 通常用的 token 少一些。但 HTML 表达能力更强，而且我真正会去读，综合下来输出质量反而更好。Opus 4.7 的上下文窗口有 1M，多用一些 token 基本感觉不到。

**那你现在还用 markdown 吗？**  
老实说，几乎什么都不用了。不过我可能是比较极端的 HTML 偏执者。

**HTML 文件怎么查看？**  
本地用浏览器打开（让 Claude 帮你打开也行），或者传到 S3 就能分享链接。

**生成 HTML 不是要比 markdown 慢很多吗？**  
是的，HTML 大概要多花 2-4 倍的时间。但我觉得这个代价值得。

**版本管理怎么办？**  
这是 HTML 最大的缺点之一，HTML 的 diff 比 markdown 噪声大，不好看。

**怎么让 Claude 按我的风格来，别做得太丑？**  
前端设计插件可以帮 Claude 做出质量更好的 HTML。如果想匹配自己公司的风格，可以把 Claude 指向你的代码库，让它先做一个设计系统 HTML 文件，之后其他 HTML 文件都以这个为参考。

## 始终参与其中

说了这么多，我用 HTML 的根本原因，是它让我对 Claude 的工作真正有掌控感。我曾经有些担心——因为我不再认真读那些计划文档了，慢慢就只能任由 Claude 自己做决定。

但现在用了 HTML，反而比以前更有参与感。希望你也有这种感觉。
