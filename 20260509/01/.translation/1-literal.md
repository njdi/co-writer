# 使用 Claude Code：HTML 的惊人有效性

**作者：** Thariq ([@trq212](https://x.com/trq212))  
**日期：** 2026年5月9日  
**来源：** [Using Claude Code: The Unreasonable Effectiveness of HTML](https://x.com/Zephyr_hg/status/2052809885763747935)

Markdown 已经成为代理与我们通信所使用的主流文件格式。它简单、可移植、有一定的富文本能力，并且对你来说易于编辑。Claude 甚至已经擅长在 markdown 文件内部用 ASCII 制作图表。

但随着代理变得越来越强大，我感到 markdown 已经成为一种限制性的格式。我发现很难阅读超过一百行的 markdown 文件。我想要更丰富的可视化效果、颜色和图表，并且我希望能够轻松地分享它们。

我也越来越不自己编辑这些文件了，而是将它们用作规格说明、参考文件、头脑风暴输出等。当我确实进行编辑时，我通常是提示 Claude 来编辑它们，这消除了 markdown 最大的优势之一。

我已经开始倾向于使用 HTML 作为输出格式而不是 Markdown，并且越来越多地看到 Claude Code 团队的其他人也在使用这种方式，这就是原因所在。

（如果你想从一些示例开始，你可以在这里看到很多：https://thariqs.github.io/html-effectiveness，只是请确保回来阅读更多关于原因的内容）

# 为什么选择 HTML？

## 信息密度

![Image](images/img-01.jpg)

与 markdown 相比，HTML 可以传达更丰富的信息。它当然可以做简单的文档结构，如标题和格式，但它还可以表示各种其他信息，例如：

- 使用表格的表格数据
- 使用 CSS 的设计数据
- 使用 SVG 的插图
- 使用 script 标签的代码片段
- 使用 HTML 元素配合 javascript + CSS 的交互
- 使用 SVG 和 HTML 的工作流
- 使用绝对位置和画布的空间数据
- 使用图片标签的图像

我甚至可以说，几乎没有哪一组 Claude 能读取的信息是你不能用 HTML 相当高效地表示的。这使其成为模型向你传达深度信息以及你审阅信息的高效方式。

我发现，在无法做到这一点的情况下，模型可能会在 markdown 中做更低效的事情，比如 ASCII 图表，或者我最喜欢的，用 unicode 字符估算颜色，就像这张 Claude Code 的截图所示。

![Image](images/img-02.jpg)

*Claude Code 试图在 markdown 中展示颜色*

## 视觉清晰度与阅读便利性

![Image](images/img-03.jpg)

随着 Claude 能够做更复杂的工作，它也在编写越来越大的规格说明和计划。在实践中，我发现我往往不会真正阅读超过 100 行的 markdown 文件，而且我肯定无法让我组织中的任何其他人去读它。

但 HTML 文档更容易阅读，Claude 可以在视觉上组织结构，以便通过标签、插图、链接等进行理想的导航。它甚至可以是移动端响应式的，这样你可以根据你的设备尺寸以不同方式阅读它。

## 易于分享

Markdown 文件相当难以分享，因为大多数浏览器不能原生地良好渲染它们。你通常必须将它们作为附件添加到电子邮件或消息中。

使用 HTML，只要你上传文件（例如到 S3），你就可以轻松分享链接。你的同事可以在任何地方打开它并轻松参考它。

如果它是 HTML 格式，别人实际阅读你的规格说明、报告或 PR 写作的可能性要高出很多很多。

## 双向交互

![Image](images/img-04.jpg)

HTML 允许你与文档进行交互，例如你可能想让它添加滑块或旋钮来调整设计，或者允许你调整算法中的不同选项来查看会发生什么。你还可以要求它让你将这些更改复制到一个提示中，粘贴回 Claude Code。

阅读更多关于我的 playgrounds 文章，以查看这种双向交互的示例：https://x.com/trq212/status/2017024445244924382

## 数据摄取

为什么要用 Claude Code 来制作 HTML 文件，而不是 ClaudeAI 或 Claude Design 呢？最重要的原因之一是 Claude Code 可以摄取的所有上下文。

例如，在撰写这篇文章时，我让 Claude Code 读取我的代码文件夹，找到我生成的所有 HTML 文件，将它们分组和分类，然后制作一个 HTML 文件，其中包含代表每种类型的所有图表。你在这篇文章中看到的图表就是这一操作的直接结果。

除了文件系统，Claude Code 还可以使用你的 MCP（如 Slack、Linear 等）、你的网络浏览器（通过 Chrome 中的 Claude）、你的 git 历史等找到额外的上下文。

## 这很令人愉快

与 Claude 一起制作 HTML 文档更有趣，让我感到更多地参与和投入到创作中，这本身就已经足够了。

## 如何开始

我有点担心人们会读了这篇文章就把它变成一个 /html 技能或者什么的。虽然这可能有一些价值，但我想强调的是，你不需要做太多就能让 Claude 做到这一点。你只需要让它"制作一个 HTML 文件"或"制作一个 HTML 产物"。

诀窍在于知道你想要产物做什么以及你可能如何使用它。随着时间推移你可能会制作一个技能，但现在我建议就从头开始提示，以便了解如何在不同情况下使用它。

# 使用案例

为了让这更具体，我为不同的使用案例制作了许多不同的 HTML 文件。你可以在这里查看所有这些：https://thariqs.github.io/html-effectiveness/ 但这里有一个概述。

## 规格说明、规划与探索

HTML 是 Claude 深入研究问题的丰富画布。当我开始处理一个问题时，我期望制作一个 HTML 文件的网络，而不是一个简单的 markdown 计划。例如，我可能从让 Claude Code 进行头脑风暴开始，创建不同选项的一些探索。然后我会要求它更深入地探讨其中一个，也许制作模型图或代码片段。最后，当我感觉良好时，我会让它编写一个实施计划。当我对计划满意时，我会创建一个新会话，将所有这些文件传入以供实施。

在验证时，我还会让验证代理读取这些文件，它将拥有更广泛的关于所需内容的上下文。

![Image](images/img-05.jpg)

**示例提示：**

- 我不确定要如何处理引导屏幕的方向。生成 6 种截然不同的方法——变化布局、语气和密度——并将它们以网格形式排列在一个 HTML 文件中，以便我并排比较。为每种方法标注它所做的权衡。
- 在 HTML 文件中创建一个详尽的实施计划，确保制作一些模型图，展示数据流，并添加我可能想要审查的重要代码片段。让它易于阅读和消化。

**使用案例：**

- 探索在代码中实现某事的其他方式
- 探索多种视觉设计

## 代码审查与理解

代码在 Markdown 文件中难以阅读。但使用 HTML，我们可以渲染差异、注释、流程图、模块等。使用它来理解代理编写的代码，获取代码审查，或者向审查你代码的人解释 PR。我发现这通常比默认的 Github diff 视图效果更好，现在我为每个我制作的 PR 都附上了一个 HTML 代码解释器。

![Image](images/img-06.jpg)

**示例提示：**

通过创建一个描述此 PR 的 HTML 产物来帮助我审查它。我对流式处理/背压逻辑不太熟悉，所以请专注于那个方面。渲染实际的差异并带有内联边距注释，按严重性对发现进行颜色编码，以及其他任何可能有助于传达概念的内容。

**使用案例：**

- 创建 PR
- 审查 PR
- 理解代码中的主题

## 设计与原型

Claude Design 基于 HTML，因为 HTML 在设计方面具有令人难以置信的表现力，即使你的最终界面不是 HTML。Claude 可以在 HTML 中草绘设计，然后用你选择的语言编写，无论是 React、Swift 等。

你也可以原型化交互，比如动画、操作等。考虑让 Claude 制作滑块、旋钮等，以精确调整你想要的效果。

![Image](images/img-07.jpg)

**示例提示：**

我想原型化一个新的结账按钮，点击时它会执行播放动画，然后快速变成紫色。创建一个带有多个滑块和选项的 HTML 文件，让我尝试这个动画的不同选项，给我一个复制按钮来复制效果良好的参数。

**用于：**

- 创建设计系统产物
- 调整组件
- 可视化组件库
- 原型化令人愉快的动画

## 报告、研究与学习

Claude Code 非常擅长综合多个数据源的信息，并将其转换为可读性报告。你可以提示 Claude 搜索你的 Slack、你的代码库、git 历史、互联网等，并用它来为你自己、领导层、你的团队等生成极其可读的报告。

你可以以长 HTML 文档、交互式解释器甚至幻灯片/演示文稿的形式组装这些内容。让 Claude 使用 SVG 来绘制图表以帮助可视化。

例如，对于我关于提示缓存的文章，我在阅读 git 历史后，让 Claude 为我准备了一个关于我们对提示缓存所有更改的 HTML 深度研究文件。

![Image](images/img-08.jpg)

**示例提示：**

我不了解我们的速率限制器实际上是如何工作的。读取相关代码并生成一个单页 HTML 解释器：一个令牌桶流程图，3-4 个关键代码片段带注释，以及底部的"注意事项"部分。针对只读一次的人进行优化。

**用于：**

- 总结某个功能是如何工作的
- 向我解释一个概念
- 给你老板的每周状态报告
- 给你领导层的事件报告
- SVG 插图、流程图、技术图表等

# 自定义编辑界面

有时候，很难仅凭文本框来描述你想要什么。在这种情况下，我会让 Claude 为我正在处理的确切事物构建一个一次性编辑器。不是一个产品，或者一个可重用的工具，而是一个单一的 HTML 文件，专门为这一份数据而构建。

诀窍始终是以导出结束：一个"复制为 JSON"或"复制为提示"按钮，将我在 UI 中所做的任何事情转换回我可以粘贴到 Claude Code 中的内容。

![Image](images/img-09.jpg)

**示例提示：**

- 我需要重新排列这 30 个 Linear 票据的优先级。给我制作一个 HTML 文件，每个票据作为一个可拖动的卡片，跨越现在/下一步/以后/削减列。按你的最佳猜测预先排序。添加一个"复制为 markdown"按钮，导出最终排序，每个桶附带一行理由。
- 这是我们的功能标志配置。为它构建一个基于表单的编辑器，按区域分组标志，显示它们之间的依赖关系，如果我启用一个前提条件关闭的标志则警告我。添加一个"复制差异"按钮，给我仅更改的键。
- 我正在调整这个系统提示。制作一个并排编辑器：左侧是可编辑的提示，变量槽突出显示，右侧是三个示例输入，实时重新渲染填充好的模板。添加字符/令牌计数器和复制按钮。

**用于：**

- 重新排序、分类或分桶任何内容（票据、测试用例、反馈）
- 编辑结构化配置（功能标志、环境变量、带约束的 JSON/YAML）
- 调整提示、模板或带实时预览的文案
- 整理数据集，批准/拒绝行，标记示例，导出选择
- 注释文档、对话记录或差异，并导出注释
- 选择难以用文字表达的值：颜色、缓动曲线、裁剪区域、cron 计划、正则表达式。

## 常见问题

我已经告诉了很多人我是如何切换到 HTML 的，我看到了一些重复的问题。

**它的令牌效率不是更低吗？**  
虽然 markdown 通常使用更少的令牌，但我发现 HTML 的额外表现力以及我更高的阅读可能性意味着我获得了整体上更好的输出。有了 Opus 4.7 中的 1MM 上下文窗口，增加的令牌使用在上下文窗口中并不真正明显。

**你现在什么时候使用 markdown？**  
老实说，我已经停止几乎所有事情都使用 markdown 了，但我可能处于 HTML 最大化主义的极端一侧。

**我如何查看 HTML 文件？**  
我倾向于在本地浏览器中打开它（你可以让 Claude 打开它），或者如果你想要一个可分享的链接，则上传到 S3。

**这不是比 markdown 花更长时间来生成吗？**  
确实更长！HTML 可能比 Markdown 长 2-4 倍，但我发现结果是值得的。

**版本控制呢？**  
老实说，这是 HTML 最大的缺点之一，HTML 差异与 Markdown 相比噪声更大，难以审查。

**我如何让 Claude 符合我的品味/不让它变得丑陋？**  
前端设计插件帮助 Claude 制作好的 HTML 文件。但要与你自己公司的风格匹配，你可以通过将 Claude 指向你的代码库来创建一个单一的设计系统 HTML 文件。然后你可以将该设计系统文件作为其他 HTML 文件的参考。

## 保持同步

综上所述，我认为我使用 HTML 的真正原因是我感觉与 Claude 更加同步。我曾经开始担心，因为我已经停止深入阅读计划，我将不得不让 Claude 自己做选择。

但我很高兴地说，使用 HTML 时，我感觉比以往任何时候都更加同步。我希望你也是如此。
