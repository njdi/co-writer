# 直译稿问题清单

## 标题

- "那个让我产出提升10倍的CLAUDE.md文件" — "产出提升10倍"读起来像公司KPI语言，失去了原文"10x'd"的那种爽感。建议改为"让效率直接翻10倍"或"帮我翻了10倍效率"。

## 开篇段落

- "将其视为绝对真实" — "绝对真实"像哲学术语，不自然。原文 ground truth 在工程语境里是"唯一依据"/"最终参考"，改为"把它当作整个会话的最终依据"更贴切。
- "个性指令" — 太技术化，读起来不像口语。原文 personality instructions 是指让 Claude "扮演角色"的指令，改为"人设类指令"或"角色扮演指令"。
- "给高级工程师提供清晰简报来入职" vs "把新员工扔进没有文档的代码库" — 直译基本到位，但"扔进"比"throwing"更生动，保留这个词。

## 为什么不起作用

- "遗漏东西" — 可以，但"开始漏规则"更口语化。
- "泛泛建议" — 准确，保留。
- "没有层次结构" — 太正式。改为"没有分层"更简洁。

## 5个部分

- "Claude不知道如何构建、测试或检查你的项目 = 告诉它。" — 这个公式感的句子原文也是这样，翻译准确，保留。
- "不是完整的目录列表。只要足够让Claude知道东西在哪里和什么放在哪里。" — "东西在哪里和什么放在哪里"重复啰嗦，改为"只要让Claude知道各模块在哪、各类代码放哪就够了"。
- "硬性规则（没有你Claude会出错的事情）" — 括号里的解释很绕，改为"（少了你，Claude必然出错的地方）"更通顺。
- "同样重要的是你省略的内容" — 直译正确，但语感平。改为"留白同样重要"更有力。

## 完整模板

- "可以复制和调整的生产就绪CLAUDE.md" — "生产就绪"是工程黑话，对目标读者来说不友好。改为"拿来就能用的CLAUDE.md"。
- "涵盖Claude需要的一切，没有它不需要的" — 啰嗦。改为"包含所有必要信息，没有一行多余"。

## 改变一切的规则

- "这些是对输出质量产生最大差异的行" — "产生最大差异"又是一个翻译腔。改为"这几行对输出质量影响最大"。

## 每个人都会犯的错误

- "这个文件随时间复利增长" — "复利增长"这个比喻原文也用了（compounds），但"复利"在中文里主要是金融术语，可以改为"这个文件的价值会随时间累积"，更自然。
- "和真正重要的指令争夺注意力" — 这句话原文是 "noise competing for attention with the instructions that actually matter"，直译还不错，但可以更简练：改为"把 Claude 的注意力从真正重要的指令上分走"。

## 语气总体问题

直译稿整体偏"翻译腔"，句子结构照搬英文，尤其是复合从句。
意译时需要把长句拆开，或者用更口语化的方式重组。
原作者的语气是直接、略带俏皮、懂行的工程师朋友，意译要保持这个劲。
