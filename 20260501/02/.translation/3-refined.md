# 动态软件

**作者：** Ashpreet Bedi ([@ashpreetbedi](https://x.com/ashpreetbedi))  
**日期：** 2026年5月1日  
**来源：** [Dynamic Software](https://x.com/Zephyr_hg/status/2049904901371633815)

五十年来，软件一直是静态的。

你用过的每一个程序，都是一堆函数跑在硬编码的控制流里：If、else、while、for。函数做实际的工作——从数据库读数据，调用 API，转换数据。

相同输入 = 相同输出。这是五十年来的契约。

然后 2024 年来了。控制流活了，创造出一个全新的软件类别：活的、动态的、按需的软件。

## 软件已死，软件万岁

静态软件是一段录音。你按下播放键，得到的就是当时录下的东西。同样的音符，同样的顺序，每次都一样。那次演奏发生在某个开发环境里，从此它就一遍又一遍播放同一首曲子。

动态软件是一支现场管弦乐队。

乐谱在。乐器在。乐手在。但今晚这个房间里会发生什么，取决于指挥，取决于演奏者，取决于此刻。模型是指挥，工具是乐器，控制流是演奏本身——不是录音。

这就是人们在使用一个出色的 agent 时的感受，却说不清楚哪里不一样。他们一辈子都在和按钮打交道。现在，他们第一次置身于一场现场演出。软件正在回应他们，就在此刻，带着判断，带着在场感。它在倾听。它在调整。它以一种软件从未有过的方式，活着。

录音是完美的。现场演出不是。现场管弦乐队会做选择，有时失误，有时给你惊喜。我们之所以还愿意花钱听现场音乐，是因为房间里会发生不同的事。

演奏本身就是意义所在。

动态软件是活的。它不是确定性的。它不是完美的。而一旦你感受过这种不同，录音就会变成它本来的样子——凝固的。

我们不是在制作更好的录音。我们正在构建第一代能够演奏的软件。

## 动态软件打破的假设

当软件活过来，所有建立在静态软件上的假设都会崩塌。

**确定性崩塌了。** 相同的输入不再意味着相同的输出。模型会考虑上下文、记忆、经验。软件在周二下午做的事，和周一上午做的可能不一样。这在文本层面还能（在一定程度上）控制，但下一个浪潮是视觉——图表、仪表板、整个屏幕都按需生成。别试图在非确定性软件上强加确定性，干脆放手，享受这趟旅程。

**状态和时间的运作方式不同了。** 静态程序不需要记太多东西。控制流每次都一样，所以状态存在数据库里，只做 CRUD。在动态软件里，状态*就是*上下文——过去会话的记忆，什么方法奏效的历史，对领域的了解。数据库不再是存储，而变成了软件运行所依赖的上下文。

由此带来的是会话问题。静态 API 端点在设计上是无状态的，每个请求都是独立的。动态软件恰恰相反——会话是一个连续的上下文，跨越几分钟、几天，有时甚至几周。用户回来了，agent 从上次离开的地方继续。会话变成了一等公民。

时间也变了。静态软件在毫秒内返回，最多几秒。动态软件要推理，要调用工具，要等工具返回，然后再次推理。一次请求有时要花好几分钟。流式传输是默认方式，后台执行是核心原语。HTTP 请求/响应模型开始吃力、断裂，默认的 29 秒负载均衡器超时也撑不住了。

**软件需要监视自己。** 静态软件，你可以读代码知道它做了什么。动态软件，不行。控制流是一个模型，模型是不透明的。了解软件做了什么的唯一办法，就是记录它做的每一件事——每一步推理，每一次工具调用，每一次检索。追踪从一个调试工具，变成了理解你的软件的唯一途径。

只是看还不够。静态程序不做决策，所以没什么需要审批的。动态软件做决策，决策有后果。有些可以自由决定，有些需要用户确认，有些需要管理员审批。你的软件必须把这些说清楚，你的运行时必须强制执行。

这些每一个都是真实的工程问题。每个构建动态软件的团队都会碰到所有这些。大多数人要花好几个月，从头解决。

## 新类别需要新的运行时

静态软件有成熟的运行时。你写 Django 或 Express，部署到托管平台，不需要考虑 HTTP、会话、扩展或恢复。基础设施是解决好的，平台帮你处理。

动态软件没有等价物。你写了一个 agent，然后在它周围花六个月时间堆基础设施，手动修每一个边缘情况。这些边缘情况只有在大规模跑 agent 之后才会遇到。SSE + WebSocket，流式传输 + 后台执行，能在重启后恢复的会话，你实际上能查询的存储而不是五个供应商硬拼在一起，等待管理员签字的审批门而不只是用户确认，每资源、每工具的 RBAC，能在 Slack、Telegram、WhatsApp 上用的 agent——因为没人想用一个定制 UI。

这就是为什么 80% 的 agent 跑不起来，最后一公里有太多苦活要磨。

上一次这么大的转变，是从桌面应用到 Web 应用。Web 软件需要自己的运行时、自己的协议、自己的基础设施、自己的开发者工具。我们花了二十年把这一切建起来。

动态软件来了，从零开始。自己的运行时。自己的协议。自己的基础设施。自己的开发者工具。

## 未来十年

静态软件用了五十年才成熟。操作系统、数据库、Web 服务器、部署平台、可观测性栈、身份提供商——我们忘了这些东西有多新。Heroku 是 2007 年，Kubernetes 是 2014 年，Vercel 是 2015 年。我们如今习以为常的基础设施，比大多数用它来构建东西的人还年轻。

动态软件在第一年。

谁来构建运行时、协议、开发者工具、平台，谁就定义下一个软件时代。前方的工作是巨大的。这也是我过去十五年里做过的最有趣的工作。
