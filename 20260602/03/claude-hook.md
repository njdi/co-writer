# 老错过 Claude 干完活？我干脆让它自己给自己配了个提示音

> 用 Claude Code 的时候，我常有个小烦恼：让它干点活，我顺手就把窗口切走忙别的去了，结果它干完了、或者中途卡在「等我点确认」那儿，我完全不知道，回头一看才发现早就停那儿等着了。这篇就从给它配一个提示音讲起：先确认它现在到底会不会响，再认识一下背后那个叫 hook 的机制，最后一步步把声音配好——包括怎么挑一个不吵的音、怎么调音量。原理只点到为止，跟着做就能用上。

## 一个很现实的烦恼

我用 Claude Code 的习惯是这样的：交代它一件事，比如改段代码、跑个命令，然后我就切到别的窗口去忙了——回个消息、看个文档。

问题就出在这个「切走」上。

Claude 可能几十秒就干完了，也可能干到一半停下来，等我点一下「允许」才能继续。可我人不在这个窗口啊。它干完了我不知道，它卡住等我我也不知道，常常是过了好久我才想起来切回去看一眼——好家伙，早就停那儿了。

我就想，能不能让它**干完、或者需要我确认的时候，响一声**？哪怕我切走了，听到声也能马上回来。

于是我就把这个困扰，原原本本地说给了 Claude Code 听。注意，我当时**什么都不懂**——不知道有「hook」这么个东西，更不知道能直接用系统自带的声音，我就只是描述了一下我的难处：

> **我：** 你执行完、或者需要我确认的时候，能不能给我个提示音？要不然我窗口一切走，你干完了我也不知道。
>
> **Claude Code：** 明白。这个需求可以用 Claude Code 的「hook」机制实现——在「我答完」和「我需要你确认」这两个时机，自动播放系统提示音。要我帮你配吗？

你看，我抛出的只是一个大白话的烦恼，「hook」「系统提示音」这些词，都是它先讲出来的。

## 先别急着配，它现在到底会不会响？

我的第一反应是：是不是本来就有提示音，只是我没开、或者没注意到？

这里有个细节我想先说一下：整个过程，我**没有自己去翻文档、查教程**研究「Claude Code 怎么配提示音」。我是直接把这个需求丢给了 Claude Code 自己——让它去查、去确认、去配置。说白了，这是一个 Claude Code 自己的问题（它干完活不提醒我），而我做的，就是**让它自己来解决自己的问题**。这一点后面我还会再聊，你先记住这个味道。

所以在动手配置之前，我先喊了停，让它帮我确认了一下现状：

> **我：** 你先别配。先帮我看看，我现在到底会不会有提示音？
>
> **Claude Code：** （翻了一遍配置文件后）现在不会响。你的配置里只有主题、默认档位这些基础项，没有任何跟声音或事件提醒相关的设置——所以切走窗口收不到提醒，跟你的感受一致。

结果很干脆：**什么都没有**。配置文件里只有主题颜色、默认档位这些基础项，一条跟声音、跟「事件提醒」相关的设置都没有。

这下踏实了。不是我没开，是它**默认就不会响**——切走窗口收不到任何提醒，跟我的感受完全对得上。那接下来要做的，就是给它**加**一个。

这里插一句，我挺推荐这个「先确认现状再动手」的习惯。很多时候我们急着改设置，结果改了半天发现方向就不对。先花十几秒看一眼当前是什么样，比埋头瞎调要省事得多。

## 认识一下 hook：不是写代码，是一句「约定」

那「加一个提示音」具体是加什么呢？

这里要请出 Claude Code 的一个机制，叫 **hook**，中文一般叫「钩子」。名字听着技术，但它干的事特别朴素，你可以把它理解成你和这个工具之间的一句**约定**：

> 「**当某件事发生的时候，你帮我自动跑一下这条命令。**」

就这么简单。在我们这个场景里，这句约定就是：

> 「当你干完活的时候，帮我**播放一声提示音**。」

注意，这里有两个常见的误会，先澄清掉：

- **它不需要写代码。** 配置 hook 就是往一个文本配置文件里加几行，描述清楚「什么事件 → 跑什么命令」，是纯粹的配置，不是编程。
- **不是让 AI 去执行。** 这条命令是 Claude Code 这个工具本身在合适的时机替你跑的，跟 AI「聪不聪明」没关系，所以它非常稳定、可预期。

明白了 hook 是「事件 → 命令」这么个约定，剩下的就是两个问题：盯哪个**事件**？跑哪条**命令**（也就是怎么发出声音）？

## 两个关键事件：Stop 和 Notification

Claude Code 能盯的事件有好几种，但跟「提示音」真正相关的，其实就两个，名字也很直白：

- **Stop**：我（Claude）**每次回答结束**的那一刻。对应你的需求就是——「活干完了」。
- **Notification**：我**需要你来一下**的时候——比如要你点「允许」才能继续，或者我停下来在等你输入。对应的就是——「需要你确认」。

你发现没有，这俩**刚好**把开头那个烦恼完整覆盖了：一个管「干完了」，一个管「等你呢」。所以我就让这两个事件各绑一声提示音。

（还有别的事件，比如整个会话开始、结束之类的，但跟「切走窗口怕错过」这件事没关系，加上反而徒增杂音。所以就这两个，够用且干净。）

## 怎么发出声音：挑一个不吵的

事件定了，接下来是「跑哪条命令」——也就是用什么发声。

Mac 自己就带了一批系统音效，藏在系统目录里，有十几个，名字五花八门：Glass、Ping、Hero、Funk、Pop、Submarine…… 播放它们用的是 Mac 自带的一个小命令 `afplay`，后面跟上音效文件就行，不用额外装任何东西。

光看名字我哪知道 Glass、Funk 是啥动静？所以我干脆提了个要求：

> **我：** 我能先听一下这些系统音效吗？
>
> **Claude Code：** 当然，我直接放给你听。我挑几个适合做提示音的，按顺序一个个播。

接下来有意思的来了：它**当场写了一小段命令，把音效挨个播放出来**——放完一个停一下，再放下一个，还在旁边标好「现在放的是第几个、叫什么名字」。后来我嫌听不清，又让它**每个连放三遍**，它就把命令改成连播三次。再后来我看中了第 5 个，让它单独再放一次，它也照做。

> **我：** 你再播一下第五个，我听一下。
>
> **Claude Code：** 好，这是第 5 个 Funk——两声低沉的「嘟嘟」，挺有「该你了」的提醒感。

就这么边放边挑，我心里就有数了——这一步我觉得特别值，光看名字根本想象不出来，听一遍全明白了：有的很清脆，像玻璃叮一声（Glass）；有的是一声短促的「叮」（Ping）；有的上扬，像「完成、过关」的感觉（Hero）；而我最后选中的 Funk，那两声低沉的「嘟嘟」最对我的胃口。

### 顺手把音量也调了

听的过程中我发现，能不能让它响得更明显一点？毕竟是用来「把我从别的窗口拉回来」的，太小声就白搭了。

答案是可以。播放命令上加一个音量参数就行：

- 原始音量是 `1.0`；
- 想轻一点，就 `0.5`、`0.2`；
- 想更响、更抓人，就往上调，`2.0` 就是放大到两倍。

我同样是挨个档位听了一遍，最后选了 **2.0**——毕竟目的就是「确保我能注意到」，宁可响一点。

这里也提醒一句：这个音量是**相对这个音效本身**的放大，最终多大声还会叠加你 Mac 的系统音量。所以它调的是「这个提示音相对其它声音有多突出」，不是绝对响度。

到这一步，我的配置就成型了：**干完活（Stop）和需要确认（Notification）时，都用两倍音量播放那声 Funk。**

## 把配置落到文件里

说了这么多，到底改的是哪个文件、写成什么样？这一节直接给你，照抄即可。

要改的文件是 Claude Code 的全局配置（放这里，所有项目都生效）：

```
~/.claude/settings.json
```

打开它，你大概会看到类似这样的内容（具体几项因人而异）：

```json
{
  "effortLevel": "medium",
  "theme": "dark"
}
```

我们要做的，是给它**加上 `hooks` 这一段**。加完之后长这样：

```json
{
  "effortLevel": "medium",
  "theme": "dark",
  "hooks": {
    "Stop": [
      {
        "hooks": [
          { "type": "command", "command": "afplay -v 2.0 /System/Library/Sounds/Funk.aiff" }
        ]
      }
    ],
    "Notification": [
      {
        "hooks": [
          { "type": "command", "command": "afplay -v 2.0 /System/Library/Sounds/Funk.aiff" }
        ]
      }
    ]
  }
}
```

对着前面讲的几样东西，这段配置就一目了然了：

- `Stop` / `Notification` —— 就是前面说的两个事件：「干完了」和「需要你确认」。
- `afplay` —— Mac 自带的播放命令。
- `-v 2.0` —— 两倍音量。想轻一点就把它改成 `0.5`；不想调音量，整个 `-v 2.0` 去掉也行。
- `Funk.aiff` —— 选中的音效。想换别的，把文件名换掉即可，比如 `Glass.aiff`、`Hero.aiff`。

几个容易踩的小地方，提醒一下：

- **JSON 对逗号、括号很挑剔。** 加完最好检查一下格式有没有错（多一个逗号、少一个括号都会让它失效）。
- **改完要重开一次 Claude Code 才生效**——这类配置通常是在启动时加载的，当前这个会话里改了往往不会立刻响。重开后随便让它干点小事，结束时就能听到了。
- 上面用的音效路径是 Mac 的系统目录，**这套写法是 macOS 专属的**（`afplay` 是 Mac 的命令）。其它系统发声的命令不一样，思路则是相通的。

## 往深一层：为什么它过一会儿又响了一次？

用起来之后，我注意到一个现象，又顺口问了它：

> **我：** 我好像觉得，有时候它已经响过一次了，我没做任何操作，过一会儿它又响了一次，会这样吗？
>
> **Claude Code：** 会，这是预期行为，你没记错。

不是 bug，是「设计如此」。

关键在 Notification 这个事件。它其实有**两种**触发来源：

1. **我真的需要你确认权限**——弹出「是否允许执行 XX」那种，这是你最想要的提醒;
2. **我停下来等你输入、而你一直没动静**——这种「在等你哦」的提醒，是会**隔一阵子重复响**的，本意就是怕你彻底忘了还有事没处理。

我遇到的「响过一次、没操作、过会儿又响」，就是第 2 种在重复。

知道原因之后，处理起来就有选择了：可以接受它偶尔重复（我自己就是这么选的，多提醒一下也不坏）；如果嫌烦，也可以做成「只在真正要你点确认时才响、单纯空闲不响」。怎么选看个人，但**先搞懂它为什么这样，再决定要不要管它**——这一点比直接把它关掉重要。

## 写在最后

回头看，这件事本身很小——不过是让工具在干完活时「叮」一声。但我挺喜欢这个过程带来的两点感受。

第一点，前面埋的那个伏笔：**这是一个 Claude Code 自己的问题，最后是 Claude Code 自己解决的。**

你品一下这个过程：它干完活不提醒我（问题出在它身上）→ 我把这个困扰直接说给它听 → 它去查自己的配置、确认现状、放音效给我挑、改好配置文件、还顺手解释了「为什么会重复响」。从头到尾，我没有自己去啃文档、没有去搜「Claude Code 提示音怎么设」，我只是**把问题描述清楚，然后让这个工具自己把自己修好**。

这其实是用 AI 工具一个很省力、也很容易被忽略的方式——**遇到工具本身的毛病，先别急着自己研究，试试直接问它、让它来。** 很多时候它比你更清楚自己有哪些开关。

第二点感受，也是落点：

**你用的工具，是可以被你按自己的习惯调教的。**

hook 这个机制，能做的远不止提示音。它的本质是「某件事发生时，自动替你做点什么」——你完全可以顺着这个思路，想想自己平时还有哪些「老是忘、老是错过、老是要手动来一下」的瞬间，都可以交给它。

而起点，往往就是一个很小的、具体的烦恼，比如——「它干完了，我怎么不知道呢？」
